# 03 反馈回路

TDD 将反馈回路从"宏循环"（小时/天级）压缩到"微循环"（秒/分钟级）。反馈越快，bug 发现越早，修复成本越低。

## 宏循环 vs 微循环

```
  ┌──────────────────────────────────────────────────────────┐
  │  不用 TDD 的宏循环：                                     │
  │                                                          │
  │  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐          │
  │  │ 写代码│──→│手动测试│──→│发现bug│──→│ 修复  │──→ ...  │
  │  └──────┘    └──────┘    └──────┘    └──────┘          │
  │      │                                                  │
  │      反馈延迟：数小时到数天                              │
  │      每个循环代价：高（定位 bug、理解 bug、修 bug）      │
  │      bug 定位难度：高（大量代码中找）                    │
  │                                                          │
  │  TDD 的微循环：                                          │
  │                                                          │
  │  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐          │
  │  │写测试│──→│看失败 │──→│写代码│──→│看通过 │──→ 重构  │
  │  └──────┘    └──────┘    └──────┘    └──────┘          │
  │      │                                                  │
  │      反馈延迟：数秒到数分钟                              │
  │      每个循环代价：低（写一个测试、写几行代码）          │
  │      bug 定位难度：极低（只可能是最后那个改动）          │
  └──────────────────────────────────────────────────────────┘
```

## 修复成本的时间曲线

```
  ┌──────────────────────────────────────────────────────────┐
  │  修复成本与发现时间的关系：                              │
  │                                                          │
  │  修复成本                                                │
  │  ────────                                                │
  │  高 ┤                                          ┌───     │
  │     │                                    ┌───┘         │
  │     │                              ┌───┘               │
  │     │                        ┌───┘                     │
  │     │                  ┌───┘                           │
  │     │            ┌───┘                                 │
  │     │      ┌───┘                                      │
  │  低 ┤─────┘                                            │
  │     └──┬──────┬───────┬──────┬──────┬──────┬──         │
  │       写代码时  提交时  代码审查  QA测试  发布后  用户   │
  │                                                          │
  │  TDD 的位置：写代码时                                    │
  │  → 修复成本最低                                         │
  │  → bug 被引入后 30 秒内被发现                           │
  │  → 不需要"定位"——只有最后那个改动可能有问题             │
  │                                                          │
  │  不用 TDD 的位置：QA 测试或发布后                       │
  │  → 修复成本高出 10-100 倍                               │
  │  → 需要定位、理解、修复、回归测试                       │
  │  → 可能引入新 bug                                       │
  └──────────────────────────────────────────────────────────┘
```

## 微循环为什么快？

```
  ┌──────────────────────────────────────────────────────────┐
  │  微循环的三个加速因素：                                  │
  │                                                          │
  │  1. bug 发现时间 = bug 引入时间 ± 30 秒                 │
  │     ──────────────────────────────────────               │
  │     不需要"找" bug——你知道是最后 30 秒的代码引入的     │
  │     不需要"理解" bug——只有一行新代码可能是原因         │
  │     不需要"回归测试"——只改了一行，其他测试已经保证      │
  │                                                          │
  │  2. 每次只改一件事                                      │
  │     ────────────────                                     │
  │     宏循环中：改了 20 处 → 发现 bug → 哪个引入的？      │
  │     微循环中：改了 1 处 → 发现 bug → 就是这处引入的     │
  │     → 变更量与定位难度成正比                             │
  │     → 微循环的变更量 = 1 → 定位难度 = 1                 │
  │                                                          │
  │  3. 测试即文档                                          │
  │     ────────────                                         │
  │     每个测试记录了"这个行为为什么存在"                  │
  │     → 不需要回忆"3 个月前为什么这么写"                 │
  │     → 测试名就是行为规范                                │
  │     → 重构时测试保护行为不变                            │
  │                                                          │
  │  Superpowers 的数据：                                    │
  │  "Systematic approach: 15-30 minutes to fix              │
  │   Random fixes approach: 2-3 hours of thrashing          │
  │   First-time fix rate: 95% vs 40%"                       │
  │  → 微循环的修复成功率是宏循环的 2 倍以上               │
  └──────────────────────────────────────────────────────────┘
```

## "TDD 会拖慢我"——速度的错觉

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的合理化反驳：                              │
  │  "TDD will slow me down" → "TDD faster than debugging"  │
  │                                                          │
  │  为什么 TDD "看起来"慢？                                 │
  │                                                          │
  │  写代码的时间：                                          │
  │  不用 TDD：10 分钟写代码                                │
  │  用 TDD：15 分钟（5 分钟写测试 + 10 分钟写代码）        │
  │  → TDD 确实多花了 5 分钟                                │
  │                                                          │
  │  但完整时间线：                                          │
  │  ┌──────────────────────────────────────────────┐       │
  │  │ 不用 TDD：                                    │       │
  │  │ 10min 写代码 + 30min 手动测试                 │       │
  │  │ + 60min 调试发现的 bug                        │       │
  │  │ + 30min 回归测试                              │       │
  │  │ = 130min                                      │       │
  │  │                                                │       │
  │  │ 用 TDD：                                      │       │
  │  │ 5min 写测试 + 10min 写代码                    │       │
  │  │ + 5min 重构 + 0min 调试（微循环内解决）       │       │
  │  │ + 0min 手动测试                               │       │
  │  │ = 20min                                       │       │
  │  └──────────────────────────────────────────────┘       │
  │                                                          │
  │  速度错觉的原因：                                        │
  │  - 我们只看到"写代码"的时间（可见部分）                │
  │  - 看不到"调试+手动测试+回归"的时间（隐藏部分）        │
  │  - TDD 把隐藏部分的时间转移到了可见部分                │
  │  - 总时间更短，但"写代码"阶段更长 → 感觉更慢         │
  │                                                          │
  │  类比：                                                  │
  │  TDD 像"边走边扫"——看起来走路慢了，                    │
  │  但到了终点不用回头扫地。                               │
  │  不用 TDD 像"先走后扫"——走路快，                      │
  │  但回头扫的地更多，总时间更长。                         │
  └──────────────────────────────────────────────────────────┘
```

## 微循环的累积效应

```
  ┌──────────────────────────────────────────────────────────┐
  │  单次微循环 vs 单次宏循环：微循环只快一点               │
  │  但 N 次微循环 vs N 次宏循环：差距指数增长              │
  │                                                          │
  │  原因：宏循环的 bug 会交叉影响                           │
  │                                                          │
  │  假设做 10 个功能，每个有 10% 概率引入 bug：            │
  │                                                          │
  │  微循环（即时发现）：                                    │
  │  → 每个 bug 在 30 秒内修复                              │
  │  → bug 之间不交叉影响                                   │
  │  → 总 bug 修复时间 ≈ 10 × 30s = 5min                   │
  │                                                          │
  │  宏循环（延后发现）：                                    │
  │  → bug 在大量代码中交叉影响                             │
  │  → bug A 的修复可能影响 bug B                           │
  │  → 1 个 bug: 10min                                      │
  │  → 2 个交互 bug: 30min（不是 20min）                   │
  │  → 3 个交互 bug: 60min（不是 30min）                   │
  │  → 总 bug 修复时间 ≈ 超线性增长                        │
  │                                                          │
  │  微循环消除了 bug 的交互效应：                           │
  │  每个功能独立验证 → bug 不累积 → 修复时间线性           │
  │  宏循环允许 bug 累积 → 交互效应 → 修复时间超线性       │
  └──────────────────────────────────────────────────────────┘
```
