# 01 认知模式切换

TDD 的三个阶段不是随意的分工，而是三种完全不同的认知模式。混合使用 = 认知冲突 = 质量下降。

## 三阶段对应三种认知

```
  ┌──────────────────────────────────────────────────────────┐
  │                                                          │
  │  RED（写失败测试）                                       │
  │  ════════════════                                        │
  │  认知模式：规范思考                                      │
  │  核心问题："这段代码应该做什么？"                        │
  │  不关心：怎么做                                          │
  │  输出：一个精确的行为规范                                │
  │                                                          │
  │  GREEN（写最简代码通过）                                 │
  │  ══════════════════════                                  │
  │  认知模式：实现思考                                      │
  │  核心问题："怎么让这个测试通过？"                        │
  │  不关心：优雅、未来需求                                  │
  │  输出：一个满足规范的实现                                │
  │                                                          │
  │  REFACTOR（清理代码）                                    │
  │  ═════════════════════                                   │
  │  认知模式：设计思考                                      │
  │  核心问题："怎么让这段代码更好？"                        │
  │  不关心：新功能                                          │
  │  输出：一个结构更好的实现（行为不变）                    │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

## 三种模式的互斥性

```
  ┌──────────────────────────────────────────────────────────┐
  │  为什么三种模式不能混合？                                │
  │                                                          │
  │  RED + 实现思考 = 实现偏见                               │
  │  ─────────────────────────────────                       │
  │  你在写测试时想"代码会怎么实现"                         │
  │  → 测试被实现污染                                        │
  │  → 测试验证的是"代码做了什么"而非"需求要求什么"        │
  │  → TDD 的核心价值丧失                                    │
  │                                                          │
  │  GREEN + 设计思考 = 过度工程                              │
  │  ──────────────────────────────────                      │
  │  你在写实现时想"这段代码未来可能需要"                   │
  │  → 写了超出测试要求的代码                                │
  │  → 未验证的代码 = 技术债                                │
  │  → YAGNI 被违反                                          │
  │                                                          │
  │  REFACTOR + 功能思考 = 行为漂移                          │
  │  ──────────────────────────────────                      │
  │  你在重构时"顺便"加了一个功能                           │
  │  → 行为改变但没有测试覆盖                                │
  │  → 重构不再是"安全"的                                   │
  │  → 测试的保护被绕过                                      │
  │                                                          │
  │  三种互斥性解释了 Superpowers 的断言：                   │
  │  "Violating the letter of the rules                      │
  │   is violating the spirit of the rules."                 │
  │                                                          │
  │  → 规则的形式（分离三种模式）和规则的精神（防止认知冲突）│
  │    是同一件事。                                          │
  │  → "形式即精神"——没有无意义的形式约束。                 │
  └──────────────────────────────────────────────────────────┘
```

## RED 阶段：规范思考的三个要素

```
  ┌──────────────────────────────────────────────────────────┐
  │  写一个好测试 = 写一个好规范。规范需要三个要素：         │
  │                                                          │
  │  要素 1: 明确的期望行为                                  │
  │  ────────────────────────                                │
  │  "retries failed operations 3 times"                     │
  │  → 不是 "retry works"                                    │
  │  → 测试名 = 行为规范                                     │
  │  → 模糊的测试名 = 模糊的规范 = 模糊的实现              │
  │                                                          │
  │  要素 2: 一个行为                                        │
  │  ──────────────                                          │
  │  "and" 在测试名中？拆分。                                │
  │  → "validates email and domain" → 两个测试               │
  │  → 每个测试验证一个行为                                  │
  │  → 组合行为 = 规范模糊 = 实现难以精确                   │
  │                                                          │
  │  要素 3: 真实代码（除非不得已才 mock）                  │
  │  ──────────────────────────────────────────              │
  │  测试真实行为，不是 mock 行为                            │
  │  → mock 让测试通过 ≠ 代码让测试通过                     │
  │  → mock 是隔离工具，不是测试对象                         │
  │  → 测试 mock = 验证测试自己 = 无信息量                  │
  │                                                          │
  │  Superpowers 的 Good/Bad 对比：                          │
  │  Good: 测试 retryOperation 的真实行为（3 次重试）       │
  │  Bad: 测试 mock 被调用了 3 次                            │
  │  → Good 测试的是"系统做什么"                            │
  │  → Bad 测试的是"mock 做什么"                            │
  └──────────────────────────────────────────────────────────┘
```

## GREEN 阶段：实现思考的约束

```
  ┌──────────────────────────────────────────────────────────┐
  │  GREEN 阶段的核心约束：                                  │
  │  "Write simplest code to pass the test."                 │
  │                                                          │
  │  "Simplest" 不是"最丑"或"最烂"——                         │
  │  是"刚好满足测试要求，不多不少"。                        │
  │                                                          │
  │  为什么不能"顺便"做更多？                                │
  │  ──────────────────────────                              │
  │  1. 超出测试的代码 = 未验证的代码                        │
  │     → 你不知道它对不对                                   │
  │     → 它可能引入 bug                                     │
  │                                                          │
  │  2. "顺便"做的事没有测试驱动                             │
  │     → 你回到了"先写代码"的模式                           │
  │     → 认知模式切换被打破                                 │
  │                                                          │
  │  3. 增加了不必要的复杂度                                  │
  │     → 代码越复杂，维护成本越高                           │
  │     → YAGNI 的本质：不为假设的未来买单                  │
  │                                                          │
  │  Superpowers 的 Good/Bad 对比：                          │
  │  Good: 3 次重试的简单 for 循环                           │
  │  Bad: maxRetries + backoff + onRetry 参数化              │
  │  → Bad 版本在 GREEN 阶段做了 REFACTOR 阶段才该做的事    │
  │  → 混合了实现思考和设计思考                              │
  │  → 认知模式冲突                                          │
  └──────────────────────────────────────────────────────────┘
```

## REFACTOR 阶段：设计思考的安全网

```
  ┌──────────────────────────────────────────────────────────┐
  │  REFACTOR 之所以安全，是因为测试保护了行为不变。         │
  │                                                          │
  │  没有测试的重构：                                        │
  │  ────────────────                                        │
  │  修改代码 → 行为可能改变 → 但你不知道                   │
  │  → 不敢大改 → 代码越来越腐化                            │
  │  → 技术债积累 → 重构成本越来越高                        │
  │                                                          │
  │  有测试的重构：                                          │
  │  ──────────────                                          │
  │  修改代码 → 运行测试 → 测试通过 = 行为不变              │
  │  → 敢于大改 → 代码质量持续提升                          │
  │  → 技术债被持续清理 → 重构成本保持低位                  │
  │                                                          │
  │  REFACTOR 的约束：                                       │
  │  ────────────────                                        │
  │  - 不改行为（测试保证）                                  │
  │  - 不加功能（否则就不是重构，是新的 RED）               │
  │  - 保持测试通过（"stay green"）                         │
  │                                                          │
  │  如果重构导致测试失败：                                  │
  │  1. 行为确实改变了 → 撤销修改，重新考虑                  │
  │  2. 测试耦合了实现 → 测试本身有问题                     │
  │     → 需要重写测试，让它测行为不测实现                   │
  │     → 这是测试质量的反馈信号                            │
  └──────────────────────────────────────────────────────────┘
```

## 认知模式切换的类比

```
  ┌──────────────────────────────────────────────────────────┐
  │  TDD 的三阶段可以类比建筑：                              │
  │                                                          │
  │  RED = 画蓝图                                            │
  │  ──────────────                                          │
  │  "这栋楼应该有几层？每层多高？什么用途？"               │
  │  → 不关心用什么砖、怎么砌墙                              │
  │  → 只关心楼的规格                                        │
  │  → 如果边画蓝图边砌墙 → 规格被施工限制                 │
  │                                                          │
  │  GREEN = 施工                                            │
  │  ────────────                                            │
  │  "按蓝图把楼建起来。"                                    │
  │  → 不加蓝图以外的房间、楼层                              │
  │  → 只做蓝图要求的                                        │
  │  → 如果施工时"顺便"加了一层 → 没有蓝图 = 未验证       │
  │                                                          │
  │  REFACTOR = 装修                                         │
  │  ──────────────────                                      │
  │  "让楼更好用、更好看。"                                  │
  │  → 不改变楼的层数和用途                                  │
  │  → 只改变内部布局和外观                                  │
  │  → 蓝图（测试）保证楼的规格不变                         │
  │                                                          │
  │  类比的启示：                                            │
  │  - 蓝图和施工同时做 = 设计被施工限制                    │
  │  - TDD 中：测试和实现同时做 = 需求被实现限制            │
  │  - 先蓝图后施工 = 设计独立于实现                        │
  │  - 先测试后实现 = 需求独立于实现                        │
  └──────────────────────────────────────────────────────────┘
```
