# 02 信息论视角

TDD 的核心价值可以用信息论来精确表述：先写测试保证了信息从需求流向实现的单向性，后写测试则让实现反向污染了测试。

## 信息流方向

```
  ┌──────────────────────────────────────────────────────────┐
  │  TDD（先测试后实现）：                                   │
  │                                                          │
  │  需求 ──→ 测试 ──→ 实现                                 │
  │          ↑ 信息单向流动：                                 │
  │          需求定义测试，测试约束实现                      │
  │          → 测试是需求的形式化表达                        │
  │          → 实现是测试的可执行满足                        │
  │          → 实现"不知道"其他实现方式的存在               │
  │            它只"知道"测试的要求                          │
  │                                                          │
  │  后写测试（先实现后测试）：                              │
  │                                                          │
  │  需求 ──→ 实现 ──→ 测试                                 │
  │                 ↑ 信息双向流动：                          │
  │                 实现反向影响测试                          │
  │          → 测试被实现污染了                              │
  │          → 你倾向于测试"代码做了什么"                   │
  │            而非"需求要求什么"                            │
  │                                                          │
  │  用函数表达：                                            │
  │  TDD:     测试 = f(需求)                                 │
  │  后测试:  测试 = f(需求, 实现)                           │
  │                                                          │
  │  后测试的测试多了一个参数"实现" →                        │
  │  测试不再是纯需求驱动的，而是被实现影响                  │
  └──────────────────────────────────────────────────────────┘
```

## "看到失败"的信息论意义

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的核心原则：                                │
  │  "If you didn't watch the test fail,                    │
  │   you don't know if it tests the right thing."          │
  │                                                          │
  │  为什么"看到失败"如此重要？                              │
  │                                                          │
  │  一个测试能通过，有 4 种可能：                           │
  │                                                          │
  │  ┌────────────────────────────────────────────────┐     │
  │  │ 可能 1: 测试正确，代码正确                      │     │
  │  │ → 我们想要的                                    │     │
  │  │                                                 │     │
  │  │ 可能 2: 测试错误，代码恰好让它通过              │     │
  │  │ → 我们以为测试正确，其实不正确                  │     │
  │  │                                                 │     │
  │  │ 可能 3: 测试什么都没验证（恒真断言）            │     │
  │  │ → 最危险——以为有覆盖，实际没有                  │     │
  │  │                                                 │     │
  │  │ 可能 4: 测试验证了实现细节，不是行为            │     │
  │  │ → 重构会失败，测试成了实现的枷锁                │     │
  │  └────────────────────────────────────────────────┘     │
  │                                                          │
  │  "看到测试失败"排除了可能 3：                            │
  │  ──────────────────────────────────                      │
  │  如果测试能失败 → 不是恒真断言                           │
  │  → 测试至少在"测试某件事"                              │
  │  → 信息量 > 0                                           │
  │                                                          │
  │  "确认失败原因正确"排除了可能 2：                        │
  │  ────────────────────────────────────                    │
  │  如果测试因为"功能缺失"而失败（不是 typo）              │
  │  → 测试确实在验证正确的行为                              │
  │  → 信息量 = 验证正确行为的信息量                        │
  │                                                          │
  │  但"看到失败"无法排除可能 4（实现耦合）。               │
  │  这就是为什么 TDD 还有 REFACTOR 阶段——                  │
  │  重构时如果测试失败了，说明测试耦合了实现。              │
  │  REFACTOR 是可能 4 的检测机制。                         │
  │                                                          │
  │  信息论总结：                                            │
  │  ┌───────────────────────────────────────────┐          │
  │  │  不验证 RED → 信息量可能 = 0（恒真断言）  │          │
  │  │  验证 RED   → 信息量 > 0（确认非恒真）    │          │
  │  │  确认原因   → 信息量 = 验证正确行为的信息  │          │
  │  │  REFACTOR   → 检测可能 4（实现耦合）       │          │
  │  └───────────────────────────────────────────┘          │
  └──────────────────────────────────────────────────────────┘
```

## Verify RED 的三个确认项

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 要求 Verify RED 时确认三件事：              │
  │                                                          │
  │  1. Test fails (not errors)                              │
  │     ────────────────────────                              │
  │     区分"测试失败"和"测试出错"                          │
  │     - 失败：断言不满足 → 测试在验证行为                 │
  │     - 出错：代码抛异常 → 测试本身有问题                 │
  │     - 出错 ≠ 失败 → 出错的测试信息量为 0                │
  │                                                          │
  │  2. Failure message is expected                           │
  │     ──────────────────────────────────                    │
  │     失败信息与期望一致                                    │
  │     - 期望 "Email required"，得到 "Email required"       │
  │     - → 测试在验证正确的行为                             │
  │     - 期望 "Email required"，得到 "undefined is not      │
  │       a function" → 测试在验证实现细节                   │
  │     - → 失败信息 = 行为规范的反馈信号                   │
  │                                                          │
  │  3. Fails because feature missing (not typos)             │
  │     ────────────────────────────────────────────          │
  │     失败原因 = 功能缺失，不是代码错误                    │
  │     - 功能缺失 → 正确的 RED → 下一步是 GREEN            │
  │     - 代码错误 → 错误的 RED → 需要先修测试              │
  │     - → 区分"应该失败"和"错误地失败"                   │
  │                                                          │
  │  三项确认的信息论意义：                                   │
  │  确认 1 → 测试有信息量（不是 error）                     │
  │  确认 2 → 信息指向正确方向（验证行为不是实现）           │
  │  确认 3 → 信息来源正确（功能缺失不是 typo）              │
  │  → 三项全确认 = 测试是有效的信息源                      │
  └──────────────────────────────────────────────────────────┘
```

## "Test passes immediately" = 信息量为零

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的红旗：                                    │
  │  "Test passes immediately"                               │
  │                                                          │
  │  如果写了测试但立刻通过，意味着：                        │
  │                                                          │
  │  1. 你在测试已有行为                                     │
  │     → 实现已经存在，测试只是"事后确认"                 │
  │     → 你回到了"后写测试"模式                           │
  │     → 测试 = f(需求, 实现) 而非 f(需求)                 │
  │                                                          │
  │  2. 测试没有断言或断言恒真                               │
  │     → 测试永远通过，不管代码对不对                       │
  │     → 信息量 = 0                                         │
  │                                                          │
  │  3. 测试的名字和已有实现恰好匹配                         │
  │     → 你写的测试是在描述现有代码，不是规范需求           │
  │     → 实现偏见已经发生                                   │
  │                                                          │
  │  Superpowers 的指示：                                    │
  │  "Test passes? You're testing existing behavior.          │
  │   Fix test."                                             │
  │                                                          │
  │  "Fix test" = 重写测试，让它描述你期望的行为             │
  │  而不是描述已有实现的行为                                │
  │  → 让测试回到 f(需求) 的纯函数                          │
  └──────────────────────────────────────────────────────────┘
```

## 信息不对称：测试知道什么，实现不知道什么

```
  ┌──────────────────────────────────────────────────────────┐
  │  TDD 创造了一种有益的信息不对称：                        │
  │                                                          │
  │  测试知道"期望行为"但不知道"如何实现"                  │
  │  实现知道"如何通过测试"但不知道"还有哪些行为"          │
  │                                                          │
  │  这种不对称保证了：                                      │
  │  - 测试不受实现影响（测试先写，不知道实现）             │
  │  - 实现只满足测试要求（不知道其他行为尚未定义）         │
  │  - 每一轮 RED-GREEN 只增加一个行为                      │
  │    → 增量 = 信息量 = 可控                                │
  │                                                          │
  │  如果没有这种不对称（后写测试）：                        │
  │  - 测试知道"实现做了什么"→ 被实现污染                   │
  │  - 实现"知道"所有行为 → 可能过度工程                    │
  │  - 信息流双向 → 不可控                                   │
  │                                                          │
  │  信息不对称是 TDD 的设计精髓：                           │
  │  用顺序约束人为制造信息不对称，                          │
  │  从而保证测试的独立性和实现的精确性。                    │
  └──────────────────────────────────────────────────────────┘
```
