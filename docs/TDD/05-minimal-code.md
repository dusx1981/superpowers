# 05 最小代码

GREEN 阶段的核心约束："Write simplest code to pass the test." 这不是偷懒——这是 YAGNI 原则的强制执行机制。

## YAGNI 的数学基础

```
  ┌──────────────────────────────────────────────────────────┐
  │  假设你"预判"了一个未来需求，写了额外代码。             │
  │                                                          │
  │  额外代码有用的概率：                                    │
  │  P(有用) = P(预判正确) × P(代码正确|无测试)             │
  │          × P(需求不变) × P(API设计正确|无测试)           │
  │                                                          │
  │  每个概率都 < 1：                                        │
  │  - P(预判正确) ≈ 0.3-0.5（大部分预判是错的）           │
  │  - P(代码正确|无测试) ≈ 0.7-0.9（没有测试验证）        │
  │  - P(需求不变) ≈ 0.5-0.8（需求经常变）                 │
  │  - P(API设计正确|无测试) ≈ 0.6-0.8（没有测试反馈）     │
  │                                                          │
  │  P(有用) ≈ 0.4 × 0.8 × 0.65 × 0.7 ≈ 0.15              │
  │                                                          │
  │  → 预判性代码有用的概率只有 ~15%                        │
  │  → 85% 的情况下，额外代码是浪费                          │
  │  → 浪费 = 复杂度 + 维护成本 + 潜在 bug                  │
  │                                                          │
  │  预判性代码的成本：                                      │
  │  - 写代码的时间（确定成本）                              │
  │  - 代码复杂度增加（确定成本）                            │
  │  - 维护额外代码（持续成本）                              │
  │  - 额外代码可能有 bug（风险成本）                        │
  │  - 额外代码可能误导后续实现（间接成本）                  │
  │                                                          │
  │  期望值 = 0.15 × 收益 - 1.0 × 成本                      │
  │  → 期望值几乎总是负的                                    │
  │  → "最小代码"原则是理性选择                              │
  └──────────────────────────────────────────────────────────┘
```

## "顺便"做更多为什么是认知模式混用

```
  ┌──────────────────────────────────────────────────────────┐
  │  GREEN 阶段"顺便"做更多 = 混合了两种认知模式：         │
  │                                                          │
  │  GREEN 模式：实现思考——"怎么通过测试？"                 │
  │  REFACTOR 模式：设计思考——"怎么让代码更好？"           │
  │                                                          │
  │  混合的后果：                                            │
  │  1. 你写了测试没要求的代码                               │
  │     → 这部分代码没有测试覆盖                             │
  │     → 它的行为是未验证的                                │
  │                                                          │
  │  2. 你在"实现"时做了"设计"决策                          │
  │     → 但设计决策应该基于已有代码的观察                   │
  │     → 而不是基于"未来可能"的猜测                        │
  │                                                          │
  │  3. 你把两个步骤合为一个                                 │
  │     → 如果出 bug，你不知道是"实现"错还是"设计"错       │
  │     → 回退困难——要撤回"实现"还是"设计"？              │
  │                                                          │
  │  Superpowers 的 Good/Bad 对比：                          │
  │  Good: retryOperation 只做 3 次重试                      │
  │  Bad:  retryOperation 加了 maxRetries/backoff/onRetry    │
  │                                                          │
  │  Bad 版本做了什么？                                      │
  │  - 在 GREEN 阶段做了 REFACTOR 阶段的事（参数化）         │
  │  - 在 GREEN 阶段做了未来 RED 阶段的事（backoff 策略）   │
  │  - 一个阶段干了三个阶段的活                              │
  │  → 认知模式完全混乱                                      │
  │  → 代码正确性无法验证（没有测试覆盖这些功能）           │
  └──────────────────────────────────────────────────────────┘
```

## "最小代码"不等于"最差代码"

```
  ┌──────────────────────────────────────────────────────────┐
  │  常见误解：GREEN 阶段写"最烂的代码"                     │
  │                                                          │
  │  纠正："最小"是"刚好满足测试"，不是"故意写差"         │
  │                                                          │
  │  区别：                                                  │
  │  最小代码：通过测试的最简实现                             │
  │  最差代码：通过测试但结构混乱的实现                      │
  │                                                          │
  │  最小代码的例子：                                        │
  │  async function retryOperation<T>(fn: () => Promise<T>)  │
  │    : Promise<T> {                                        │
  │    for (let i = 0; i < 3; i++) {                         │
  │      try { return await fn(); }                          │
  │      catch (e) { if (i === 2) throw e; }                 │
  │    }                                                     │
  │    throw new Error('unreachable');                        │
  │  }                                                       │
  │  → 结构清晰，意图明确，但只做测试要求的                 │
  │  → 没有参数化，没有策略模式，没有配置选项               │
  │  → 这些应该在未来的 RED 中逐步加入                      │
  │                                                          │
  │  如果觉得代码需要优化 → 那是 REFACTOR 阶段的事          │
  │  如果觉得需要加功能 → 那是下一个 RED 阶段的事           │
  │  → GREEN 只做一件事：让测试通过                         │
  └──────────────────────────────────────────────────────────┘
```

## 测试驱动增量的正确节奏

```
  ┌──────────────────────────────────────────────────────────┐
  │  TDD 的增量节奏：                                        │
  │                                                          │
  │  RED 1: "重试 3 次"                                      │
  │  GREEN 1: for 循环 + 硬编码 3                            │
  │  REFACTOR: (可选) 提取重试逻辑                           │
  │                                                          │
  │  RED 2: "可配置重试次数"                                 │
  │  GREEN 2: 参数 maxRetries 替代硬编码                     │
  │  REFACTOR: (可选)                                        │
  │                                                          │
  │  RED 3: "指数退避"                                       │
  │  GREEN 3: backoff 参数                                   │
  │  REFACTOR: 提取退避策略                                  │
  │                                                          │
  │  每个 RED 只定义一个行为。                               │
  │  每个 GREEN 只实现一个行为。                             │
  │  每个行为都有测试覆盖。                                  │
  │  → 代码增量 = 测试增量 = 信息增量 = 可控增量           │
  │                                                          │
  │  非增量方式：                                            │
  │  一次性写：for 循环 + maxRetries + backoff + onRetry     │
  │  → 4 个行为只有 1 个测试覆盖                            │
  │  → 3 个行为是未验证的                                   │
  │  → 复杂度跳跃 = 风险跳跃                               │
  └──────────────────────────────────────────────────────────┘
```
