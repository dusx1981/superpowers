# 07 顺序为什么重要

Superpowers 列出了五个"为什么后写测试不行"的论点。本章从原理层面逐一拆解。

## 论点 1："Tests after pass immediately, proving nothing"

### 后写测试为什么"证明不了什么"？

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的表述：                                    │
  │  "Tests written after code pass immediately.              │
  │   Passing immediately proves nothing."                   │
  │                                                          │
  │  原理：                                                  │
  │  一个测试的价值不在于它能通过，而在于它曾经失败过。      │
  │                                                          │
  │  测试通过可能是因为：                                    │
  │  1. 代码正确（我们期望的）                               │
  │  2. 测试验证了错误的行为（但我们不知道）                 │
  │  3. 测试什么都没验证（恒真断言，但我们不知道）           │
  │  4. 测试耦合了实现（通过是因为实现匹配，不是行为正确）   │
  │                                                          │
  │  后写测试 = 测试立刻通过 → 你无法区分以上 4 种可能      │
  │  先写测试 = 测试先失败 → 你确认了至少排除了可能 3       │
  │  确认失败原因正确 → 你还排除了可能 2                     │
  │                                                          │
  │  "看到失败" = 确认测试有信息量                          │
  │  "立刻通过" = 不知道测试有没有信息量                     │
  │                                                          │
  │  不确定信息量 > 0 → 测试价值不可知                       │
  │  确认信息量 > 0 → 测试价值可确认                         │
  │  → 顺序差异 = 确认成本的差异                            │
  └──────────────────────────────────────────────────────────┘
```

## 论点 2："Tests-after = 'what does this do?' Tests-first = 'what should this do?'"

### 问题的方向决定了答案的方向

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的表述：                                    │
  │  "Tests-after answer 'What does this do?'                 │
  │   Tests-first answer 'What should this do?'"             │
  │                                                          │
  │  原理：                                                  │
  │  你问的问题决定了你得到的答案。                           │
  │                                                          │
  │  后写测试时你问：                                        │
  │  "这段代码做了什么？"                                    │
  │  → 你观察实现，然后写测试验证你观察到的                  │
  │  → 你在确认实现，不是在规范需求                          │
  │  → 即使实现是错的，测试也会通过                         │
  │    （因为测试验证的是"实现做了什么"，                    │
  │     不是"实现应该做什么"）                               │
  │                                                          │
  │  先写测试时你问：                                        │
  │  "这段代码应该做什么？"                                  │
  │  → 你从需求出发，写测试规范期望行为                      │
  │  → 实现必须满足测试的规范                                │
  │  → 如果实现是错的，测试会失败                           │
  │    （因为测试验证的是"应该做什么"，                      │
  │     不是"实际做了什么"）                                 │
  │                                                          │
  │  信息论再次出现：                                        │
  │  "做了什么" → 信息来源 = 实现 → 被实现污染              │
  │  "应该做什么" → 信息来源 = 需求 → 独立于实现            │
  │                                                          │
  │  顺序改变了信息的来源。                                  │
  │  来源的独立性决定了测试的有效性。                        │
  └──────────────────────────────────────────────────────────┘
```

## 论点 3："Manual testing is ad-hoc"

### 手动测试为什么不可替代自动化测试？

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的表述：                                    │
  │  "Manual testing is ad-hoc. You think you tested          │
  │   everything but:                                        │
  │   - No record of what you tested                         │
  │   - Can't re-run when code changes                       │
  │   - Easy to forget cases under pressure                  │
  │   - 'It worked when I tried it' ≠ comprehensive"         │
  │                                                          │
  │  原理：手动测试 vs 自动化测试的差异是                    │
  │  不可重复 vs 可重复。                                    │
  │                                                          │
  │  不可重复的测试：                                        │
  │  → 你测了一个场景 → 但没有记录                          │
  │  → 代码改了 → 你需要重新手动测                          │
  │  → 但你忘了上次测了哪些场景                             │
  │  → 回归测试不可靠                                       │
  │                                                          │
  │  可重复的测试：                                          │
  │  → 测试代码就是记录                                     │
  │  → 代码改了 → 运行测试即可                              │
  │  → 每次运行都验证所有场景                               │
  │  → 回归测试可靠                                         │
  │                                                          │
  │  深层差异：                                              │
  │  手动测试 = 一次性验证 = 时刻 T 的快照                  │
  │  自动化测试 = 持续性验证 = 任何时刻 T 的不变量          │
  │                                                          │
  │  快照只在一个时刻有效。                                  │
  │  不变量在所有时刻有效。                                  │
  │  → 自动化测试是代码行为的不变量，不是快照。              │
  └──────────────────────────────────────────────────────────┘
```

## 论点 4："Sunk cost fallacy"

### 为什么"删除更理性"？

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的表述：                                    │
  │  "Sunk cost fallacy. The time is already gone.            │
  │   Your choice now:                                       │
  │   - Delete and rewrite with TDD (X more hours,            │
  │     high confidence)                                     │
  │   - Keep it and add tests after (30 min, low confidence,  │
  │     likely bugs)"                                        │
  │                                                          │
  │  原理：决策应该基于未来收益，不是过去成本。              │
  │                                                          │
  │  错误的决策框架：                                        │
  │  "我已经花了 3 小时写代码，删除 = 浪费 3 小时"          │
  │  → 3 小时已经花了，不可回收                             │
  │  → 决策应该基于"未来哪个选项收益更高"                   │
  │                                                          │
  │  正确的决策框架：                                        │
  │  ┌──────────────────────────────────────────────┐       │
  │  │ 选项 A: 删除 + TDD 重写                       │       │
  │  │ 未来成本: X 小时                              │       │
  │  │ 未来收益: 高信心代码 + 有效测试 + 可维护      │       │
  │  │                                                │       │
  │  │ 选项 B: 保留 + 后补测试                       │       │
  │  │ 未来成本: 30 分钟 + 后续调试时间               │       │
  │  │ 未来收益: 低信心代码 + 被污染测试 + 技术债     │       │
  │  │                                                │       │
  │  │ 选项 B 的"30 分钟"是虚假的：                  │       │
  │  │ 被污染测试 + 未验证代码 → 后续 bug             │       │
  │  │ → 调试时间 >> X 小时                           │       │
  │  └──────────────────────────────────────────────┘       │
  │                                                          │
  │  "Working code without real tests is technical debt."    │
  │  → 技术债需要偿还利息。                                 │
  │  → 利息 = 每次修改时的额外风险和调试时间。              │
  │  → 长期来看，利息总和远超删除重写的"本金"。            │
  └──────────────────────────────────────────────────────────┘
```

## 论点 5："TDD IS pragmatic"

### TDD 为什么是务实的，不是教条的？

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的表述：                                    │
  │  "TDD IS pragmatic:                                     │
  │   - Finds bugs before commit (faster than debugging)     │
  │   - Prevents regressions (tests catch breaks immediately)│
  │   - Documents behavior (tests show how to use code)      │
  │   - Enables refactoring (change freely, tests catch)"    │
  │                                                          │
  │  原理："务实"的定义是"用最少的总时间解决问题"。        │
  │                                                          │
  │  反 TDD 的"务实"：                                     │
  │  "跳过测试，快速实现" → 写代码快                       │
  │  → 但调试慢、回归慢、重构慢、文档缺失                   │
  │  → 总时间 = 快写 + 慢调试 + 慢回归 + 慢重构            │
  │                                                          │
  │  TDD 的"务实"：                                        │
  │  "先写测试，增量实现" → 写代码稍慢                     │
  │  → 但调试快（微循环）、回归快（自动测试）、             │
  │    重构快（测试保护）、文档在测试中                     │
  │  → 总时间 = 稍慢写 + 快调试 + 快回归 + 快重构           │
  │                                                          │
  │  关键洞察：                                              │
  │  软件开发的"总时间"中，写代码只占 20-30%。              │
  │  调试、回归、重构、理解占 70-80%。                       │
  │  → 优化"写代码"速度的效果有限。                        │
  │  → 优化"调试/回归/重构"速度的效果更大。               │
  │  → TDD 优化的是 70-80% 的部分。                         │
  │  → 这才是真正的务实。                                   │
  │                                                          │
  │  "Pragmatic shortcuts = debugging in production           │
  │   = slower."                                             │
  │  → "务实捷径"省了写测试的时间，                        │
  │    但花了更多时间在生产环境调试。                       │
  │  → 不是务实，是时间债务。                               │
  └──────────────────────────────────────────────────────────┘
```

## 五个论点的共同结构

```
  ┌──────────────────────────────────────────────────────────┐
  │  五个论点都指向同一个底层原理：                          │
  │                                                          │
  │  顺序不是偏好，是信息完整性的保障。                      │
  │                                                          │
  │  论点 1: 后写测试 = 信息量不可知                         │
  │  论点 2: 后写测试 = 信息来源被污染                       │
  │  论点 3: 手动测试 = 信息不可重复                         │
  │  论点 4: 保留代码 = 信息被沉没成本扭曲                   │
  │  论点 5: 跳过 TDD = 信息延迟导致成本转移                 │
  │                                                          │
  │  共性：                                                  │
  │  "顺序正确" → 信息完整、来源独立、可重复、决策理性      │
  │  "顺序错误" → 信息缺失、来源污染、不可重复、决策扭曲    │
  │                                                          │
  │  → TDD 的本质：通过顺序约束保证信息的完整性。           │
  │  → 不是仪式，是信息论。                                  │
  └──────────────────────────────────────────────────────────┘
```
