# 04 删除原则

"Write code before the test? Delete it. Start over." 这是 TDD 中最反直觉的规则。为什么先写的代码必须删？不是因为代码一定错，而是因为它的存在会污染后续的测试。

## 三种污染

```
  ┌──────────────────────────────────────────────────────────┐
  │  污染 1: 实现偏见                                       │
  │  ──────────────────                                      │
  │  如果你看到了先写的代码，你写测试时会不自觉地            │
  │  测试"代码做了什么"而非"需求要求什么"                  │
  │                                                          │
  │  大脑的工作方式：                                        │
  │  看到实现 → 理解实现 → 测试基于理解 → 测试被实现污染   │
  │  → 即使你试图"不看"，潜意识已经知道实现方式             │
  │  → 测试不再是 f(需求)，变成了 f(需求, 实现)             │
  │                                                          │
  │  "Don't look at it" 不是为了惩罚，                       │
  │  是为了防止实现偏见进入测试。                            │
  │                                                          │
  │  污染 2: 路径依赖                                        │
  │  ──────────────                                          │
  │  如果你保留了先写的代码，你写测试时会在脑中             │
  │  "适配"这个实现——                                      │
  │  "我需要测试这个接口，因为代码就是这样写的"            │
  │                                                          │
  │  但正确的测试应该从需求出发，                            │
  │  不受任何实现路径的影响。                                │
  │  你可能设计出完全不同的接口——                            │
  │  如果你没看到已有实现的话。                              │
  │                                                          │
  │  "Don't 'adapt' it while writing tests"                  │
  │  防止的是路径依赖——                                     │
  │  让测试定义接口，而不是让接口定义测试。                  │
  │                                                          │
  │  污染 3: 沉没成本                                        │
  │  ──────────────                                          │
  │  如果你花了 3 小时写代码，你不会愿意删掉它。            │
  │  → 你会找理由保留："先留着当参考"                       │
  │  → 然后你会"参考"它 → 适配它 → 等于没做 TDD           │
  │                                                          │
  │  沉没成本谬误的数学：                                    │
  │  保留旧代码的期望收益 = P(代码正确) × 节省的时间         │
  │  保留旧代码的期望成本 = 1（确定性）× 测试被污染的风险   │
  │                                                          │
  │  P(代码正确) 不高（不然为什么需要 TDD？）                │
  │  测试被污染的风险 ≈ 1（人类心理几乎不可避免）           │
  │  → 保留旧代码的期望成本 > 期望收益                      │
  │  → 删除是理性选择                                       │
  └──────────────────────────────────────────────────────────┘
```

## "留作参考"为什么不行？

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 明确禁止：                                  │
  │  - "Don't keep it as 'reference'"                        │
  │  - "Don't 'adapt' it while writing tests"                │
  │  - "Don't look at it"                                    │
  │                                                          │
  │  看似极端，但每一条都有逻辑：                            │
  │                                                          │
  │  "留作参考"的真实路径：                                 │
  │                                                          │
  │  Step 1: "我只看看代码结构，不会直接复制"               │
  │  → 你看到了实现方式                                     │
  │  → 测试被实现偏见污染                                   │
  │                                                          │
  │  Step 2: "这个接口设计得不错，我在测试中用这个接口"     │
  │  → 测试被路径依赖污染                                   │
  │  → 你不再从需求出发设计接口                             │
  │                                                          │
  │  Step 3: "这段代码逻辑是对的，我只需要稍微修改"         │
  │  → 你在适配现有代码                                     │
  │  → 等于没做 TDD                                        │
  │                                                          │
  │  三步走完 = TDD 完全失效                                 │
  │  → 每一步都是"看起来无害"的滑坡                        │
  │  → 只有"Delete means delete"能截断滑坡                  │
  │                                                          │
  │  Superpowers 的表达极其精确：                            │
  │  "Delete means delete"                                   │
  │  不是"建议删除"——是"删除=删除"                        │
  │  没有灰色地带。"参考"不是"删除"。"适配"不是"删除"。    │
  │  "删除"就是"删除"。                                     │
  └──────────────────────────────────────────────────────────┘
```

## 删除的心理成本

```
  ┌──────────────────────────────────────────────────────────┐
  │  删除几小时的工作在心理上极其痛苦。                      │
  │  Superpowers 承认这一点：                                │
  │                                                          │
  │  "Deleting X hours is wasteful?"                         │
  │  → "Sunk cost fallacy. The time is already gone."        │
  │                                                          │
  │  沉没成本谬误的解释：                                    │
  │  ────────────────                                        │
  │  已花费的时间是不可回收的。                              │
  │  你的选择不是"保留 3 小时的成果 vs 浪费 3 小时"，       │
  │  而是"未来用 TDD 重写（高信心）vs                       │
  │   保留未验证代码（低信心 + 可能的 bug）"。              │
  │                                                          │
  │  选择的正确框架：                                        │
  │  ┌──────────────────────────────────────────────┐       │
  │  │ 选项 A: 删除 + TDD 重写                       │       │
  │  │ 成本: X 小时                                  │       │
  │  │ 收益: 高信心代码 + 有效测试                   │       │
  │  │                                                │       │
  │  │ 选项 B: 保留 + 后补测试                       │       │
  │  │ 成本: 30 分钟                                  │       │
  │  │ 收益: 低信心代码 + 被污染的测试               │       │
  │  │                                                │       │
  │  │ 选项 A 的成本更高，但收益远更高。              │       │
  │  │ 而选项 B 的"30 分钟"成本是虚假的——            │       │
  │  │ 未验证代码 + 被污染测试的后续调试成本          │       │
  │  │ 可能远超 X 小时。                              │       │
  │  └──────────────────────────────────────────────┘       │
  │                                                          │
  │  "Working code without real tests is technical debt."    │
  │  → 技术债的利息会在未来偿还——                          │
  │    而利息通常远超本金。                                  │
  └──────────────────────────────────────────────────────────┘
```

## "需要先探索"的豁免

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 允许一种"豁免"：                            │
  │  "Need to explore first?                                 │
  │   Fine. Throw away exploration, start with TDD."         │
  │                                                          │
  │  这不是矛盾——这是对认知阶段的承认：                     │
  │                                                          │
  │  探索阶段 = 你还不知道"做什么"                         │
  │  → 不写正式代码，只写实验性代码                         │
  │  → 实验性代码不进版本控制                               │
  │  → 实验性代码用完即删                                   │
  │                                                          │
  │  TDD 阶段 = 你知道了"做什么"                            │
  │  → 从测试开始正式实现                                   │
  │  → 正式代码受测试驱动                                   │
  │  → 正式代码进版本控制                                   │
  │                                                          │
  │  关键约束：探索代码 ≠ 正式代码                          │
  │  → 探索代码必须被删除，不能"升级"为正式代码            │
  │  → 如果探索代码被保留 = 实现偏见 + 路径依赖 + 沉没成本  │
  │  → "Throw away exploration, start with TDD"              │
  │  → 删除是必须的，不是可选的                             │
  └──────────────────────────────────────────────────────────┘
```
