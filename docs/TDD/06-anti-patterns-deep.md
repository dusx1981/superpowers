# 06 反模式深析

testing-anti-patterns.md 列出了 5 种反模式。本章从原理层面分析为什么它们是反模式，以及 TDD 如何预防。

## 反模式 1：测试 Mock 行为

### 为什么测试 mock 是信息量为零的行为？

```
  ┌──────────────────────────────────────────────────────────┐
  │  信息论视角：                                            │
  │                                                          │
  │  测试的目标：验证系统 S 在输入 I 下产生输出 O            │
  │  → 测试的信息量 = P(O|I,S) 的验证程度                  │
  │                                                          │
  │  测试真实代码：                                          │
  │  输入 → [真实代码] → 输出                                │
  │  → 测试验证了 P(O|I, 真实代码)                           │
  │  → 信息量 > 0                                            │
  │                                                          │
  │  测试 mock：                                             │
  │  输入 → [mock] → mock 输出                               │
  │  → 测试验证了 P(mock输出|I, mock)                        │
  │  → 但 mock 输出是你自己定义的                            │
  │  → 你在验证自己写的数据，不是系统的行为                  │
  │  → 信息量 = 0（等同于验证 1+1=2）                       │
  │                                                          │
  │  Superpowers 的 Gate Function：                          │
  │  "BEFORE asserting on any mock element:                   │
  │   Ask: 'Am I testing real component behavior              │
  │   or just mock existence?'                                │
  │   IF testing mock existence: STOP."                       │
  │                                                          │
  │  → Gate Function 是一个信息量检查——                      │
  │    如果断言的信息量为零，就不要写它。                    │
  └──────────────────────────────────────────────────────────┘
```

### Mock 的正确使用：隔离而非替代

```
  ┌──────────────────────────────────────────────────────────┐
  │  Mock 的唯一正当理由：隔离外部依赖                       │
  │                                                          │
  │  正确的 mock：                                           │
  │  测试 → [真实代码 A] → [mock 外部服务 B]                 │
  │  → A 是被测对象（真实代码）                              │
  │  → B 是外部依赖（mock 隔离）                             │
  │  → 测试验证的是 A 的行为，不是 B 的行为                 │
  │                                                          │
  │  错误的 mock：                                           │
  │  测试 → [mock A] → [mock B]                              │
  │  → A 也被 mock 了                                       │
  │  → 测试验证的是 mock 的行为                              │
  │  → 信息量 = 0                                            │
  │                                                          │
  │  原则：mock 最外层的外部依赖，                            │
  │  不要 mock 被测代码本身。                                │
  │                                                          │
  │  Superpowers 的原则：                                    │
  │  "Test what the code does, not what the mocks do."       │
  │  → 被测对象永远应该是真实代码                            │
  │  → Mock 只是让真实代码在可控环境下运行                  │
  └──────────────────────────────────────────────────────────┘
```

## 反模式 2：生产代码中的测试专用方法

### 为什么 test-only 方法破坏设计？

```
  ┌──────────────────────────────────────────────────────────┐
  │  测试专用方法的问题不在测试本身，                        │
  │  而在于它暴露了设计的缺陷。                              │
  │                                                          │
  │  如果你需要一个 destroy() 方法仅用于测试清理，           │
  │  这说明：                                                │
  │  1. Session 类管理了不属于它的资源                       │
  │     → 违反了单一职责原则                                │
  │  2. 测试需要手动管理生命周期                             │
  │     → 说明生命周期管理没有被正确抽象                    │
  │  3. 生产代码和测试代码的职责混淆                         │
  │     → 测试工具应该在自己的文件中，不是在生产类中        │
  │                                                          │
  │  TDD 如何预防：                                          │
  │  如果你先写测试，你会先设计"如何使用"这个类，            │
  │  然后才实现。                                            │
  │  → 你会发现"测试需要一个 destroy 方法"                  │
  │  → 但 destroy 不应该是类的 API                           │
  │  → 你会把清理逻辑放到测试工具中                          │
  │  → 设计在测试阶段就被修正了                              │
  │                                                          │
  │  后写测试则不同：                                        │
  │  → 代码已经写好了，类已经有了 destroy()                  │
  │  → 你发现测试需要清理，直接调用 destroy()                │
  │  → 没有动机去修正设计——"代码已经有了"                  │
  │  → 设计缺陷被固化                                       │
  │                                                          │
  │  Superpowers 的 Gate Function：                          │
  │  "BEFORE adding any method to production class:           │
  │   Ask: 'Is this only used by tests?'                     │
  │   IF yes: STOP. Put it in test utilities instead."        │
  │                                                          │
  │  → Gate Function 防止设计缺陷被测试需求引入生产代码     │
  └──────────────────────────────────────────────────────────┘
```

## 反模式 3：不理解依赖就 Mock

### "为了安全而 mock" 为什么危险？

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的反模式 3 的例子：                         │
  │                                                          │
  │  测试"检测重复服务器"时，mock 了 ToolCatalog，           │
  │  但 mock 破坏了测试依赖的副作用（写入配置）。            │
  │  → 测试通过了，但测试验证的行为不存在（重复检测失效）   │
  │  → 测试的信息量 = 0（甚至为负——给了错误的信心）        │
  │                                                          │
  │  深层原因：                                              │
  │  mock 是对真实行为的简化，但简化可能遗漏关键行为。       │
  │  如果你不理解真实行为有哪些关键部分，                    │
  │  你就无法知道简化是否安全。                              │
  │                                                          │
  │  "I'll mock this to be safe" 的逻辑错误：               │
  │  → 假设：mock 更安全，因为消除了外部依赖                │
  │  → 事实：mock 可能消除了测试需要的行为                  │
  │  → "安全"的 mock 反而让测试更危险                       │
  │                                                          │
  │  Superpowers 的 Gate Function：                          │
  │  "BEFORE mocking any method:                             │
  │   STOP — Don't mock yet.                                 │
  │   1. Ask: 'What side effects does the real method have?' │
  │   2. Ask: 'Does this test depend on any of those?'"      │
  │                                                          │
  │  → 先理解，再 mock。                                     │
  │  → 不理解就 mock = 盲目简化 = 信息丢失。                │
  └──────────────────────────────────────────────────────────┘
```

## 反模式 4：不完整的 Mock 数据

### 部分 mock = 隐式假设

```
  ┌──────────────────────────────────────────────────────────┐
  │  部分 mock 数据的问题：                                  │
  │                                                          │
  │  mockResponse = {                                        │
  │    status: 'success',                                    │
  │    data: { userId: '123', name: 'Alice' }                │
  │    // 缺失: metadata.requestId                           │
  │  }                                                       │
  │                                                          │
  │  你只 mock 了"你知道需要的"字段。                        │
  │  但下游代码可能依赖你不知道的字段。                      │
  │                                                          │
  │  信息论视角：                                            │
  │  部分 mock = 对真实数据的压缩                            │
  │  → 压缩丢失了信息                                       │
  │  → 丢失的信息可能是关键的                               │
  │  → 测试通过 ≠ 真实环境通过                              │
  │                                                          │
  │  Superpowers 的铁律：                                    │
  │  "Mock the COMPLETE data structure as it exists           │
  │   in reality, not just fields your immediate test uses." │
  │                                                          │
  │  → 完整 mock = 无损表示                                 │
  │  → 无损 = 测试通过 ≈ 真实环境通过                       │
  │  → 损失零容忍——宁可多 mock 几个字段，                   │
  │    不要遗漏可能关键的字段。                              │
  └──────────────────────────────────────────────────────────┘
```

## 反模式 5：测试作为事后补充

### TDD 如何预防所有反模式

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的断言：                                    │
  │  "Following strict TDD prevents these anti-patterns."    │
  │                                                          │
  │  为什么？因为 TDD 的每个阶段都在对抗对应的反模式：       │
  │                                                          │
  │  反模式 1（测试 mock）:                                  │
  │  TDD 的 RED 阶段：先写测试，用真实代码                  │
  │  → 如果测试对 mock 断言，RED 阶段就会暴露               │
  │    （mock 还不存在，你必须先写真实代码来让测试通过）     │
  │  → TDD 天然阻止"只测 mock"                              │
  │                                                          │
  │  反模式 2（test-only 方法）:                              │
  │  TDD 的 RED 阶段：测试定义了类的公共 API                │
  │  → 如果需要 destroy()，测试会先写                       │
  │  → 但测试只关心公共 API，不关心内部实现                 │
  │  → test-only 方法不会出现在测试驱动的 API 设计中        │
  │                                                          │
  │  反模式 3（盲目 mock）:                                  │
  │  TDD 的 Verify RED：测试必须正确失败                    │
  │  → 如果 mock 破坏了测试需要的行为                       │
  │  → Verify RED 会发现测试没有正确失败                    │
  │  → TDD 的验证步骤暴露了 mock 的问题                     │
  │                                                          │
  │  反模式 4（不完整 mock）:                                │
  │  TDD 的微循环：每次只加一个行为                         │
  │  → 不完整 mock 会在后续 RED 中暴露                     │
  │    （新测试依赖缺失字段 → 失败 → 修正 mock）            │
  │  → 微循环持续修正 mock 的完整性                         │
  │                                                          │
  │  反模式 5（测试作为补充）:                                │
  │  TDD 的本质：测试先于实现                                │
  │  → 不存在"后补测试"的场景                               │
  │  → 这个反模式在 TDD 中不可能出现                        │
  │                                                          │
  │  总结：TDD 不是"防止反模式的规则列表"——                │
  │  它是一个让反模式不可能出现的流程。                      │
  │  规则列表是"禁止做错事"，TDD 是"让做错事变得不可能"。  │
  └──────────────────────────────────────────────────────────┘
```
