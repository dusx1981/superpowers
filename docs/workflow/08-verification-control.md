# 08 验证控制

验证控制是 Superpowers 的**最后一道防线**——在 AI 声称"完成了"之前，必须用证据支持。

## 铁律

```
  NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE

  如果你在本条消息中没有运行验证命令，你不能声称它通过了。
```

### "FRESH" 的含义

```
  ┌──────────────────────────────────────────────────────┐
  │  "新鲜"意味着：                                      │
  │                                                      │
  │  ✗ 上一轮运行的结果（可能已过时）                    │
  │  ✗ "之前通过了"的回忆（可能记忆不准确）              │
  │  ✗ "应该没问题"的推断（不是证据）                    │
  │  ✗ 子智能体说"我验证了"的报告（需要独立验证）        │
  │                                                      │
  │  ✓ 本条消息中运行了完整命令                          │
  │  ✓ 读取了完整输出                                    │
  │  ✓ 检查了退出码                                     │
  │  ✓ 计算了失败数                                      │
  └──────────────────────────────────────────────────────┘
```

## 门控函数

```
  在声称任何状态前：

  1. 识别：什么命令证明这个声称？
  2. 运行：执行完整命令（新鲜的，完整的）
  3. 读取：完整输出，检查退出码，计算失败数
  4. 验证：输出确认了声称？
     - 否 → 陈述实际状态及证据
     - 是 → 陈述声称及证据
  5. 只有在这之后：做出声称

  跳过任何步骤 = 撒谎，不是验证
```

## 常见声称的验证要求

| 声称 | 需要的证据 | 不充分的证据 |
|------|-----------|------------|
| "测试通过" | 测试命令输出：0 失败 | 上次运行、"应该通过" |
| "Linter 无错误" | Linter 输出：0 错误 | 部分检查、推断 |
| "构建成功" | 构建命令：exit 0 | Linter 通过、日志看起来好 |
| "Bug 修复了" | 测试原始症状：通过 | 代码改了，假设修了 |
| "回归测试有效" | Red-Green 循环验证 | 测试通过一次 |
| "Agent 完成了" | VCS diff 显示变更 | Agent 报告"成功" |
| "需求满足了" | 逐行清单检查 | 测试通过 |

## "撒谎"强语言

verification-before-completion 技能使用了极强的语言：

```
  "Claiming work is complete without verification is
   dishonesty, not efficiency."

  翻译：声称工作完成而不验证是不诚实，不是效率。
```

```
  "Skip any step = lying, not verifying"

  翻译：跳过任何步骤 = 撒谎，不是验证。
```

### 为什么用"撒谎"而不是"不准确"？

```
  语言强度对比：
  ────────────

  "不经验证声称完成是不好的做法"  → 建议，AI 可能忽略
  "不经验证声称完成是不准确的"    → 事实陈述，AI 可能接受"足够准确"
  "不经验证声称完成是不诚实的"    → 道德判断，AI 极不愿意被标记为不诚实
  "不经验证声称完成 = 撒谎"      → 终极标签，AI 绝对要避免

  越强的语言 → 越大的心理压力 → 越不容易跳过验证
```

## 红旗：停下来验证

```
  这些想法意味着停下来：
  ────────────────────

  - 使用"应该"、"可能"、"看起来"
  - 在验证前表达满意（"Great!"、"Perfect!"、"Done!"）
  - 即将提交/推送/创建 PR 而未验证
  - 信任 Agent 成功报告
  - 依赖部分验证
  - 想着"就这一次"
  - 累了想让工作结束
  - 任何暗示成功但未运行验证的措辞
```

## 合理化防御

| 借口 | 现实 |
|------|------|
| "应该没问题了" | 运行验证命令 |
| "我很自信" | 自信 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Linter 通过了" | Linter ≠ 编译器 |
| "Agent 说成功了" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "部分检查够了" | 部分证明不了什么 |
| "换了个说法所以规则不适用" | 精神超越字面 |

## 关键模式

### 测试验证

```
  ✓ [运行测试命令] [看到：34/34 通过] "所有测试通过"
  ✗ "应该通过了" / "看起来正确"
```

### 回归测试的 Red-Green 验证

```
  ✓ 写 → 运行（通过）→ 还原修复 → 运行（必须失败）→ 恢复 → 运行（通过）
  ✗ "我写了回归测试"（没有 Red-Green 验证）
```

### 构建验证

```
  ✓ [运行构建] [看到：exit 0] "构建通过"
  ✗ "Linter 通过了"（Linter 不检查编译）
```

### 需求验证

```
  ✓ 重读计划 → 创建清单 → 逐项验证 → 报告缺口或完成
  ✗ "测试通过，阶段完成"
```

### Agent 委派验证

```
  ✓ Agent 报告成功 → 检查 VCS diff → 验证变更 → 报告实际状态
  ✗ 信任 Agent 报告
```

## 历史教训

技能中引用了 24 次失败记忆作为警示：

```
  从 24 次失败记忆中：
  ──────────────────
  - 用户说"我不相信你" — 信任已破坏
  - 未定义函数被提交 — 会导致崩溃
  - 缺失需求被提交 — 不完整功能
  - 时间浪费在虚假完成 → 重定向 → 返工
  - 违反原则："诚实是核心价值观。如果你撒谎，你会被替换。"
```

## 规则的适用范围

```
  始终在以下情况之前应用：
  ──────────────────
  - 任何形式的成功/完成声称
  - 任何满意表达
  - 任何关于工作状态的积极陈述
  - 提交、创建 PR、完成任务
  - 移到下一个任务
  - 委派给 Agent

  规则适用于：
  ──────────
  - 确切短语
  - 释义和同义词
  - 暗示成功的含义
  - 任何暗示完成/正确的沟通
```
