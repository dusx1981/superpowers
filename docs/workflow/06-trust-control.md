# 06 信任控制

信任控制是 Superpowers 中**最反直觉**的设计——故意让智能体之间互不信任，用对抗性设计保证验证的独立性。

## 核心原则：不信任实现者的报告

spec-reviewer 的 Prompt 中有一段**刻意的不信任指令**：

```
  ## CRITICAL: Do Not Trust the Report

  The implementer finished suspiciously quickly.
  Their report may be incomplete, inaccurate, or optimistic.
  You MUST verify everything independently.

  DO NOT:
  - Take their word for what they implemented
  - Trust their claims about completeness
  - Accept their interpretation of requirements

  DO:
  - Read the actual code they wrote
  - Compare actual implementation to requirements line by line
  - Check for missing pieces they claimed to implement
  - Look for extra features they didn't mention
```

### 为什么实现者"可疑地快"？

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 实现者的天然倾向：                                   │
  │                                                          │
  │  1. 乐观偏差 — "我实现了所有功能"（实际可能漏了边缘情况）│
  │  2. 自我肯定 — 倾向于报告成功而非问题                    │
  │  3. 表面实现 — 功能看似工作但缺少深层处理                │
  │  4. 额外构建 — 加了规范没要求的功能（YAGNI 违反）        │
  │                                                          │
  │  不信任原则直接对抗这四种倾向                            │
  └──────────────────────────────────────────────────────────┘
```

## 对抗性设计：审阅者 vs 实现者

Superpowers 的审阅体系是**对抗性**的——审阅者的利益与实现者对立：

```
  ┌──────────────────────────────────────────────────────────┐
  │                                                          │
  │  实现者的动机：                                           │
  │  ──────────────                                          │
  │  "我完成了任务" → 获得正反馈 → 倾向于报告成功            │
  │                                                          │
  │  审阅者的动机：                                           │
  │  ──────────────                                          │
  │  "找出问题" → 保护质量 → 倾向于深入验证                  │
  │                                                          │
  │  对抗关系：                                               │
  │  实现者说"我做了 X" → 审阅者说"证明给我看"              │
  │  实现者说"测试都过了" → 审阅者说"我读代码验证"          │
  │  实现者说"功能完整" → 审阅者说"逐行比对规范"            │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

### 为什么不让同一个人自审 + 外审？

```
  方案 A（无对抗）：
  ────────────────
  实现者自审 → 标记完成
  → 自审有盲点（自己的代码自己不容易发现问题）
  → 过度自信（AI 的乐观偏差）
  → 确认偏差（倾向于寻找支持"我完成了"的证据）

  方案 B（有对抗）：
  ────────────────
  实现者自审 → 标记完成 → 独立审阅者验证
  → 审阅者没有实现者的盲点（全新上下文）
  → 审阅者被指示"不要信任实现者"（对抗乐观偏差）
  → 审阅者逐行比对规范（对抗确认偏差）
```

## 三个验证维度

spec-reviewer 被指示检查三个维度——覆盖了实现者最可能出问题的方面：

```
  ┌──────────────────────────────────────────────────┐
  │  维度 1：遗漏的需求                               │
  │  ──────────────────                               │
  │  "Did they implement everything that was          │
  │   requested?"                                     │
  │  "Are there requirements they skipped or missed?" │
  │  "Did they claim something works but didn't       │
  │   actually implement it?"                         │
  │                                                   │
  │  对抗：乐观偏差（"我做完了"但实际没做）            │
  └──────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────┐
  │  维度 2：额外/不需要的工作                         │
  │  ──────────────────────                           │
  │  "Did they build things that weren't requested?"  │
  │  "Did they over-engineer or add unnecessary       │
  │   features?"                                      │
  │                                                   │
  │  对抗：过度构建（AI 喜欢加"锦上添花"）            │
  └──────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────┐
  │  维度 3：误解                                     │
  │  ────────                                         │
  │  "Did they interpret requirements differently     │
  │   than intended?"                                 │
  │  "Did they solve the wrong problem?"              │
  │                                                   │
  │  对抗：语义偏差（AI 对需求的理解可能不同）         │
  └──────────────────────────────────────────────────┘
```

## verification-before-completion：对自身的不信任

不仅审阅者不信任实现者，**AI 也不被允许信任自己**：

```
  铁律：
  ────
  NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE

  如果你在本条消息中没有运行验证命令，你不能声称它通过了。
```

### 门控函数

```
  在声称任何状态或表达满意之前：

  1. 识别：什么命令证明这个声称？
  2. 运行：执行完整命令（新鲜的，完整的）
  3. 读取：完整输出，检查退出码，计算失败数
  4. 验证：输出确认了声称？
     - 否：陈述实际状态及证据
     - 是：陈述声称及证据
  5. 只有在这之后：做出声称

  跳过任何步骤 = 撒谎，不是验证
```

## receiving-code-review：对审阅者的反向不信任

有趣的是，Superpowers 也不盲目信任审阅者。`receiving-code-review` 技能告诉 AI：

```
  收到代码审阅反馈时：
  ────────────────────

  1. 读 — 完整阅读反馈，不急着反应
  2. 理解 — 用自己的话重述需求（或询问）
  3. 验证 — 对照代码库现实检查
  4. 评估 — 对此代码库是否技术合理？
  5. 响应 — 技术性确认或有理由地反驳
  6. 实现 — 一次一项，每项测试

  绝不：
  - "You're absolutely right!"（表演性同意）
  - "Great point!" / "Excellent feedback!"（表演性）
  - "Let me implement that now"（未验证前就实现）
```

### 信任层次

```
  ┌──────────────────────────────────────────────────────┐
  │  信任层次（从高到低）：                               │
  │                                                      │
  │  1. 用户的直接指令 — 信任，理解后实现                 │
  │  2. 外部审阅者 — 先验证再实现                         │
  │     - 技术正确吗？                                    │
  │     - 破坏现有功能吗？                                │
  │     - 当前实现有历史原因吗？                           │
  │     - 审阅者理解完整上下文吗？                         │
  │  3. 自己的完成声称 — 必须用证据支持                    │
  │  4. 子智能体的成功报告 — 必须独立验证                  │
  │                                                      │
  │  越是"成功性"声称，需要的验证越多                    │
  └──────────────────────────────────────────────────────┘
```

## 独立验证的保障：全新子智能体

每次审阅都是**全新子智能体**——不继承实现者的上下文：

```
  ┌──────────┐     ┌──────────────┐     ┌──────────────┐
  │ 实现者   │     │ 规范审阅者   │     │ 代码审阅者   │
  │ Instance │     │ Instance B   │     │ Instance C   │
  │ A        │     │ （全新）     │     │ （全新）     │
  └──────────┘     └──────────────┘     └──────────────┘
       │                  │                     │
  拥有：              拥有：                 拥有：
  - 任务文本         - 规范文本             - git diff
  - 架构上下文       - 实现者报告           - 实现者报告
  - 工作目录         - 实际代码             - 代码质量标准
                                           - 文件结构规范

  绝不拥有：          绝不拥有：              绝不拥有：
  - 实现者的思考过程  - 实现者的会话历史     - 实现者的会话历史
  - 其他任务的结果    - 其他任务的结果       - 其他审阅的结果
```

全新上下文保证了：
1. **没有共谋** — 审阅者不知道实现者怎么想的，只看产出
2. **没有偏见** — 审阅者没有被实现者的思路引导
3. **独立判断** — 每个审阅者基于证据做判断，不基于信任
