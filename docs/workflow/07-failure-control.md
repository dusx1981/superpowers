# 07 失败控制

失败控制确保 AI 在遇到困难时**不会盲目重试**，而是按照升级链改变条件后重新尝试。

## 四种状态：结构化失败信号

实现者子智能体必须报告以下四种状态之一，Controller 根据状态决定下一步：

```
  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  DONE                                                    │
  │  ┌─────────────────────────────────────────────────────┐│
  │  │ 完成了任务，没有疑虑                                 ││
  │  │ Controller → 派遣 spec-reviewer                      ││
  │  └─────────────────────────────────────────────────────┘│
  │                                                         │
  │  DONE_WITH_CONCERNS                                      │
  │  ┌─────────────────────────────────────────────────────┐│
  │  │ 完成了任务，但对正确性有疑虑                         ││
  │  │ Controller → 先读疑虑，再决定下一步                   ││
  │  │   - 关于正确性/范围 → 先解决再审阅                    ││
  │  │   - 只是观察（如"文件变大了"）→ 记录，继续审阅        ││
  │  └─────────────────────────────────────────────────────┘│
  │                                                         │
  │  NEEDS_CONTEXT                                           │
  │  ┌─────────────────────────────────────────────────────┐│
  │  │ 缺少完成所需的信息                                   ││
  │  │ Controller → 提供缺的上下文，重新派遣（同一模型）      ││
  │  └─────────────────────────────────────────────────────┘│
  │                                                         │
  │  BLOCKED                                                 │
  │  ┌─────────────────────────────────────────────────────┐│
  │  │ 无法完成任务                                         ││
  │  │ Controller → 评估阻塞原因（见升级链）                ││
  │  └─────────────────────────────────────────────────────┘│
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

### 为什么需要结构化状态？

```
  无结构化状态（自由文本报告）：
  ──────────────────────────────
  实现者："我完成了大部分功能，但有个地方我不太确定..."
  → Controller 需要解析自然语言来判断状态
  → 可能误判（"大部分完成"= DONE 还是 DONE_WITH_CONCERNS？）
  → 可能忽略重要信号（"不太确定"可能是 BLOCKED）

  有结构化状态：
  ────────────
  实现者："Status: DONE_WITH_CONCERNS"
  → Controller 程序化地判断下一步
  → 无歧义：DONE_WITH_CONCERNS 的处理方式是固定的
  → 可追踪：状态记录在 TodoWrite 中
```

## 升级链：改变条件后重试

Controller 处理 BLOCKED 状态时，按升级链逐步改变条件：

```
  ┌──────────────────────────────────────────────────────────┐
  │                     升级链                                │
  │                                                          │
  │  实现者报告 NEEDS_CONTEXT                                 │
  │       │                                                  │
  │       ▼                                                  │
  │  Controller 提供更多上下文                                 │
  │       │                                                  │
  │       ▼                                                  │
  │  重新派遣（同一模型）                                      │
  │       │                                                  │
  │       ├── 成功 → 继续                                     │
  │       │                                                  │
  │       └── 仍失败 / 报告 BLOCKED                            │
  │              │                                           │
  │              ▼                                           │
  │  Controller 换更强模型重新派遣                              │
  │       │                                                  │
  │       ├── 成功 → 继续                                     │
  │       │                                                  │
  │       └── 仍失败                                           │
  │              │                                           │
  │              ▼                                           │
  │  Controller 拆分任务为更小任务                              │
  │       │                                                  │
  │       ├── 成功 → 继续                                     │
  │       │                                                  │
  │       └── 仍失败                                           │
  │              │                                           │
  │              ▼                                           │
  │  升级给用户（计划本身可能有问题）                            │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

### 为什么禁止无条件重试？

```
  AI 的倾向（无控制）：
  ────────────────────
  实现者失败了 → "再试一次" → 又失败了 → "再试一次"...
  → 浪费 token 和时间
  → 不改变条件 = 同样结果
  → 可能陷入死循环

  Superpowers 的控制：
  ────────────────────
  "绝不强迫同一模型无条件重试。
   实现者说卡住了，必须改变什么。"

  每次重试必须改变至少一个条件：
  1. 更多上下文（信息维度）
  2. 更强模型（能力维度）
  3. 更小任务（复杂度维度）
  4. 升级给人（权威维度）
```

## 实现者的升级触发条件

实现者的 Prompt 模板中明确列出了应该升级的情况：

```
  ## When You're in Over Your Head

  停下来升级是可以的。坏工作比没工作更糟。

  STOP and escalate when:
  - 任务需要有多种有效方法的架构决策
  - 需要理解超出提供范围的代码且无法找到清晰度
  - 对方法是否正确感到不确定
  - 任务涉及计划未预期的现有代码重构
  - 读了一个又一个文件仍无法理解系统

  How to escalate:
  报告 BLOCKED 或 NEEDS_CONTEXT
  + 具体卡在哪
  + 试过什么
  + 需要什么帮助
```

### "坏工作比没工作更糟"

```
  ┌──────────────────────────────────────────────────────┐
  │  这是 Superpowers 反复出现的主题：                    │
  │                                                      │
  │  - TDD："先写代码？删掉重来" — 错误代码 < 无代码    │
  │  - 升级："坏工作比没工作更糟" — 错误实现 < 停下来   │
  │  - 验证："撒谎，不是效率" — 未验证声称 < 不声称     │
  │  - 调试："3+ 失败就停" — 继续猜 < 质疑架构          │
  │                                                      │
  │  统一原则：宁可少做也不做错                            │
  └──────────────────────────────────────────────────────┘
```

## 模型选择：成本与能力的平衡

升级链中的"换更强模型"涉及模型选择策略：

```
  任务复杂度               模型选择           成本
  ─────────────────────────────────────────────────
  机械实现                 快速便宜模型        低
  （1-2 文件，规范完整）

  集成判断                 标准模型            中
  （多文件协调、模式匹配）

  架构/设计/审阅           最强模型            高
  （需要广泛代码库理解）

  升级重试                 比上次更强的模型     递增
  （上次失败的原因）
```

### 复杂度信号

```
  触碰 1-2 文件 + 完整规范     → 便宜模型足够
  触碰多个文件 + 集成关注      → 标准模型
  需要设计判断 + 广泛代码库理解 → 最强模型
  之前模型失败了               → 升级到更强模型
```

## systematic-debugging 的失败控制

调试场景有专门的失败控制——3+ 修复失败后质疑架构：

```
  修复 1：假设 A，最小修改 → 失败
       │
       ▼
  回到 Phase 1（根因调查），用新信息重新分析
       │
       ▼
  修复 2：假设 B，最小修改 → 失败
       │
       ▼
  回到 Phase 1，再分析
       │
       ▼
  修复 3：假设 C，最小修改 → 失败
       │
       ▼
  ┌──────────────────────────────────────────────────────┐
  │  3 次修复都失败了 → 停下来！                         │
  │                                                      │
  │  这不是假设失败——这是架构有问题。                     │
  │                                                      │
  │  模式：                                              │
  │  - 每次修复暴露新的共享状态/耦合/问题在别处          │
  │  - 修复需要"大规模重构"才能实现                      │
  │  - 每次修复在别处创造新症状                          │
  │                                                      │
  │  行动：与用户讨论，不再尝试更多修复                   │
  └──────────────────────────────────────────────────────┘
```

### 为什么是 3 次而不是 2 次或 5 次？

3 次是一个经验阈值：
- 1-2 次失败可能只是假设错误，换假设可能成功
- 3 次失败开始呈现模式——"每个修复都在别处出新问题"
- 5+ 次失败已经浪费了大量时间

## 审阅循环的失败控制

审阅不通过时的失败控制也是结构化的：

```
  规范审阅者报告：❌ Issues found
       │
       ▼
  Controller 让实现者修复
       │
       ▼
  重新派遣规范审阅者（必须是新的实例）
       │
       ▼
  规范审阅者重新审阅
       │
    通过？── 否 → 回到修复
       │
      是
       ▼
  代码质量审阅者审阅
       │
    通过？── 否 → 回到修复
       │
      是
       ▼
  标记完成
```

关键控制：**审阅不通过 ≠ 跳过审阅**。必须重新审阅直到通过。不能"看起来修了就过"。
