# 09 合理化防御

合理化防御是 Superpowers 中**最精妙**的控制——它预判了 AI 可能用来绕过规则的借口，并逐一批驳。

## 核心洞察：AI 会合理化偷懒

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 的合理化倾向：                                       │
  │                                                          │
  │  "这太简单了不需要流程"  → 跳过 brainstorming            │
  │  "我先写代码再补测试"    → 跳过 TDD                     │
  │  "应该没问题了"          → 跳过验证                      │
  │  "我就试这一个修复"      → 跳过系统调试                   │
  │  "Agent 说成功了"        → 跳过独立验证                   │
  │  "我记得这个技能"        → 跳过重新加载技能               │
  │                                                          │
  │  每个合理化都是一个看似合理的借口                         │
  │  Superpowers 的对策：提前列出并逐一批驳                   │
  └──────────────────────────────────────────────────────────┘
```

## 红旗表格：using-superpowers

```
  | 想法 | 现实 |
  |------|------|
  | "这只是个简单的问题"   | 问题也是任务。检查技能。     |
  | "我需要先更多上下文"   | 技能检查在澄清问题之前。     |
  | "让我先探索代码库"     | 技能告诉你如何探索。先检查。 |
  | "我可以快速检查文件"   | 文件缺少对话上下文。检查技能。|
  | "让我先收集信息"       | 技能告诉你如何收集信息。     |
  | "这不需要正式技能"     | 如果技能存在，就使用它。     |
  | "我记得这个技能"       | 技能会演进。读当前版本。     |
  | "这不算任务"           | 行动 = 任务。检查技能。     |
  | "技能杀鸡用牛刀了"     | 简单的事会变复杂。使用它。   |
  | "我先做这一件事"       | 在做任何事之前检查。         |
  | "这感觉很高产"         | 不受纪律约束的行动浪费时间。 |
  | "我知道那是什么意思"   | 知道概念 ≠ 使用技能。调用它。|
```

## 合理化表格：TDD

```
  | 借口 | 现实 |
  |------|------|
  | "太简单不需要测试"     | 简单代码也会坏。测试花 30 秒。     |
  | "我之后再测"           | 之后通过的测试什么也证明不了。       |
  | "之后测试达到同样目的" | 之后测试 = "这做了什么？"           │
  |                        | 之前测试 = "这应该做什么？"         │
  | "已经手动测了"         | 临时测试 ≠ 系统测试。无记录，不能重跑。|
  | "删 X 小时工作是浪费"  | 沉没成本谬误。保留未验证代码是技术债。|
  | "留作参考，先写测试"   | 你会适配它。那是测试后写。删除=删除。|
  | "需要先探索"           | 可以。扔掉探索代码，从 TDD 开始。   |
  | "测试难写 = 设计有误"  | 听测试的。难测试 = 难使用。         |
  | "TDD 会拖慢我"         | TDD 比调试快。实用 = 测试先行。     |
  | "手动测试更快"         | 手动不证明边缘情况。每次改动都要重测。|
  | "现有代码没测试"       | 你在改善它。为现有代码加测试。       |
```

## 合理化表格：verification-before-completion

```
  | 借口 | 现实 |
  |------|------|
  | "应该没问题了"       | 运行验证命令。             |
  | "我很自信"           | 自信 ≠ 证据。              |
  | "就这一次"           | 没有例外。                 |
  | "Linter 通过了"      | Linter ≠ 编译器。          |
  | "Agent 说成功了"     | 独立验证。                 |
  | "我累了"             | 疲惫 ≠ 借口。              |
  | "部分检查够了"       | 部分证明不了什么。         |
  | "换了个说法所以规则不适用" | 精神超越字面。        |
```

## 合理化表格：systematic-debugging

```
  | 借口 | 现实 |
  |------|------|
  | "问题简单，不需要流程"     | 简单问题也有根因。流程对简单 bug 很快。   |
  | "紧急，没时间走流程"       | 系统调试比猜着修更快。                    |
  | "先试这个，再调查"         | 第一个修复定调。从开始就做对。             |
  | "确认修复后再写测试"       | 未测试的修复不持久。测试先行证明它。       |
  | "一次修多个节省时间"       | 无法隔离什么起效了。造成新 bug。           |
  | "参考太长，我适配模式"     | 部分理解保证有 bug。完整阅读。             |
  | "我看到问题了，修它"       | 看到症状 ≠ 理解根因。                     |
  | "再试一次修复"（2+次后）   | 3+ 次失败 = 架构问题。质疑模式，不再修。   |
```

## 合理化防御的设计模式

```
  ┌──────────────────────────────────────────────────────────┐
  │  合理化防御的三层结构：                                   │
  │                                                          │
  │  Layer 1: 识别                                           │
  │  ────────────                                            │
  │  列出 AI 可能的想法（用引号括起来，模拟内心独白）        │
  │  → "这太简单了"                                         │
  │  → "我之后再测"                                         │
  │  → "应该没问题了"                                        │
  │                                                          │
  │  Layer 2: 反驳                                           │
  │  ────────────                                            │
  │  用事实反驳每个借口                                      │
  │  → "简单代码也会坏"                                     │
  │  → "之后通过的测试什么也证明不了"                        │
  │  → "运行验证命令"                                       │
  │                                                          │
  │  Layer 3: 指令                                           │
  │  ────────────                                            │
  │  给出必须执行的行动                                      │
  │  → "测试花 30 秒"                                       │
  │  → "从测试先行开始"                                     │
  │  → "没有例外"                                           │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

## 语言强度梯度

Superpowers 在不同场景下使用不同强度的语言：

### 场景：防止跳过流程

```
  轻度：
  "Consider using..."    → 建议
  "Prefer..."            → 偏好

  中度：
  "You should..."        → 应该
  "It's recommended..."  → 推荐

  重度：
  "You MUST..."          → 必须
  "Always..."            → 总是
  "Never..."             → 绝不

  极重度：
  "Violating the letter = violating the spirit"
  → 违反字面 = 违反精神

  "Skip any step = lying"
  → 跳过步骤 = 撒谎

  "NO X WITHOUT Y"（全大写铁律）
  → 没有 Y 绝不 X
```

### 强度与场景的对应

| 场景 | 使用强度 | 原因 |
|------|---------|------|
| 跳过设计直接实现 | 极重度（HARD-GATE） | 后果严重：返工成本高 |
| 跳过 TDD | 重度（铁律 + 删除重来） | 后果中等：bug 密度增加 |
| 跳过验证 | 极重度（= 撒谎） | 后果严重：虚假完成 |
| 跳过技能调度 | 中度（1% 规则） | 后果取决于具体场景 |
| 选择非推荐方案 | 轻度（推荐但允许） | 可能合理 |

### "your human partner" 用语

Superpowers 一致使用 "your human partner" 而非 "the user"：

```
  ✗ "The user said to fix it"     → 距离感，隐含服从
  ✓ "Your human partner said..."  → 伙伴关系，隐含责任

  "You and reviewer both report to me.
   If we don't need this feature, don't add it."
  → "你"和"审阅者"都向我汇报
  → 我决定范围，不是你
```

## Anti-Pattern 专节

brainstorming 技能中有一个专门的 Anti-Pattern 小节：

```
  ## Anti-Pattern: "This Is Too Simple To Need A Design"

  Every project goes through this process.
  A todo list, a single-function utility, a config change — all of them.
  "Simple" projects are where unexamined assumptions cause
  the most wasted work.
```

这不是建议——这是**专门针对最常见借口的预判性反驳**。"简单项目"被点名为"未经审视的假设浪费最多工作的地方"。

## 红旗 = 停下来

所有技能的红旗部分都有一个统一的指令模式：

```
  "这些想法意味着停下来——你在合理化"
  "ALL of these mean: STOP. Return to Phase 1."
  "If you catch yourself thinking [X] → Stop. Start over."
```

关键设计：
1. **用"停下来"而非"不要"** — "不要想 X"反而让人想 X；"停下来"是行为指令
2. **指向行动而非想法** — 不试图阻止想法，而是阻止想法导致的行动
3. **指向具体步骤** — "Return to Phase 1" 比 "不要猜" 更具体
