# 智能体协作模式

Superpowers 的智能体不是孤立工作的，而是通过 Controller 编排形成**多种协作模式**。本章梳理这些模式的结构、适用场景和约束。

## 模式一：串行审阅流水线

这是最核心的协作模式——实现者 → 规范审阅者 → 代码审阅者的严格有序流水线。

```
  ┌──────────┐     ┌──────────────┐     ┌──────────────┐
  │ 实现者   │────▶│ 规范审阅者   │────▶│ 代码审阅者   │
  │Implementer│    │Spec Reviewer │    │Code Reviewer │
  └──────────┘     └──────────────┘     └──────────────┘
       │                  │                     │
       │              不通过？               不通过？
       │                  │                     │
       │                  ▼                     ▼
       │◀───────────── 实现者修复 ◀────────── 实现者修复
       │                  │                     │
       │              重新审阅               重新审阅
       │                  │                     │
       │               通过                    通过
       │                  │                     │
       │                  ▼                     ▼
       │             ✅ 规范合规            ✅ 代码质量
       │                                       │
       └───────────────────────────────────────┘
                            │
                            ▼
                      标记任务完成

  关键约束：
  1. 顺序不可颠倒 — 规范审阅必须先于代码审阅
  2. 不通过必须重审 — 实现者修复后，审阅者必须重新审阅
  3. 每步都是全新子智能体 — 审阅者不继承实现者上下文
```

### 为什么顺序不能颠倒？

```
  ✗ 错误顺序：代码审阅 → 规范审阅
  问题：代码写得再漂亮，做错了事也没用
  例：完美实现了用户注册，但规范要求的是用户登录

  ✓ 正确顺序：规范审阅 → 代码审阅
  先确认"做对了事"，再确认"把事做对"
  例：确认实现了用户登录 → 再检查代码质量
```

### 审阅循环的闭环保证

```
  审阅发现问题
       │
       ▼
  Controller 把问题反馈给实现者子智能体
       │
       ▼
  实现者修复
       │
       ▼
  Controller 重新派遣同一个审阅者模板
  （注意：是新的子智能体实例，但使用相同的审阅模板）
       │
       ▼
  审阅者重新审阅 ←── 必须重新审阅！不能"看起来修了就过"
       │
    通过？── 否 → 回到修复
       │
      是
       ▼
    下一步
```

## 模式二：最终整体审阅

所有任务完成后，Controller 派遣一个**跨任务整体审阅子智能体**。

```
  Task 1: ✅ 完成
  Task 2: ✅ 完成
  Task 3: ✅ 完成
  Task 4: ✅ 完成
       │
       ▼
  ┌──────────────────────────────────────┐
  │        最终代码审阅子智能体            │
  │                                      │
  │  视角：整个实现，而非单个任务          │
  │  检查：                              │
  │  - 跨任务一致性                      │
  │  - 任务间的集成问题                   │
  │  - 全局架构合理性                     │
  │  - 整体代码质量                       │
  └──────────────────────────────────────┘
       │
       ▼
  finishing-a-development-branch
```

## 模式三：并行调查

当有多个独立问题时，Controller 使用 `dispatching-parallel-agents` 技能并行派遣多个调查子智能体。

```
  ┌──────────────────────────────────────────────────┐
  │                 Controller                        │
  │                                                  │
  │  6 个测试失败，3 个文件，不同根因                  │
  │                                                  │
  │  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
  │  │ Agent A  │  │ Agent B  │  │ Agent C  │       │
  │  │修复      │  │修复      │  │修复      │       │
  │  │abort.test│  │batch.test│  │race.test │       │
  │  └──────────┘  └──────────┘  └──────────┘       │
  │       │              │              │             │
  │       └──────────────┴──────────────┘             │
  │                     │                             │
  │                     ▼                             │
  │            审阅每个摘要                            │
  │            检查冲突                                │
  │            运行完整测试套件                         │
  │            抽查结果                                │
  └──────────────────────────────────────────────────┘
```

### 适用条件

```
  ✓ 3+ 个独立问题（不同文件、不同子系统、不同根因）
  ✓ 各问题之间无共享状态
  ✓ 不编辑相同文件
  ✗ 问题是相关的（修一个可能修另一个 → 先一起调查）
  ✗ 需要理解完整系统状态
  ✗ 会编辑相同代码
```

### Agent Prompt 要求

```
  好的并行 agent prompt：
  ─────────────────────
  1. 聚焦 — 一个清晰的问题域
  2. 自包含 — 理解问题所需的全部上下文
  3. 明确输出 — agent 应该返回什么

  ❌ "Fix all the tests"           → agent 迷失
  ✓ "Fix agent-tool-abort.test.ts" → 聚焦范围

  ❌ "Fix the race condition"      → agent 不知道在哪
  ✓ 粘贴错误信息和测试名            → 有具体上下文

  ❌ 无约束                         → agent 可能重构一切
  ✓ "Do NOT change production code" → 有边界

  ❌ "Fix it"                       → 不知道改了什么
  ✓ "Return summary of root cause"  → 明确输出
```

### 结果集成

```
  并行 Agent 返回后：
  ─────────────────
  1. 审阅每个摘要 — 理解改了什么
  2. 检查冲突 — Agent 是否编辑了相同代码？
  3. 运行完整测试套件 — 验证所有修复一起工作
  4. 抽查 — Agent 可能犯系统性错误
```

## 模式四：升级链

当子智能体遇到困难时，Controller 按照升级链处理：

```
  ┌──────────────────────────────────────────────────────────┐
  │                     升级链                                │
  │                                                          │
  │  实现者报告                                               │
  │  NEEDS_CONTEXT                                            │
  │       │                                                  │
  │       ▼                                                  │
  │  Controller 提供更多上下文                                 │
  │       │                                                  │
  │       ▼                                                  │
  │  重新派遣（同一模型）                                      │
  │       │                                                  │
  │       ├── 成功 → 继续                                     │
  │       │                                                  │
  │       └── 仍失败/报告 BLOCKED                              │
  │              │                                           │
  │              ▼                                           │
  │  Controller 换更强模型重新派遣                              │
  │       │                                                  │
  │       ├── 成功 → 继续                                     │
  │       │                                                  │
  │       └── 仍失败                                           │
  │              │                                           │
  │              ▼                                           │
  │  Controller 拆分任务为更小任务                              │
  │       │                                                  │
  │       ├── 成功 → 继续                                     │
  │       │                                                  │
  │       └── 仍失败                                           │
  │              │                                           │
  │              ▼                                           │
  │  升级给用户（计划本身可能有问题）                            │
  │                                                          │
  │  ⚠️ 绝不：                                                │
  │  - 忽略升级请求                                            │
  │  - 强迫同一模型无条件重试                                   │
  │  - 跳过升级步骤直接给人                                    │
  └──────────────────────────────────────────────────────────┘
```

## 模式五：技能引导的智能体行为

同一个 AI 模型在不同技能指导下扮演完全不同的角色：

```
  ┌──────────────────┐
  │    AI 模型实例     │
  └────────┬─────────┘
           │
     加载不同技能
           │
   ┌───────┼────────┬───────────┬──────────────┐
   ▼       ▼        ▼           ▼              ▼
┌──────┐┌──────┐┌────────┐┌──────────┐┌──────────┐
│brain ││debug ││impl    ││reviewer  ││planner   │
│storm ││ger   ││ementer ││          ││          │
└──────┘└──────┘└────────┘└──────────┘└──────────┘
 角色：   角色：   角色：     角色：       角色：
 提问者   调查者   执行者     验证者       设计者
 好奇    系统    TDD       不信任       原子化
 耐心    严谨    自审       独立验证     零上下文

  关键洞察：智能体不是不同的代码，而是同一模型在不同技能下的不同行为模式
```

## 协作矩阵

| 智能体 | 输入来源 | 输出目标 | 与谁协作 |
|--------|---------|---------|---------|
| Controller | 用户 + 技能 + 计划 | 子智能体 Prompt + TodoWrite | 所有子智能体 |
| Implementer | Controller 构造的 Prompt | 代码 + 测试 + 提交 + 状态报告 | Spec Reviewer, Code Reviewer |
| Spec Reviewer | 规范文本 + 实现者报告 + 实际代码 | 合规判定 | Implementer（修复循环） |
| Code Reviewer | git diff + 计划 + 实现者报告 | 质量判定 | Implementer（修复循环） |
| 并行 Agent | 聚焦的问题描述 + 上下文 | 修复摘要 | Controller（集成） |
| Code Reviewer（持久化） | 计划 + 实现 + 需求 | 审阅报告 | 用户/技能 |
