# 规范合规审阅者（Spec Reviewer）

规范合规审阅者是 Superpowers **两阶段审阅的第一道关**——验证实现者做了正确的事（没有多也没有少）。

## 身份

```
  ┌─────────────────────────────────────────────────┐
  │           规范合规审阅者子智能体                    │
  │                                                  │
  │  角色：验证实现是否匹配规范                        │
  │  核心指令："不要信任实现者的报告！"                 │
  │  方法：读实际代码，逐行比对规范                     │
  │  模板：skills/subagent-driven-development/       │
  │         spec-reviewer-prompt.md                  │
  │                                                  │
  │  生命周期：单次审阅，审完即销毁                     │
  │  必须先于 code-quality-reviewer                   │
  └─────────────────────────────────────────────────┘
```

## 使用的技能

本智能体不使用其他技能。它是一个纯验证角色，专注于规范比对。

## 不信任原则

这是 spec-reviewer 的**灵魂**：

```
  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  "实现者完成得可疑地快。他们的报告可能不完整、          │
  │   不准确或乐观。你必须独立验证一切。"                    │
  │                                                         │
  │  DO NOT:                                                 │
  │  - 采信实现者对实现的描述                                │
  │  - 信任实现者关于完整性的声称                            │
  │  - 接受实现者对需求的解读                                │
  │                                                         │
  │  DO:                                                     │
  │  - 读实现者写的实际代码                                  │
  │  - 逐行比对实际实现与需求                                │
  │  - 检查实现者声称实现但实际缺失的部分                    │
  │  - 寻找实现者没提到的额外功能                            │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

## 三个验证维度

```
  ┌──────────────────────────────────────────┐
  │                                          │
  │  1. 遗漏的需求                            │
  │     ────────────                          │
  │     实现了所有请求的内容？                 │
  │     跳过或遗漏了什么？                     │
  │     声称实现了但实际没做？                 │
  │                                          │
  │  2. 额外/不需要的工作                     │
  │     ────────────────                      │
  │     构建了未请求的东西？                   │
  │     过度工程或添加了不必要功能？           │
  │     添加了规范外的"锦上添花"？            │
  │                                          │
  │  3. 误解                                  │
  │     ────                                  │
  │     对需求的解读与意图不同？               │
  │     解决了错误的问题？                     │
  │     做了正确的功能但用了错误的方式？       │
  │                                          │
  └──────────────────────────────────────────┘
```

## Prompt 结构

来源：`skills/subagent-driven-development/spec-reviewer-prompt.md`

```
  Task tool (general-purpose):
    description: "Review spec compliance for Task N"
    prompt: |
      你正在审阅实现是否匹配规范。

      ## What Was Requested
      [完整任务需求文本]

      ## What Implementer Claims They Built
      [来自实现者的报告]

      ## CRITICAL: Do Not Trust the Report
      [不信任原则...]

      ## Your Job
      [三个验证维度...]

      Report:
      - ✅ Spec compliant（代码检查后一切匹配）
      - ❌ Issues found: [具体列出缺失或额外项，附 file:line]
```

## 输出格式

```
  ✅ Spec compliant           → 进入代码质量审阅
  ❌ Issues found:            → 实现者修复 → 重新派遣 spec-reviewer
     - Missing: [what, file:line]
     - Extra: [what, file:line]
     - Misunderstood: [what, file:line]
```

## 在审阅循环中的位置

```
  实现者完成
       │
       ▼
  ┌──────────────┐
  │spec-reviewer │  ← 第一道关
  └──────┬───────┘
         │
    通过？── 否 → 实现者修复 → 重新派遣 spec-reviewer（必须重新审阅！）
         │
        是
         │
         ▼
  ┌─────────────────┐
  │code-quality-rev │  ← 第二道关（仅在 spec-reviewer 通过后）
  └─────────────────┘

  关键：顺序不可颠倒！
  代码写得再漂亮，做错了事也没用
```
