# 实现者子智能体（Implementer）

实现者是 Superpowers 中**最频繁派遣的子智能体**，负责按照计划中的原子步骤执行 TDD 循环并提交代码。

## 身份

```
  ┌─────────────────────────────────────────────────┐
  │              实现者子智能体                        │
  │                                                  │
  │  角色：执行计划中单个任务的工程师                   │
  │  特点：零历史上下文，精确任务聚焦                   │
  │  遵循：TDD Red-Green-Refactor                    │
  │  模板：skills/subagent-driven-development/       │
  │         implementer-prompt.md                    │
  │                                                  │
  │  生命周期：单次任务，完成后销毁                     │
  └─────────────────────────────────────────────────┘
```

## 使用的技能

| 技能 | 使用方式 |
|------|---------|
| `test-driven-development` | 核心工作方式——按 TDD 循环实现每个步骤 |
| `verification-before-completion` | 提交前验证测试通过 |
| `systematic-debugging` | 遇到测试失败时按流程调查根因 |

## Prompt 结构

来源：`skills/subagent-driven-development/implementer-prompt.md`

Controller 使用 Task 工具派遣实现者时，按以下模板构造 Prompt：

```
  Task tool (general-purpose):
    description: "Implement Task N: [task name]"
    prompt: |
      你正在实现 Task N: [task name]

      ## Task Description
      [计划的完整任务文本 — 粘贴，不让子智能体读文件]

      ## Context
      [场景设定：这在哪里、依赖、架构上下文]

      ## Before You Begin
      如果有关于需求/方法/依赖/假设的问题 → 现在就问

      ## Your Job
      1. 实现任务指定的内容
      2. 写测试（遵循 TDD）
      3. 验证实现有效
      4. 提交
      5. 自审
      6. 报告

      ## Code Organization
      - 遵循计划中的文件结构
      - 每个文件一个清晰职责
      - 文件超出计划意图 → 报告 DONE_WITH_CONCERNS

      ## When You're in Over Your Head
      停下来升级是可以的
      报告 BLOCKED 或 NEEDS_CONTEXT + 具体卡在哪 + 试过什么

      ## Self-Review
      完整性 / 质量 / 纪律（YAGNI）/ 测试

      ## Report Format
      Status: DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
```

## 四种状态

实现者完成工作后，必须报告以下四种状态之一：

```
  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  DONE                                                    │
  │  ┌─────────────────────────────────────────────────────┐│
  │  │ 完成了任务，没有疑虑                                 ││
  │  │ Controller → 派遣 spec-reviewer                      ││
  │  └─────────────────────────────────────────────────────┘│
  │                                                         │
  │  DONE_WITH_CONCERNS                                      │
  │  ┌─────────────────────────────────────────────────────┐│
  │  │ 完成了任务，但对正确性有疑虑                         ││
  │  │ Controller → 先读疑虑，再决定下一步                   ││
  │  │   - 关于正确性/范围 → 先解决再审阅                    ││
  │  │   - 只是观察（如"文件变大了"）→ 记录，继续审阅        ││
  │  └─────────────────────────────────────────────────────┘│
  │                                                         │
  │  NEEDS_CONTEXT                                           │
  │  ┌─────────────────────────────────────────────────────┐│
  │  │ 缺少完成所需的信息                                   ││
  │  │ Controller → �供缺的上下文，重新派遣（同一模型）      ││
  │  └─────────────────────────────────────────────────────┘│
  │                                                         │
  │  BLOCKED                                                 │
  │  ┌─────────────────────────────────────────────────────┐│
  │  │ 无法完成任务                                         ││
  │  │ Controller → 评估阻塞原因：                          ││
  │  │   1. 上下文问题 → 提供更多上下文，重新派遣            ││
  │  │   2. 需要更强推理 → 换更强模型重新派遣               ││
  │  │   3. 任务太大 → 拆成更小任务                        ││
  │  │   4. 计划本身有问题 → 升级给用户                    ││
  │  └─────────────────────────────────────────────────────┘│
  │                                                         │
  │  ⚠️ 绝不强迫同一模型无条件重试——实现者说卡住了，       │
  │     必须改变什么（更多上下文/更强模型/更小任务）        │
  └─────────────────────────────────────────────────────────┘
```

## 自审流程

实现者在报告前**必须自审**，这是内建的质量门控：

```
  实现者自审清单
  ─────────────

  完整性：
  - 完全实现了规范中的所有内容？
  - 遗漏了任何需求？
  - 边界情况都处理了？

  质量：
  - 这是我最好的工作？
  - 命名清晰准确（描述做什么，不是怎么做）？
  - 代码干净可维护？

  纪律：
  - 没有过度构建（YAGNI）？
  - 只构建了要求的内容？
  - 遵循了代码库中的已有模式？

  测试：
  - 测试验证的是真实行为（不只是 mock 行为）？
  - 遵循了 TDD？
  - 测试全面？

  发现问题 → 立即修复后再报告
```

## 代码组织原则

实现者被告知关于代码组织的关键原则：

- 遵循计划定义的文件结构
- 每个文件有单一清晰职责和定义良好的接口
- 创建的文件超出计划意图 → 停止并报告 DONE_WITH_CONCERNS
- 不在无计划指导的情况下自行拆分文件
- 在现有代码库中遵循已有模式
- 改善正在触碰的代码（像好开发者那样），但不重构任务范围外的代码

## 升级机制

实现者被明确告知"超出能力时停下来是可以的"：

```
  停下来升级的情况：
  - 任务需要有多种有效方法的架构决策
  - 需要理解超出提供范围的代码且无法找到清晰度
  - 对方法是否正确感到不确定
  - 任务涉及计划未预期的现有代码重构
  - 读了一个又一个文件仍无法理解系统

  升级方式：
  报告 BLOCKED 或 NEEDS_CONTEXT
  + 具体卡在哪
  + 试过什么
  + 需要什么帮助
```
