# Controller 主智能体

Controller 是 Superpowers 体系中的**唯一常驻智能体**——运行在用户会话中的 AI 实例，在 `using-superpowers` 技能的指导下执行所有协调工作。

## 身份

```
  ┌─────────────────────────────────────────────────┐
  │              Controller 主智能体                   │
  │                                                  │
  │  不是"另一个智能体"，而是唯一的协调者              │
  │  拥有全局上下文（会话历史、项目状态、技能知识）     │
  │  不做实现工作——把实现委托给子智能体                │
  │                                                  │
  │  激活方式：SessionStart Hook 注入 using-superpowers│
  │  技能到每轮对话开头                               │
  └─────────────────────────────────────────────────┘
```

## 核心职责

### 1. 技能调度

Controller 的第一职责是**在任何响应前检查是否有适用的技能**：

```
  用户消息到达
       │
       ▼
  ┌─────────────────┐
  │ 可能有技能适用？  │  哪怕只有 1% 可能
  └──┬──────────┬───┘
     │          │
  是 │          │ 否（确定无疑）
     ▼          ▼
  调用 Skill    直接响应
  工具加载技能
     │
     ▼
  宣布："Using [skill] to [purpose]"
     │
     ▼
  技能有清单？── 是 → 创建 TodoWrite 任务列表
     │
    否
     │
     ▼
  严格遵循技能指令
```

### 2. 上下文构造

Controller 为子智能体**精确构造所需的上下文**，而不是让子智能体自己去读文件：

```
  Controller 的上下文管理
  ─────────────────────

  ┌──────────────────┐     ┌──────────────────┐
  │  Controller 的    │     │  子智能体获得的    │
  │  全局上下文       │ ──▶ │  精确上下文       │
  │                  │     │                  │
  │  - 完整会话历史   │     │  - 任务完整文本   │
  │  - 所有技能知识   │     │  - 架构上下文     │
  │  - 项目状态       │     │  - 依赖信息       │
  │  - 前序任务结果   │     │  - 工作目录       │
  │  - 用户偏好       │     │                  │
  └──────────────────┘     └──────────────────┘

  关键：子智能体绝不继承 Controller 的会话历史
  这防止了上下文污染——子智能体只看到它需要的
```

### 3. 子智能体编排

Controller 决定何时派遣哪种子智能体，以及按什么顺序：

```
  Controller 的编排决策
  ────────────────────

  ┌──────────────┐
  │ 当前任务需要？ │
  └──┬───────┬───┘
     │       │
  实现 │    审阅 │
     ▼       ▼
  派遣       派遣
  实现者     审阅者
  子智能体   子智能体

  模型选择：
  ┌────────────────────────┬──────────────┐
  │ 任务复杂度              │ 模型选择      │
  ├────────────────────────┼──────────────┤
  │ 机械（1-2文件，规范完整）│ 快速便宜模型  │
  │ 集成（多文件协调）      │ 标准模型      │
  │ 架构/设计/审阅          │ 最强模型      │
  └────────────────────────┴──────────────┘
```

### 4. 状态管理

Controller 使用 TodoWrite 工具追踪所有任务的进度：

```
  TodoWrite 任务追踪
  ─────────────────

  - [ ] Task 1: CSS 变量基础设施     ← pending
  - [x] Task 2: 主题切换组件         ← completed
  - [ ] Task 3: 用户偏好持久化       ← pending

  Controller 在每个子智能体完成后更新状态
  每次只标记一个任务为 in_progress
```

## 使用的技能

Controller 是**所有技能的使用者**，但不直接执行实现工作：

| 技能 | Controller 的使用方式 |
|------|---------------------|
| `using-superpowers` | 自身激活——被 SessionStart Hook 注入 |
| `brainstorming` | 引导用户完成设计流程，不做实现 |
| `writing-plans` | 指导计划生成，然后组织执行 |
| `subagent-driven-development` | 按此技能的流程派遣子智能体 |
| `executing-plans` | 按此技能的流程内联执行 |
| `dispatching-parallel-agents` | 对独立问题并行派遣 |
| `test-driven-development` | 确保子智能体遵循 TDD |
| `systematic-debugging` | 遇到 bug 时按此流程调查 |
| `verification-before-completion` | 在声称完成前运行验证 |
| `requesting-code-review` | 按此技能派遣代码审阅子智能体 |
| `receiving-code-review` | 收到审阅反馈后按此技能处理 |
| `finishing-a-development-branch` | 引导完成流程 |
| `using-git-worktrees` | 设置隔离工作区 |

## 与子智能体的交互模式

```
  ┌──────────────────────────────────────────────────┐
  │  Controller                                       │
  │                                                   │
  │  1. 读计划，提取所有任务文本和上下文                │
  │  2. 创建 TodoWrite 任务列表                        │
  │  3. 对每个任务：                                   │
  │     a. 构造子智能体 Prompt（从模板+任务文本+上下文）│
  │     b. 派遣实现者子智能体                           │
  │     c. 如果实现者提问 → 回答问题，重新派遣          │
  │     d. 实现者完成 → 派遣 spec-reviewer              │
  │     e. spec-reviewer 不通过 → 实现者修复 → 重审     │
  │     f. spec-reviewer 通过 → 派遣 code-reviewer     │
  │     g. code-reviewer 不通过 → 实现者修复 → 重审     │
  │     h. code-reviewer 通过 → 标记任务完成            │
  │  4. 所有任务完成 → 派遣最终审阅子智能体              │
  │  5. 调用 finishing-a-development-branch            │
  └──────────────────────────────────────────────────┘
```

## 关键原则

- **Controller 不做实现** — 它是编排者，不是工人
- **Controller 构造上下文** — 子智能体获得精确信息，不自行探索
- **Controller 管理状态** — 通过 TodoWrite 追踪，是唯一的全局状态持有者
- **Controller 处理升级** — 子智能体 BLOCKED/NEEDS_CONTEXT 时，Controller 决定如何响应
- **Controller 决定模型** — 根据任务复杂度选择最合适的模型
