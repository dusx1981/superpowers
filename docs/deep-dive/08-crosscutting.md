# 横切关注点

贯穿所有阶段的支撑系统。

## A. Git Worktree 隔离（using-git-worktrees）

所有实现工作都在**隔离的 worktree** 中进行，不污染主工作区。

### 生命周期

```
  创建（brainstorming/executing-plans 调用）
    ① 检查 .worktrees/ 目录是否存在
    ② 验证目录被 gitignore（防止意外提交）
    ③ git worktree add .worktrees/<branch> -b <branch>
    ④ 自动检测并安装依赖 (npm/cargo/pip/go)
    ⑤ 运行基线测试确认干净

  使用（subagent-driven-development 在其中工作）

  清理（finishing-a-development-branch）
    合并/丢弃后 git worktree remove <path>
```

### 目录选择优先级

| 优先级 | 位置 | 说明 |
|--------|------|------|
| 1 | `.worktrees/`（已存在） | 首选（隐藏） |
| 2 | `worktrees/`（已存在） | 备选 |
| 3 | CLAUDE.md 中的偏好 | 不问直接用 |
| 4 | 问用户 | `.worktrees/` 或 `~/.config/superpowers/worktrees/` |

### 安全验证

**项目本地目录**（.worktrees 或 worktrees）—— 必须验证被 gitignore：

```bash
git check-ignore -q .worktrees 2>/dev/null
```

**如果未被忽略**：立即添加到 .gitignore + 提交 + 然后继续。

**全局目录**（~/.config/superpowers/worktrees）：无需验证，在项目外。

### 基线验证

创建 worktree 后运行测试确保起点干净：

```bash
npm test    # 或 cargo test / pytest / go test ./...
```

**如果测试失败**：报告失败，问是否继续或调查。
**如果测试通过**：报告就绪。

## B. TDD 铁律（test-driven-development）

在每个任务的步骤级别强制执行 Red-Green-Refactor。

### 铁律

```
  NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

先写代码？删掉。从头来。不留作"参考"。不"适配"。

### Red-Green-Refactor 循环

```
  ┌────────┐     ┌────────┐     ┌──────────┐
  │  RED   │────▶│ GREEN  │────▶│ REFACTOR │──┐
  │写失败测试│     │最小实现  │     │ 清理     │  │
  └────────┘     └────────┘     └──────────┘  │
       ▲              │               │        │
       │              ▼               ▼        │
       │         验证通过          验证仍通过     │
       │              │               │        │
       │              └───────────────┘        │
       │                                       │
       └───────────────────────────────────────┘
                    下一个测试
```

**RED** — 写一个最小测试展示应该发生什么。必须验证失败。

**GREEN** — 写最简单的代码通过测试。不添加功能、不重构、不"改进"。

**REFACTOR** — 绿灯后才能重构：去重、改善命名、提取辅助。保持测试绿灯。

### 好测试 vs 坏测试

| 质量 | 好 | 坏 |
|------|---|---|
| 最小 | 一件事。名称有"and"？拆分 | `test('validates email and domain and whitespace')` |
| 清晰 | 名称描述行为 | `test('test1')` |
| 展示意 | 演示期望的 API | 模糊代码应该做什么 |
| 用真代码 | 测试真实行为 | 测试 mock 行为 |

### 例外（需征得用户同意）

- 一次性原型
- 生成代码
- 配置文件

### 常见合理化及反驳

| 借口 | 现实 |
|------|------|
| "太简单不需要测试" | 简单代码也会坏。测试花 30 秒 |
| "我之后再测" | 之后通过的测试什么也证明不了 |
| "之后测试达到同样目的" | 之后测试 = "这做了什么？" 之前测试 = "这应该做什么？" |
| "已经手动测了" | 临时测试 ≠ 系统测试。无记录，不能重跑 |
| "删 X 小时工作是浪费" | 沉没成本谬误。保留未验证代码是技术债 |

## C. 并行调度（dispatching-parallel-agents）

当多个独立问题需要同时调查时。

### 何时使用

```
  ┌──────────────┐
  │ 多个失败？    │
  └──┬───────┬───┘
     │       │
  是 │       │ 否 → 单一调查
     ▼       
  ┌──────────────┐
  │ 它们独立吗？  │
  └──┬───────┬───┘
     │       │
  是 │       │ 否（相关）→ 一起调查
     ▼
  ┌──────────────┐
  │ 可并行工作？  │
  └──┬───────┬───┘
     │       │
  是 │       │ 否（共享状态）→ 顺序
     ▼
  并行派遣
```

### 串行 vs 并行

```
  串行：                         并行：
  ┌──────────┐                  ┌──────────┐
  │ 修 问题 A │ 30min           │ 修 问题 A │ ┐
  └──────────┘                  └──────────┘ │ 同时
  ┌──────────┐                  ┌──────────┘ │ 30min
  │ 修 问题 B │ 30min           │ 修 问题 B │ ┘
  └──────────┘                  └──────────┘
  ┌──────────┐
  │ 修 问题 C │ 30min           总时间：30min vs 90min
  └──────────┘
```

### Agent Prompt 结构

好的并行 agent prompt：

1. **聚焦** — 一个清晰的问题域
2. **自包含** — 理解问题所需的全部上下文
3. **明确输出** — agent 应该返回什么

```
  ❌ 太宽泛："Fix all the tests" — agent 迷失
  ✓ 具体："Fix agent-tool-abort.test.ts" — 聚焦范围

  ❌ 无上下文："Fix the race condition" — agent 不知道在哪
  ✓ 有上下文：粘贴错误信息和测试名

  ❌ 无约束：Agent 可能重构一切
  ✓ 有约束："Do NOT change production code" 或 "Fix tests only"

  ❌ 模糊输出："Fix it" — 你不知道改了什么
  ✓ 具体输出："Return summary of root cause and changes"
```

### 验证

Agent 返回后：
1. **审阅每个摘要** — 理解改了什么
2. **检查冲突** — Agent 是否编辑了相同代码？
3. **运行完整测试套件** — 验证所有修复一起工作
4. **抽查** — Agent 可能犯系统性错误

### 何时不使用

- **相关失败**：修一个可能修另一个——先一起调查
- **需要完整上下文**：理解需要看到整个系统
- **探索性调试**：还不知道什么坏了
- **共享状态**：Agent 会互相干扰（编辑相同文件、使用相同资源）

## D. 技能调度系统（using-superpowers）

### 规则

**在做出任何响应或行动之前，调用相关的技能。** 哪怕只有 1% 的可能性技能适用，也应该调用。

### 红旗思维

| 想法 | 现实 |
|------|------|
| "这只是个简单问题" | 问题也是任务。检查技能 |
| "我需要先更多上下文" | 技能检查在澄清问题之前 |
| "让我先探索代码库" | 技能告诉你如何探索。先检查 |
| "我记得这个技能" | 技能会演进。读当前版本 |
| "这不需要正式技能" | 如果技能存在，就用它 |
| "我先做这一件事" | 在做任何事之前检查 |

### 技能优先级

当多个技能可能适用时：

1. **流程技能优先**（brainstorming、debugging）— 决定如何接近任务
2. **实现技能其次**（frontend-design、mcp-builder）— 指导执行

例如："让我们构建 X" → 先 brainstorming，然后实现技能。
"修复这个 bug" → 先 debugging，然后领域特定技能。
