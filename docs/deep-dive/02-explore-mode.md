# 阶段 ②：探索模式（/opsx-explore）

这是一个**思考姿态**，不是流程步骤。可以在任何时候切入。

## 定位

```
  ┌────────────────────────────────────────┐
  │       这不是工作流，而是一种姿态         │
  │                                        │
  │  ✓ 好奇而非处方                         │
  │  ✓ 开放线索而非审讯                     │
  │  ✓ 可视化（大量 ASCII 图）              │
  │  ✓ 自适应（跟着有趣线索走）             │
  │  ✓ 耐心（让问题形状自然浮现）           │
  │  ✓ 扎根代码库（不光理论推演）           │
  │                                        │
  │  ✗ 不写代码                             │
  │  ✗ 不自动捕获（只提议保存）             │
  │  ✗ 不强制结构                           │
  └────────────────────────────────────────┘
```

## 可以做什么

- **探索问题空间** — 澄清问题、挑战假设、重新框架、寻找类比
- **调查代码库** — 映射架构、找集成点、识别模式、发现隐藏复杂性
- **比较选项** — 头脑风暴多种方法、构建比较表、勾勒权衡
- **可视化** — 系统图、状态机、数据流、架构草图、依赖图
- **发现风险** — 识别可能出错的地方、理解差距、建议调查

## OpenSpec 集成

### 启动时检查上下文

```bash
openspec list --json
```

获取当前活跃的变更、名称、模式和状态。

### 无变更时

自由思考。当洞察结晶时，可以提议：

- "这足够成熟了，要不要创建一个 change proposal？"
- 或者继续探索——没有压力

### 有变更时

读取现有工件作为上下文：

| 工件 | 路径 |
|------|------|
| proposal | `openspec/changes/<name>/proposal.md` |
| design | `openspec/changes/<name>/design.md` |
| tasks | `openspec/changes/<name>/tasks.md` |

在对话中自然引用，当决策做出时提议捕获：

| 洞察类型 | 捕获位置 |
|---------|---------|
| 新需求发现 | `specs/<capability>/spec.md` |
| 需求变更 | `specs/<capability>/spec.md` |
| 设计决策 | `design.md` |
| 范围变更 | `proposal.md` |
| 新工作识别 | `tasks.md` |
| 假设失效 | 相关工件 |

**用户决定** — 提议后继续，不施压，不自动捕获。

## 守卫

- **不实现** — 绝不写代码或实现功能。创建 OpenSpec 工件可以（那是捕获思考），写应用代码不行
- **不假装理解** — 不清楚就深挖
- **不急** — 探索是思考时间，不是任务时间
- **不强制结构** — 让模式自然浮现
- **要可视化** — 好的图表值很多段落
- **要探索代码库** — 把讨论扎根于现实
