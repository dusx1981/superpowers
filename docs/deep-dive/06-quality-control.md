# 阶段 ⑥：质量控制

这是 superpowers 最精心设计的部分——保证**不做错的事**和**把事做对**。

## 三层审阅体系

```
  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  Layer 1: 规范合规审阅 (spec-reviewer)                   │
  │  ┌───────────────────────────────────────────────────┐  │
  │  │ "不要信任实现者的报告！"                             │  │
  │  │                                                    │  │
  │  │  读实际代码，逐行比对规范：                          │  │
  │  │  - 遗漏的需求（他们说实现了但实际没做？）            │  │
  │  │  - 额外的东西（没要求但多做了？）                    │  │
  │  │  - 误解需求（解决的是错误的问题？）                  │  │
  │  └───────────────────────────────────────────────────┘  │
  │                         ▼                               │
  │  Layer 2: 代码质量审阅 (code-quality-reviewer)           │
  │  ┌───────────────────────────────────────────────────┐  │
  │  │  基于 git diff (base_sha..head_sha) 审阅：          │  │
  │  │  - 每个文件是否有单一清晰职责？                     │  │
  │  │  - 单元是否可独立理解和测试？                       │  │
  │  │  - 是否遵循计划的文件结构？                         │  │
  │  │  - 是否创造了过大的新文件/显著膨胀了现有文件？       │  │
  │  │                                                    │  │
  │  │  返回：Strengths / Issues (Critical/Important/Minor) │  │
  │  │        / Assessment                                 │  │
  │  └───────────────────────────────────────────────────┘  │
  │                         ▼                               │
  │  Layer 3: 最终整体审阅 (全部任务完成后)                    │
  │  ┌───────────────────────────────────────────────────┐  │
  │  │  派遣一个代码审阅子智能体审阅整个实现                  │  │
  │  │  跨任务一致性、集成问题、全局视角                      │  │
  │  └───────────────────────────────────────────────────┘  │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

## 关键约束：顺序不可颠倒

```
  规范审阅必须先于代码质量审阅

  ✗ 代码质量审阅 → 规范审阅    （代码写得再漂亮，做错了事也没用）
  ✓ 规范审阅    → 代码质量审阅  （先确认做对了事，再确认把事做对）
```

## 审阅循环机制

审阅发现问题 → 实现者修复 → **必须重新审阅**，不能跳过：

```
  审阅发现问题
       │
       ▼
  实现者修复（同一个子智能体）
       │
       ▼
  审阅者重新审阅  ←── 必须重新审阅！不能"看起来修了就过"
       │
    通过？── 否 → 回到修复
       │
      是
       ▼
    下一步
```

## 规范合规审阅 Prompt

来源：`skills/subagent-driven-development/spec-reviewer-prompt.md`

```
  你正在审阅实现是否匹配规范。

  ## What Was Requested
  [完整任务需求文本]

  ## What Implementer Claims They Built
  [来自实现者报告]

  ## CRITICAL: Do Not Trust the Report
  实现者完成得可疑地快。他们的报告可能不完整、不准确或乐观。
  你必须独立验证一切。

  DO NOT:
  - 采信他们对实现的描述
  - 信任他们关于完整性的声称
  - 接受他们对需求的解读

  DO:
  - 读他们写的实际代码
  - 逐行比对实际实现与需求
  - 检查他们声称实现但实际缺失的部分
  - 寻找他们没提到的额外功能

  验证维度：
  - 遗漏的需求：跳过或遗漏了什么？
  - 额外/不需要的工作：构建了未要求的东西？过度工程？
  - 误解：解读需求的方式不同于意图？解决了错误的问题？

  报告：
  - ✅ Spec compliant（一切匹配）
  - ❌ Issues found: [具体列出缺失或额外项，附 file:line 引用]
```

## 代码质量审阅 Prompt

来源：`skills/subagent-driven-development/code-quality-reviewer-prompt.md`

只在规范合规审阅通过后派遣。

```
  Task tool (superpowers:code-reviewer):
    WHAT_WAS_IMPLEMENTED: [来自实现者报告]
    PLAN_OR_REQUIREMENTS: Task N from [plan-file]
    BASE_SHA: [任务前提交]
    HEAD_SHA: [当前提交]
    DESCRIPTION: [任务摘要]

  额外检查项（在标准代码质量关注之外）：
  - 每个文件是否有单一清晰职责和定义良好的接口？
  - 单元是否分解为可独立理解和测试的？
  - 是否遵循计划中的文件结构？
  - 是否创造了已经很大的新文件，或显著膨胀了现有文件？
    （不标记已有文件大小——关注此变更贡献了什么）
```

返回：**Strengths** / **Issues (Critical/Important/Minor)** / **Assessment**

## 计划文档审阅

来源：`skills/writing-plans/plan-document-reviewer-prompt.md`

在计划写完后派遣，验证计划本身的质量：

| 检查类别 | 查找内容 |
|---------|---------|
| 完整性 | TODOs、占位符、不完整任务、缺失步骤 |
| 规范对齐 | 计划覆盖规范需求、无重大范围蔓延 |
| 任务分解 | 任务边界清晰、步骤可操作 |
| 可构建性 | 工程师能按此计划执行而不卡住吗？ |

**校准**：只标记会在实施中造成真正问题的东西。措辞/风格偏好和"锦上添花"建议不算。

## 规范文档审阅

来源：`skills/brainstorming/spec-document-reviewer-prompt.md`

在设计文档写完后派遣，验证规范本身的质量：

| 检查类别 | 查找内容 |
|---------|---------|
| 完整性 | TODOs、占位符、"TBD"、不完整章节 |
| 一致性 | 内部矛盾、冲突需求 |
| 清晰性 | 需求模糊到可能导致构建错误的东西 |
| 范围 | 是否聚焦于单个计划——不覆盖多个独立子系统 |
| YAGNI | 未请求的功能、过度工程 |

**校准**：只标记会在计划编写中造成真正问题的东西。
