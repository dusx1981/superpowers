## 5A：子智能体驱动开发（subagent-driven-development）— 推荐

核心思想：**每个任务派一个全新子智能体 + 两阶段审阅**。

### 为什么用子智能体

- 委派任务给有隔离上下文的专门智能体
- 精确构造它们的指令和上下文，确保专注和成功
- 它们绝不继承会话的上下文或历史——你构造它们需要的全部
- 这也保留了你自己上下文用于协调工作

### 每个任务的生命周期

```
  ┌─────────────────────────────────────────────────┐
  │              Task N 的处理流程                     │
  │                                                  │
  │  ┌─────────────────┐                             │
  │  │ 1. 派遣实现者     │  全新子智能体                │
  │  │    (implementer) │  不继承会话上下文             │
  │  └────────┬────────┘  控制器精确构造其所需信息      │
  │           │                                     │
  │      有问题？── 是 → 回答问题，重新派遣             │
  │           │                                     │
  │          否                                      │
  │           ▼                                     │
  │  ┌─────────────────┐                             │
  │  │ 2. 实现者工作     │  实现 → 测试 → 提交 → 自审  │
  │  └────────┬────────┘                             │
  │           │                                     │
  │           ▼                                     │
  │  ┌─────────────────┐                             │
  │  │ 3. 规范合规审阅   │  另一个全新子智能体           │
  │  │  (spec-reviewer) │  "不要信任实现者的报告！"      │
  │  └────────┬────────┘  读实际代码，逐行比对规范      │
  │           │                                     │
  │      不通过？── 是 → 实现者修复 → 重新规范审阅      │
  │           │                                     │
  │          通过                                    │
  │           ▼                                     │
  │  ┌─────────────────┐                             │
  │  │ 4. 代码质量审阅   │  又一个全新子智能体           │
  │  │  (code-quality)  │  基于 git diff (base..head) │
  │  └────────┬────────┘                             │
  │           │                                     │
  │      不通过？── 是 → 实现者修复 → 重新质量审阅      │
  │           │                                     │
  │          通过                                    │
  │           ▼                                     │
  │  ┌─────────────────┐                             │

  └─────────────────────────────────────────────────┘

  所有任务完成后：
  ┌─────────────────────────────────┐
  │ 6. 派遣最终代码审阅子智能体       │
```

### 模型选择策略

用最便宜够用的模型来节约成本和提高速度：

| 任务类型 | 模型选择 | 典型场景 |
|---------|---------|---------|
| 机械实现 | 快速便宜模型 | 孤立函数、清晰规范、1-2 文件 |
| 集成判断 | 标准模型 | 多文件协调、模式匹配、调试 |
| 架构设计/审阅 | 最强模型 | 需要设计判断或广泛代码库理解 |

### 实现者的四种状态

| 状态 | 含义 | 处理方式 |
|------|------|---------|
| `DONE` | 完成了 | 进入规范审阅 |
| `DONE_WITH_CONCERNS` | 完成但有疑虑 | 先读疑虑再决定——如果关于正确性/范围，先解决；如果只是观察，记录并继续 |
| `NEEDS_CONTEXT` | 缺信息 | 提供缺的上下文，重新派遣 |
| `BLOCKED` | 卡住了 | 1. 上下文问题→提供更多上下文 2. 需要更强推理→换更强模型 3. 任务太大→拆更小 4. 计划本身有问题→升级给人 |

**绝不**忽略升级请求或强迫同一模型无条件重试。实现者说卡住了，必须改变什么。

### 实现者 Prompt 模板

来源：`skills/subagent-driven-development/implementer-prompt.md`

核心结构：

```
  你正在实现 Task N: [task name]

  ## Task Description
  [计划中的完整任务文本 — 粘贴，不让子智能体读文件]

  ## Context
  [场景设定：这在哪里、依赖、架构上下文]

  ## Before You Begin
  如果有关于需求/方法/依赖/假设的问题 → 现在就问

  ## Your Job
  1. 实现任务指定的内容
  2. 写测试（遵循 TDD）
  3. 验证实现有效
  4. 提交
  5. 自审
  6. 报告

  ## Code Organization
  - 遵循计划中的文件结构
  - 每个文件一个清晰职责
  - 文件超出计划意图 → 报告 DONE_WITH_CONCERNS

  ## When You're in Over Your Head
  停下来升级是可以的。坏工作比没工作更糟。
  报告 BLOCKED 或 NEEDS_CONTEXT + 具体卡在哪 + 试过什么 + 需要什么帮助

  ## Self-Review
  完整性 / 质量 / 纪律（YAGNI）/ 测试

  ## Report Format
  Status: DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
  + 实现了什么 / 测试结果 / 变更文件 / 自审发现 / 问题
```

### 红旗

**绝不**：
- 在 main/master 分支上开始实现（未经明确同意）
- 跳过审阅（规范合规或代码质量）
- 有未修复问题就继续
- 并行派遣多个实现子智能体（会冲突）
- 让子智能体读计划文件（提供完整文本）
- 跳过场景设定上下文
- 忽略子智能体问题
- 接受规范审阅"差不多就行"
- 在规范合规通过前开始代码质量审阅（顺序不可颠倒）
- 在任一审阅有开放问题时移到下一个任务

## 5B：计划执行（executing-plans）— 不推荐

如果平台支持子智能体，优先用 `subagent-driven-development`。

```
  ① 加载计划 → 批判性审阅（有疑虑先问人）
         ▼
  ② 逐任务执行（严格按计划步骤）
     - 每步标记 in_progress
     - 按步骤走（计划已经原子化了）
     - 运行验证
     - 标记 completed
         ▼
  ③ 遇到阻塞立即停下问人
         ▼
  ④ 全部完成 → finishing-a-development-branch
```

### 何时停下问人

- 碰到阻塞器（缺依赖、测试失败、指令不清楚）
- 计划有严重缺口导致无法开始
- 不理解某个指令
- 验证反复失败

### 何时回到审阅

- 伙伴基于反馈更新了计划
- 基本方法需要重新思考

**不要强行冲过阻塞**——停下来问。
