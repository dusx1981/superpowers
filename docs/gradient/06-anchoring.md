# 06 上下文锚定

上下文锚定是梯度的第六维——同样的指令，出现在上下文的不同位置，AI 的服从概率不同。先发制人比后发制人强得多。

## 6.1 注入时机：先发制人

### 两种注入方式

```
  ┌──────────────────────────────────────────────────────────┐
  │  Claude Code (SessionStart Hook)：                       │
  │  ────────────────────────────────                        │
  │  触发：会话启动事件                                      │
  │  位置：第一条系统附加上下文                              │
  │  效果：AI 看到的第一个内容                               │
  │                                                          │
  │  OpenCode (messages.transform)：                         │
  │  ─────────────────────────────────                       │
  │  触发：消息转换事件                                      │
  │  位置：第一条用户消息的最前面                            │
  │  效果：AI 在处理用户输入前先处理 bootstrap              │
  │                                                          │
  │  共同特征：                                              │
  │  - 都在 AI 处理用户输入之前完成注入                     │
  │  - 都使用 <EXTREMELY_IMPORTANT> 包裹                    │
  │  - 都只注入一次（幂等性）                               │
  └──────────────────────────────────────────────────────────┘
```

### 为什么注入位置是权重？

```
  ┌──────────────────────────────────────────────────────────┐
  │  先发制人的三个权重来源：                                │
  │                                                          │
  │  1. 注意力峰值                                           │
  │     ────────────                                         │
  │     AI 处理长文本时，开头内容的注意力权重最高            │
  │     → "首因效应"（Primacy Effect）                      │
  │     → AI 最早看到的内容塑造了它的"初始状态"            │
  │     → 初始状态 = "我有超能力" → 后续行为以此为锚       │
  │                                                          │
  │  2. 锚定效应                                             │
  │     ────────                                             │
  │     AI 对后续内容的理解受开头内容影响                    │
  │     先看到 "You have superpowers" + 铁律                 │
  │     → 后续交互中 AI 会以"我有超能力"为锚点              │
  │     → 不太可能忽略技能体系                               │
  │     → 锚 = 参照系，所有后续判断都相对于锚               │
  │                                                          │
  │  3. 不可跳过                                             │
  │     ──────────                                           │
  │     用户不能跳过 SessionStart 注入                       │
  │     它在用户输入之前就已经存在                           │
  │     → 用户的第一条消息前面已经有了 bootstrap             │
  │     → AI 处理用户消息时必然先处理 bootstrap              │
  │     → 没有任何"跳过"的 UI 或交互方式                   │
  └──────────────────────────────────────────────────────────┘
```

## 6.2 位置权重：消息序列中的注意力分布

### 注意力的 U 型分布

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 对消息序列的注意力分布（示意）：                     │
  │                                                          │
  │  注意力                                                  │
  │  ──────                                                  │
  │  高 ┤  ┌───┐                                     ┌───┐  │
  │     │  │ B │                                     │ U │  │
  │     │  │ o │                                     │ s │  │
  │     │  │ o │                                     │ e │  │
  │     │  │ t │              ┌───┐                  │ r │  │
  │     │  │ s │              │ M │                  │   │  │
  │     │  │ t │              │ i │                  │ M │  │
  │     │  │ r │              │ d │                  │ s │  │
  │     │  │ a │              │ d │                  │ g │  │
  │  低 ┤  │ p │──────────────│ l │──────────────────│   │  │
  │     └──┴───┴──────────────┴───┴──────────────────┴───┘──┘
  │        开头              中间                  末尾       │
  │                                                          │
  │  Bootstrap 注入位置 = 开头 = 注意力峰值                 │
  │  → 铁律和技能体系获得了最高注意力权重                   │
  │                                                          │
  │  但注意：近因效应（末尾）也高                           │
  │  → 如果用户最后一条消息说"跳过测试"                    │
  │  → 近因效应可能压过首因效应                             │
  │  → 这是梯度的脆弱点之一                                 │
  └──────────────────────────────────────────────────────────┘
```

### 用户消息 vs 系统消息的位置权重

```
  ┌──────────────────────────────────────────────────────────┐
  │  OpenCode 选择用户消息而非系统消息的原因：               │
  │                                                          │
  │  系统消息：                                              │
  │  - 每轮重复发送 → token 膨胀（#750）                    │
  │  - 多条系统消息可能破坏某些模型（#894）                  │
  │  - 但位置权重更高（系统消息 > 用户消息）                │
  │                                                          │
  │  用户消息（unshift 到最前面）：                          │
  │  - 只注入一次 → 不重复                                   │
  │  - 单条消息包含所有 bootstrap → 不破坏格式               │
  │  - 所有模型都支持 → 兼容性好                            │
  │  - 但位置权重略低于系统消息                              │
  │                                                          │
  │  Superpowers 的补偿：                                    │
  │  - 用 <EXTREMELY_IMPORTANT> 标签补偿位置权重的损失      │
  │  - 标签权重 > 消息类型差异                              │
  │  - 网络效应：位置权重 × 标签权重 × 语用权重             │
  │    = 仍然极高                                            │
  └──────────────────────────────────────────────────────────┘
```

## 6.3 近因效应：梯度的脆弱点

### 近因效应 vs 首因效应

```
  ┌──────────────────────────────────────────────────────────┐
  │  当对话变长时，近因效应开始压过首因效应：               │
  │                                                          │
  │  短对话（< 10 轮）：                                    │
  │  Bootstrap (首因) ████████████  → 主导                  │
  │  用户最新消息 (近因) ████████                            │
  │  → 铁律仍然有效                                         │
  │                                                          │
  │  长对话（> 50 轮）：                                    │
  │  Bootstrap (首因) ████████                              │
  │  用户最新消息 (近因) ████████████  → 主导               │
  │  → 铁律可能被最新的用户消息覆盖                         │
  │                                                          │
  │  超长对话（> 100 轮）：                                 │
  │  Bootstrap (首因) ████                                  │
  │  中间对话 (稀释) ██                                     │
  │  用户最新消息 (近因) ████████████  → 强主导              │
  │  → Bootstrap 几乎被遗忘                                 │
  │  → 铁律可能失效                                         │
  │                                                          │
  │  这是一个根本性的限制：                                  │
  │  - LLM 的上下文窗口有限                                  │
  │  - 注意力在长序列中会衰减                               │
  │  - 没有完美的解决方案                                    │
  └──────────────────────────────────────────────────────────┘
```

### Superpowers 的部分缓解

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 没有完全解决近因效应，                     │
  │  但有几个缓解措施：                                      │
  │                                                          │
  │  1. 技能重新加载                                         │
  │     每次调用 Skill 工具时，技能内容被重新注入            │
  │     → 不只是 Bootstrap 在开头，技能在每次使用时         │
  │       都被"刷新"到近处                                  │
  │     → 技能的铁律和红旗在调用时就在附近                  │
  │                                                          │
  │  2. 流程图的强制                                         │
  │     流程图定义了步骤和分支                               │
  │     → AI 不需要"记住"开头的规定                        │
  │     → 它在当前步骤就知道下一步该做什么                  │
  │     → 流程图 = 分布式锚定（不只在开头）                │
  │                                                          │
  │  3. 清单的追踪                                           │
  │     TodoWrite 中的 pending/in_progress/completed          │
  │     → 状态在每轮对话中都可见                            │
  │     → AI 不需要"记住"流程，流程在状态中                │
  │     → 清单 = 持续锚定（不在开头，在每一步）            │
  │                                                          │
  │  4. 审阅循环的强制                                       │
  │     审阅者子智能体是全新实例                             │
  │     → 它不受"长对话"的影响——它只有当前任务的上下文     │
  │     → 子智能体 = 隔离锚定（不受近因效应影响）          │
  └──────────────────────────────────────────────────────────┘
```

## 6.4 锚定的层级

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 实际上在多个层级进行锚定：                  │
  │                                                          │
  │  L0: SessionStart 注入                                   │
  │      → 一次性的，会话最开头                              │
  │      → 覆盖：整个技能体系                                │
  │      → 风险：长对话后衰减                                │
  │                                                          │
  │  L1: 技能调用                                            │
  │      → 每次使用技能时重新注入                            │
  │      → 覆盖：当前技能的完整内容                          │
  │      → 风险：不调用技能 = 不注入                        │
  │                                                          │
  │  L2: 清单/状态追踪                                       │
  │      → TodoWrite 中的状态始终可见                        │
  │      → 覆盖：当前流程的步骤                              │
  │      → 风险：AI 可能忽略 pending 项                     │
  │                                                          │
  │  L3: 审阅/子智能体                                       │
  │      → 全新实例，不受长对话影响                          │
  │      → 覆盖：当前任务的执行和审阅                        │
  │      → 风险：子智能体之间的协调                          │
  │                                                          │
  │  四层锚定的互补：                                        │
  │  - L0 失效（长对话）→ L1 接管（重新调用技能）           │
  │  - L1 不触发（没调用技能）→ L2 接管（清单提醒）         │
  │  - L2 被忽略 → L3 接管（子智能体强制审阅）              │
  │  - 四层锚定 = 四次机会 = 不容易全部失效                 │
  └──────────────────────────────────────────────────────────┘
```
