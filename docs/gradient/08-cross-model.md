# 08 跨模型/跨语言

梯度的效果不是普适的——不同模型尺寸、不同语言、不同 tokenizer 实现下，梯度的三个层（语用/格式/训练）的相对效力不同。

## 8.1 模型尺寸的敏感度差异

```
  ┌──────────────────────────────────────────────────────────┐
  │  不同模型尺寸对三层梯度的敏感度：                        │
  │                                                          │
  │  小模型 (1-7B)：                                         │
  │  ─────────────────                                       │
  │  格式层：最强效果——小模型更依赖格式对比度获得注意力     │
  │  训练层：最弱效果——训练语料少，统计关联弱               │
  │  语用层：中等效果——能理解 must vs should，               │
  │          但对 "NO X WITHOUT Y" 的法律关联理解不足        │
  │                                                          │
  │  中模型 (7-70B)：                                        │
  │  ──────────────────                                      │
  │  格式层：较强效果——仍依赖格式，但纯文本理解力提升       │
  │  训练层：中等效果——开始理解训练数据关联                  │
  │  语用层：较强效果——精确理解情态动词和否定式             │
  │                                                          │
  │  大模型 (70B+)：                                         │
  │  ────────────────                                        │
  │  格式层：减弱效果——大模型对小写文本的理解力已很强      │
  │  训练层：最强效果——法律/合同语料理解深入                │
  │  语用层：最强效果——精确理解情态动词和否定式的细微差异   │
  │                                                          │
  │  ┌──────────────────────────────────────────────┐       │
  │  │           小模型    中模型    大模型           │       │
  │  │  格式层   ████████  ██████    ████            │       │
  │  │  训练层   ██        ████      ████████        │       │
  │  │  语用层   ████      ██████    ████████        │       │
  │  └──────────────────────────────────────────────┘       │
  │                                                          │
  │  推论：                                                  │
  │  - 对小模型：格式层最重要（XML标签 + ALL CAPS）          │
  │  - 对大模型：语用层 + 训练层最重要，格式层是加分项      │
  │  - Superpowers 三层全上 → 对所有模型尺寸都有效          │
  └──────────────────────────────────────────────────────────┘
```

### 小模型为什么更依赖格式层？

```
  ┌──────────────────────────────────────────────────────────┐
  │  小模型的特征：                                          │
  │                                                          │
  │  1. 语义理解能力弱                                       │
  │     "You MUST write tests" → 小模型可能不理解 MUST       │
  │     的法律关联，只是模糊地知道"这很重要"               │
  │     → 语用层的精细差别对小模型效果有限                  │
  │                                                          │
  │  2. 注意力容量小                                         │
  │     小模型的注意力头少、维度低                           │
  │     → 它更依赖"格式异常"来分配注意力                   │
  │     → ALL CAPS 在小模型中的注意力提升比例更高           │
  │                                                          │
  │  3. 训练数据少                                           │
  │     小模型见过的法律/合同语料更少                        │
  │     → "NO X WITHOUT Y" 与法律关联的统计信号弱           │
  │     → 训练层效果有限                                    │
  │                                                          │
  │  4. 格式是最可靠的信号                                   │
  │     即使不理解语义，小模型也能感知格式差异               │
  │     → 格式是"不依赖语义"的注意力加压方式               │
  │     → 对小模型最可靠                                    │
  └──────────────────────────────────────────────────────────┘
```

### 大模型为什么格式层效果减弱？

```
  ┌──────────────────────────────────────────────────────────┐
  │  大模型的特征：                                          │
  │                                                          │
  │  1. 语义理解能力强                                       │
  │     大模型能理解 "MUST" 的法律含义                       │
  │     → 不需要格式来"提醒"它注意                          │
  │     → 即使是小写文本，它也能正确理解力度                │
  │                                                          │
  │  2. 注意力容量大                                         │
  │     大模型有更多的注意力头和更高的维度                   │
  │     → 它能同时关注多个部分，不只依赖格式异常            │
  │     → 格式的"对比度效应"在容量大的模型中占比更小       │
  │                                                          │
  │  3. 训练数据丰富                                         │
  │     大模型见过大量法律/合同/安全语料                     │
  │     → "NO X WITHOUT Y" 直接触发法律级服从               │
  │     → 不需要 ALL CAPS 来额外激活                        │
  │                                                          │
  │  但注意：                                                │
  │  格式层效果减弱 ≠ 格式层无效                             │
  │  格式层仍是大模型的加分项，只是不是必要条件              │
  │  → 对大模型，三层全上的净效果仍 > 只用两层              │
  └──────────────────────────────────────────────────────────┘
```

## 8.2 Superpowers 的模型选择与梯度适配

```
  ┌──────────────────────────────────────────────────────────┐
  │  subagent-driven-development 的模型选择策略：            │
  │                                                          │
  │  机械实现任务 → 快速/廉价模型（通常较小）               │
  │  集成判断任务 → 标准模型（中等）                        │
  │  架构/审阅任务 → 最强模型（最大）                       │
  │                                                          │
  │  梯度适配的隐含逻辑：                                    │
  │                                                          │
  │  小模型做机械任务：                                      │
  │  → 任务规范完整（writing-plans 保证了这一点）           │
  │  → 不需要理解铁律的深层含义                             │
  │  → 只需要知道"必须这样做"                              │
  │  → Prompt 模板的格式层（XML、ALL CAPS）对小模型足够     │
  │                                                          │
  │  大模型做审阅任务：                                      │
  │  → 需要理解"为什么不能信任实现者报告"                  │
  │  → 需要独立判断代码质量                                 │
  │  → 语用层和训练层的深度理解更重要                       │
  │  → 大模型能充分利用 "DO NOT TRUST" 的语用力度           │
  │                                                          │
  │  匹配原则：                                              │
  │  模型尺寸 × 梯度层级 = 最优配置                         │
  │  小模型 + 格式层权重高 = 机械任务                       │
  │  大模型 + 语用/训练层权重高 = 审阅任务                   │
  └──────────────────────────────────────────────────────────┘
```

## 8.3 中文梯度的特殊性

### 情态动词梯度的跨语言差异

```
  ┌──────────────────────────────────────────────────────────┐
  │  英语情态动词：                                          │
  │  might → could → should → must → shall                   │
  │  → 训练数据中 MUST 与法律/义务高度关联                   │
  │  → 梯度清晰，效果强                                     │
  │                                                          │
  │  中文情态表达：                                          │
  │  可以 → 应该 → 必须 → 不得 → 绝不                       │
  │  → "必须" 与制度/规定有关联，但弱于 MUST                │
  │  → 因为中文训练数据中法律/合同语料比例低于英语          │
  │  → "不得" 更强（公文/法律用语），但日常使用少           │
  │  → "绝不" 最强，但缺少条件禁止的句式                    │
  └──────────────────────────────────────────────────────────┘
```

### 否定式的跨语言差异

```
  ┌──────────────────────────────────────────────────────────┐
  │  中文否定式的强度梯度：                                  │
  │                                                          │
  │  "别跳过测试"        ──→  口语化否定（弱）              │
  │  "不要跳过测试"      ──→  指示性否定（中弱）            │
  │  "不可跳过测试"      ──→  正式否定（中）                │
  │  "不得跳过测试"      ──→  制度性否定（强）              │
  │  "绝不跳过测试"      ──→  绝对性否定（极强）            │
  │                                                          │
  │  中文缺少 "NO X WITHOUT Y" 的精确等价：                 │
  │  - "没有测试就不能写代码" → 接近，但不够正式            │
  │  - "未经测试不得编写代码" → 更正式，但不够有力          │
  │  - "无测试，不代码" → 简洁有力，但不是标准中文          │
  │                                                          │
  │  推论：                                                  │
  │  英语的梯度体系更成熟，因为有更丰富的法律/合同           │
  │  训练语料支撑 "NO X WITHOUT Y" 句式。                   │
  │  中文 Prompt 设计需要额外依赖格式层（XML 标签仍可用，   │
  │  但全大写在中文中不可用）来补偿语用层的差距。           │
  └──────────────────────────────────────────────────────────┘
```

### 格式层的跨语言差异

```
  ┌──────────────────────────────────────────────────────────┐
  │  ALL CAPS 在中文中不可用：                               │
  │  - 中文没有大小写区分                                    │
  │  - "绝不" 无法用全大写表示                              │
  │  - 格式层丢失了一个重要工具                             │
  │                                                          │
  │  中文可用的格式替代：                                    │
  │  1. XML 标签（跨语言通用）                              │
  │     <铁律> 未经测试不得编写代码 </铁律>                  │
  │     → 仍有效，但标签名用中文可能不如英文直观            │
  │                                                          │
  │  2. 代码块（跨语言通用）                                │
  │     ```                                                  │
  │     未经测试不得编写代码                                 │
  │     ```                                                  │
  │     → 仍有效，代码块的"引用"效应跨语言通用              │
  │                                                          │
  │  3. 表格（跨语言通用）                                  │
  │     | 借口 | 现实 |                                      │
  │     → 仍有效，表格的左右对比效应跨语言通用              │
  │                                                          │
  │  4. 符号强化（中文特有）                                │
  │     【绝不】 / ⛔禁止 / ❌不允许                         │
  │     → 中文可以用符号来补偿 ALL CAPS 的缺失              │
  │     → 但训练数据中符号与强约束的关联可能弱于 ALL CAPS   │
  │                                                          │
  │  整体评估：                                              │
  │  中文梯度 = 语用层略弱 + 格式层缺ALL CAPS + 训练层略弱  │
  │  → 中文 Prompt 需要更依赖 XML 标签和代码块格式          │
  │  → 或混合使用英文铁律句式 + 中文解释                    │
  └──────────────────────────────────────────────────────────┘
```

## 8.4 Tokenizer 的影响

```
  ┌──────────────────────────────────────────────────────────┐
  │  Tokenizer 决定了文本如何被切分为 token。               │
  │  不同的 tokenizer 对梯度的格式层有直接影响。            │
  │                                                          │
  │  当前主流 tokenizer（GPT-4, Claude, Gemini）：           │
  │  - 区分大小写："MUST" 和 "must" 产生不同的 token        │
  │  - ALL CAPS 文本的 token 序列与小写不同                 │
  │  → 格式层有效                                            │
  │                                                          │
  │  假设的未来 tokenizer：                                  │
  │  - 不区分大小写：所有文本转为小写后 tokenize             │
  │  - "MUST" 和 "must" 产生相同的 token                    │
  │  → ALL CAPS 的格式层效果消失                            │
  │  → 但训练层仍有效（因为预训练时仍区分大小写）           │
  │                                                          │
  │  BPE vs SentencePiece vs WordPiece：                     │
  │  - 不同 tokenizer 对同一文本的切分方式不同               │
  │  - <EXTREMELY_IMPORTANT> 在不同 tokenizer 中             │
  │    可能被切分为不同数量的 token                          │
  │  - 更多 token = 更多注意力分配 = 更强格式效果           │
  │  - 但 token 数量差异通常不大，不会根本改变效果          │
  │                                                          │
  │  推论：                                                  │
  │  格式层的效果依赖 tokenizer 实现。                      │
  │  这是格式层"脆弱"的具体表现。                           │
  │  语用层不受 tokenizer 影响（语义不变）。                │
  │  训练层间接受影响（通过 tokenizer 改变训练数据的表示）。 │
  └──────────────────────────────────────────────────────────┘
```

## 8.5 跨模型/跨语言的设计原则

```
  ┌──────────────────────────────────────────────────────────┐
  │  设计跨模型/跨语言的梯度体系时：                         │
  │                                                          │
  │  1. 不假设特定模型                                       │
  │     → 三层全上，不依赖单一机制                           │
  │     → 任一层失效，其他层仍可兜底                         │
  │                                                          │
  │  2. 不假设特定语言                                       │
  │     → 核心铁律用英语（训练关联最强）                     │
  │     → 辅助解释用母语（理解力最强）                       │
  │     → 格式用跨语言通用的（XML标签、代码块、表格）       │
  │                                                          │
  │  3. 不假设特定 tokenizer                                 │
  │     → 格式层用多种方式叠加（ALL CAPS + XML + 代码块）    │
  │     → 即使一种格式失效，其他格式仍有效                   │
  │                                                          │
  │  4. 适配模型尺寸                                         │
  │     → 对小模型：增加格式层权重                           │
  │     → 对大模型：增加语用层和训练层权重                   │
  │     → Superpowers 的三层全上 = 免适配                    │
  │                                                          │
  │  5. 准备演化                                             │
  │     → 如果未来 ALL CAPS 效果减弱，需要新的格式替代       │
  │     → 如果未来 RLHF 改变了服从倾向，需要调整语用策略     │
  │     → 梯度体系是活的，不是死的                           │
  └──────────────────────────────────────────────────────────┘
```
