# 01 五步序列

五步序列不是随意的流程，而是一个完整的因果证明链。每一步证明一件事，缺少任何一步证明就不完整。

## 完整序列

```
  Step 1: Write          → 写回归测试
  Step 2: Run (pass)     → 运行测试，通过
  Step 3: Revert fix     → 撤回修复（放回 bug）
  Step 4: Run (MUST FAIL)→ 运行测试，必须失败
  Step 5: Restore → Run (pass) → 恢复修复，运行测试，通过
```

## 每一步证明了什么

### Step 1: Write（写回归测试）

```
  ┌──────────────────────────────────────────────────────────┐
  │  动作：写一个测试，描述 bug 修复后的期望行为             │
  │                                                          │
  │  证明：你理解了 bug 和正确行为                           │
  │                                                          │
  │  风险：                                                  │
  │  - 你可能理解错了 bug → 写了错误的测试                  │
  │  - 你可能理解错了正确行为 → 测试验证了错误的事          │
  │  - 你可能写了恒真断言 → 测试永远通过                    │
  │                                                          │
  │  这一步本身不证明测试有效——                              │
  │  它只是"声称"测试应该有效。                              │
  │  需要后续步骤来验证这个声称。                            │
  └──────────────────────────────────────────────────────────┘
```

### Step 2: Run (pass)（运行测试，通过）

```
  ┌──────────────────────────────────────────────────────────┐
  │  动作：运行测试，确认它通过                              │
  │                                                          │
  │  证明：测试和当前代码一致                                │
  │                                                          │
  │  不可证明的：                                            │
  │  - 测试是否是恒真断言（可能永远通过）                   │
  │  - 测试是否验证了正确的行为（可能碰巧通过）             │
  │  - 修复和测试之间是否有因果关系（可能巧合）             │
  │                                                          │
  │  单独这一步证明不了什么。                                │
  │  "测试通过"只是"当前状态和测试一致"——                 │
  │  但你不知道测试在"不一致"时会不会失败。                │
  │                                                          │
  │  类比：                                                  │
  │  你有一个烟雾报警器，它没响。                           │
  │  → 这可能意味着"没有火灾"                               │
  │  → 也可能意味着"报警器坏了"                             │
  │  → 你需要制造一点烟来测试报警器是否有效                 │
  │  → Step 3-4 就是"制造一点烟"                            │
  └──────────────────────────────────────────────────────────┘
```

### Step 3: Revert fix（撤回修复）

```
  ┌──────────────────────────────────────────────────────────┐
  │  动作：把 bug 放回去（git revert 或手动删除修复代码）    │
  │                                                          │
  │  证明：这是实验的"控制变量"操作                          │
  │                                                          │
  │  关键：你创造了一个"已知有 bug"的状态                   │
  │  → 在这个状态下，如果测试有效，它应该失败               │
  │  → 如果测试无效（恒真断言），它仍然通过                 │
  │  → 这一步为 Step 4 创造了"应该失败"的条件              │
  │                                                          │
  │  为什么需要 Revert 而不是"假设 bug 存在"？              │
  │  → 因为"假设"不是证据——你需要实际运行代码              │
  │  → 回归测试检测的是运行时行为，不是静态分析             │
  │  → 你必须在 bug 确实存在的运行时环境中运行测试          │
  │  → Revert 创造了这个环境                                 │
  │                                                          │
  │  技术实现：                                              │
  │  - git revert <修复的 commit>                            │
  │  - 或手动删除修复代码                                    │
  │  - 或用 feature flag 关闭修复                            │
  │  → 无论哪种方式，关键是让 bug 在运行时实际存在          │
  └──────────────────────────────────────────────────────────┘
```

### Step 4: Run (MUST FAIL)（运行测试，必须失败）

```
  ┌──────────────────────────────────────────────────────────┐
  │  动作：运行测试，确认它失败                              │
  │                                                          │
  │  证明：                                                  │
  │  ✓ 测试不是恒真断言（排除了恒真断言的可能）            │
  │  ✓ 测试确实在检测这个 bug（排除了碰巧通过的可能）      │
  │  ✓ 测试在"有 bug 时失败，无 bug 时通过"               │
  │    → 这正是回归测试需要满足的条件                       │
  │                                                          │
  │  "MUST FAIL" 的强制性：                                  │
  │  ─────────────────────                                   │
  │  不是"看看是否失败"——是"如果没失败，测试就是错的"。   │
  │                                                          │
  │  如果测试没失败：                                        │
  │  → 测试无效——它无法检测这个 bug                         │
  │  → 你需要重写测试                                       │
  │  → 不做这一步 = 你可能带着无效的回归测试继续前进        │
  │  → 无效的回归测试 = 虚假信心 = 未来 bug 回归时你不知道 │
  │                                                          │
  │  这是整个五步序列中最关键的一步：                        │
  │  Step 1-2 只能"声称"测试有效                           │
  │  Step 3 准备了验证环境                                  │
  │  Step 4 实际执行了验证                                  │
  │  Step 5 确认了因果方向                                  │
  │  → 没有 Step 4，整个验证就没有发生                     │
  └──────────────────────────────────────────────────────────┘
```

### Step 5: Restore → Run (pass)（恢复修复，运行测试，通过）

```
  ┌──────────────────────────────────────────────────────────┐
  │  动作：把修复放回去，运行测试，确认通过                  │
  │                                                          │
  │  证明：                                                  │
  │  ✓ 修复确实让测试从"失败"变为"通过"                   │
  │  ✓ 修复和测试之间存在因果关系（不是巧合）               │
  │  ✓ 回到安全状态（代码是修复后的状态，测试通过）        │
  │                                                          │
  │  为什么还需要这一步？Step 4 不是已经证明了吗？           │
  │  → Step 4 证明了"测试能检测 bug"                       │
  │  → 但 Step 4 之后代码处于"有 bug"状态                  │
  │  → 你需要回到"无 bug"状态                              │
  │  → 而且你需要确认"恢复修复后测试通过"                  │
  │    → 这确认了"修复 → 测试通过"的因果方向              │
  │    → 而不是"测试通过 → 修复"的反向解释                │
  │                                                          │
  │  因果方向的重要性：                                      │
  │  → Step 4 证明了"bug 存在 → 测试失败"                  │
  │  → Step 5 证明了"修复存在 → 测试通过"                  │
  │  → 两者的组合 = "测试通过 iff 修复存在"                │
  │  → 这是回归测试的充分必要条件                           │
  │  → 只有 Step 4 没有 Step 5 = 只有必要性没有充分性      │
  └──────────────────────────────────────────────────────────┘
```

## 五步的证明链

```
  ┌──────────────────────────────────────────────────────────┐
  │  Step 1: 声称 → "我认为这个测试能检测 bug"             │
  │  Step 2: 弱证 → "测试在当前代码上通过"                  │
  │  Step 3: 准备 → "我创造了 bug 存在的环境"               │
  │  Step 4: 强证 → "bug 存在时测试失败" ✓                  │
  │  Step 5: 确认 → "修复存在时测试通过" ✓                  │
  │                                                          │
  │  证明链：                                                │
  │  Step 4 + Step 5 =                                      │
  │  "测试在 bug 存在时失败，在 bug 不存在时通过"           │
  │  = 回归测试有效                                          │
  │  = 你可以信任这个测试在未来检测 bug 回归                │
  │                                                          │
  │  缺少 Step 4：                                          │
  │  → 你只有 Step 2 + Step 5                               │
  │  → 你只知道"测试通过"，不知道"测试会不会失败"         │
  │  → 回归测试的有效性未验证                               │
  │                                                          │
  │  缺少 Step 5：                                          │
  │  → 你只有 Step 4                                        │
  │  → 你知道"测试能检测 bug"                              │
  │  → 但代码还处于"有 bug"状态                            │
  │  → 而且你不知道修复是否能通过测试                       │
  │  → 回归测试的充分性未确认                               │
  │                                                          │
  │  → 五步缺一不可                                         │
  └──────────────────────────────────────────────────────────┘
```
