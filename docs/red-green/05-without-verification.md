# 05 不做 Red-Green 验证的风险

不做 Red-Green 验证不仅意味着"不知道测试是否有效"，更危险的是它会制造四种虚假信心——你"以为"有保障，但保障并不存在。

## 四种虚假信心

### 虚假信心 1: "我写了回归测试，所以 bug 不会回归"

```
  ┌──────────────────────────────────────────────────────────┐
  │  场景：你修了一个 bug，写了一个回归测试，测试通过。     │
  │  信心：测试会防止 bug 回归。                             │
  │                                                          │
  │  风险：                                                  │
  │  如果测试是恒真断言（永远通过）：                        │
  │  → bug 回归时测试仍然通过                               │
  │  → 你以为安全，实际不安全                               │
  │  → 比没有测试更危险——因为你不再警惕                    │
  │                                                          │
  │  类比：                                                  │
  │  你装了一个烟雾报警器，它从没响过。                     │
  │  → 你觉得"没有火灾" = 安全                             │
  │  → 但实际上可能是报警器坏了                              │
  │  → 真正的火灾来临时，报警器还是不响                    │
  │  → 你比没有报警器更危险——因为你依赖了它               │
  │                                                          │
  │  Red-Green 验证 = "制造一点烟，确认报警器会响"          │
  │  → 如果报警器不响 → 修好它再依赖它                     │
  │  → 如果报警器响了 → 你可以安全地依赖它                 │
  └──────────────────────────────────────────────────────────┘
```

### 虚假信心 2: "测试通过了，所以修复有效"

```
  ┌──────────────────────────────────────────────────────────┐
  │  场景：你修了 bug，测试通过。                            │
  │  信心：修复确实解决了 bug。                               │
  │                                                          │
  │  风险：                                                  │
  │  测试可能验证了错误的行为——                               │
  │  → 修复 A 让测试通过，但 bug 的真正原因是 B             │
  │  → 测试通过了，但 bug 可能以另一种方式存在              │
  │  → 你以为修好了，实际没修好                              │
  │                                                          │
  │  Red-Green 验证的风险检测：                              │
  │  Revert 修复 A → 测试失败 →                              │
  │  但如果你只修了表面症状（修复 A），                     │
  │  Revert 后测试失败只能证明"修复 A 让测试通过"          │
  │  不能证明"修复 A 解决了根因"                            │
  │                                                          │
  │  这就是为什么 Red-Green 验证应该和                      │
  │  systematic-debugging（先找根因再修复）结合使用。        │
  │  → Red-Green 验证的是"测试和修复的因果关系"            │
  │  → 不是"修复和根因的因果关系"                          │
  │  → 根因分析是调试技能的职责                              │
  └──────────────────────────────────────────────────────────┘
```

### 虚假信心 3: "测试覆盖了所有回归路径"

```
  ┌──────────────────────────────────────────────────────────┐
  │  场景：你写了一个回归测试，覆盖了 bug 的一个表现。      │
  │  信心：bug 不会以任何方式回归。                          │
  │                                                          │
  │  风险：                                                  │
  │  Bug 可能有多种表现形式：                                │
  │  → 测试检测了"空邮箱被接受"                            │
  │  → 但 bug 的根因是"验证函数跳过了空字符串"             │
  │  → 修复后，空字符串被验证了                              │
  │  → 但如果未来有人修改验证函数让它跳过 null              │
  │  → null 邮箱被接受了，但测试不检测 null                 │
  │  → 回归测试没有覆盖所有回归路径                         │
  │                                                          │
  │  Red-Green 验证不能解决"覆盖不全"的问题——              │
  │  它解决的是"已覆盖的路径是否真的有效"。                 │
  │                                                          │
  │  覆盖率问题的解决方案：                                  │
  │  → TDD 的增量方式：每个 RED 定义一个行为                │
  │  → 每个行为都有对应的测试                               │
  │  → 行为的完整性 = 需求的完整性（由 brainstorming 保证） │
  │  → Red-Green 保证"每个已有测试都有效"                  │
  │  → 需求分析保证"所有行为都有测试"                      │
  │  → 两者互补                                             │
  └──────────────────────────────────────────────────────────┘
```

### 虚假信心 4: "我不需要 Revert，我确定测试是对的"

```
  ┌──────────────────────────────────────────────────────────┐
  │  场景：你写了回归测试，测试通过。你觉得 Revert 太麻烦。  │
  │  信心：我的判断足够可靠，不需要额外验证。                │
  │                                                          │
  │  风险：                                                  │
  │  人类（包括 AI）对"自己写的测试是否有效"的判断         │
  │  系统性地过度自信。                                      │
  │                                                          │
  │  心理学基础：                                            │
  │  - 确认偏误：你倾向于寻找支持"测试有效"的证据          │
  │  - 知识诅咒：你理解了 bug，所以你的测试看起来显然有效  │
  │  - 计划谬误：你低估了测试出错的可能性                   │
  │  - 沉没成本：你已经花时间写了测试，不想再花时间验证    │
  │                                                          │
  │  Superpowers 的 Rationalization Table 中有对应条目：     │
  │  "I'm confident" → "Confidence ≠ evidence"              │
  │  "Just this once" → "No exceptions"                      │
  │  "Partial check is enough" → "Partial proves nothing"    │
  │                                                          │
  │  → "我确定测试是对的" = 确认偏误的信号                 │
  │  → 正因为你觉得不需要验证，才最需要验证                  │
  │  → 高信心 + 低验证 = 虚假信心的完美配方                │
  └──────────────────────────────────────────────────────────┘
```

## 虚假信心的累积效应

```
  ┌──────────────────────────────────────────────────────────┐
  │  虚假信心不只影响单个测试——它会累积。                   │
  │                                                          │
  │  第 1 个 bug 修复：不做 Red-Green 验证                   │
  │  → 可能 90% 的概率测试有效                               │
  │  → 10% 的概率测试无效                                    │
  │                                                          │
  │  第 2 个 bug 修复：不做 Red-Green 验证                   │
  │  → 可能 90% 的概率测试有效                               │
  │  → 10% 的概率测试无效                                    │
  │                                                          │
  │  ...                                                     │
  │                                                          │
  │  N 个 bug 修复后：                                       │
  │  所有测试都有效的概率 = 0.9^N                            │
  │                                                          │
  │  N = 10: 0.9^10 ≈ 0.35 → 65% 的概率至少一个测试无效    │
  │  N = 20: 0.9^20 ≈ 0.12 → 88% 的概率至少一个测试无效    │
  │  N = 50: 0.9^50 ≈ 0.005 → 99.5% 的概率至少一个无效     │
  │                                                          │
  │  → 随着项目增长，虚假信心的累积几乎保证                 │
  │    至少有一些回归测试是无效的。                          │
  │  → 你以为有 N 个回归测试保护，实际可能只有 0.9N 个有效  │
  │  → 随着时间推移，有效比例持续下降                       │
  │                                                          │
  │  Red-Green 验证的作用：                                  │
  │  将每个测试的有效概率从 ~90% 提升到 ~99%+                │
  │  → 0.99^50 ≈ 0.61 → 39% 的概率至少一个无效              │
  │  → 显著好于 99.5%                                       │
  │  → 虽然 Red-Green 不是完美保证，                        │
  │    但它将有效比例维持在高水平                            │
  └──────────────────────────────────────────────────────────┘
```

## 与 verification-before-completion 的关系

```
  ┌──────────────────────────────────────────────────────────┐
  │  Red-Green 验证是 verification-before-completion 的      │
  │  一个特例——它验证的是"回归测试"而非"普通测试"。       │
  │                                                          │
  │  verification-before-completion 的 Iron Law：            │
  │  "NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION        │
  │   EVIDENCE"                                              │
  │                                                          │
  │  应用到回归测试：                                        │
  │  "NO REGRESSION TEST CLAIMS WITHOUT RED-GREEN            │
  │   VERIFICATION EVIDENCE"                                 │
  │                                                          │
  │  → 声称"写了回归测试" = 完成声称                       │
  │  → Red-Green 验证 = 新鲜验证证据                        │
  │  → 没有 Red-Green 验证就声称"回归测试有效"             │
  │    = 没有验证证据就声称完成                             │
  │    = 违反 verification Iron Law                          │
  │                                                          │
  │  两个 Iron Law 的统一：                                  │
  │  "No X without Y" 的通用形式——                           │
  │  不论 X 是"写代码"/"声称完成"/"写回归测试"，          │
  │  Y 总是"验证证据"——                                    │
  │  验证 = 证据 = 信任的基础                                │
  └──────────────────────────────────────────────────────────┘
```
