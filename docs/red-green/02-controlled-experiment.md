# 02 对照实验

回归测试的 Red-Green 验证本质上是一个对照实验。它用科学方法证明"测试确实能检测 bug"，而不是依赖主观声称。

## 实验设计

```
  ┌──────────────────────────────────────────────────────────┐
  │  假设 H: "这个测试能检测 bug X"                         │
  │                                                          │
  │  实验设计：                                              │
  │                                                          │
  │  ┌──────────────────────────────────────────────┐       │
  │  │              实验组          对照组            │       │
  │  │              (修复后)        (撤回修复)        │       │
  │  │  ──────────────────  ──────────────────       │       │
  │  │  自变量:  bug 不存在      bug 存在             │       │
  │  │  因变量:  测试通过?       测试失败?            │       │
  │  │                                                │       │
  │  │  预期:   通过 ✓         失败 ✓                 │       │
  │  │  Step:   2, 5            4                     │       │
  │  └──────────────────────────────────────────────┘       │
  │                                                          │
  │  因果推断：                                              │
  │  自变量（bug 是否存在）变化时，                          │
  │  因变量（测试结果）随之变化 →                           │
  │  存在因果关系 →                                         │
  │  测试确实在检测 bug →                                   │
  │  假设 H 成立                                            │
  └──────────────────────────────────────────────────────────┘
```

## 对照组的关键性

```
  ┌──────────────────────────────────────────────────────────┐
  │  没有对照组的"实验"：                                   │
  │                                                          │
  │  实验组：bug 不存在 → 测试通过                           │
  │  → 你只知道"测试通过"                                   │
  │  → 你不知道测试通过是因为 bug 不存在                     │
  │    还是因为测试碰巧通过                                  │
  │  → 没有因果推断                                         │
  │                                                          │
  │  有对照组的实验：                                        │
  │                                                          │
  │  实验组：bug 不存在 → 测试通过                           │
  │  对照组：bug 存在   → 测试失败                           │
  │  → 两组之间唯一的差异 = bug 是否存在                    │
  │  → 测试结果的差异 = bug 的存在与否导致                  │
  │  → 因果推断成立                                         │
  │                                                          │
  │  Revert fix = 创建对照组                                 │
  │  MUST FAIL = 验证对照组的预期结果                        │
  │  → 没有 Revert = 没有对照组 = 没有科学验证              │
  │  → "I've written a regression test"                     │
  │    = 只有实验组 = 没有对照组 = 声称，不是证据          │
  └──────────────────────────────────────────────────────────┘
```

## 控制变量

```
  ┌──────────────────────────────────────────────────────────┐
  │  对照实验的有效性依赖于"控制其他变量"。                │
  │                                                          │
  │  在 Red-Green 验证中：                                   │
  │                                                          │
  │  实验组和对照组之间唯一的差异 = 修复代码是否存在         │
  │                                                          │
  │  必须控制的变量：                                        │
  │  1. 测试代码不变（同一个测试）                           │
  │  2. 测试环境不变（同一个运行环境）                       │
  │  3. 其他代码不变（只有修复代码被撤回/恢复）              │
  │  4. 依赖不变（同一个依赖版本）                           │
  │                                                          │
  │  如果控制失败：                                          │
  │  - 测试代码在 Revert 前后不同 → 差异不只是 bug          │
  │  - 环境在 Revert 前后不同 → 差异不只是 bug              │
  │  - 其他代码也变了 → 差异不只是 bug                      │
  │  → 因果推断不成立                                       │
  │  → Red-Green 验证无效                                   │
  │                                                          │
  │  这就是为什么 Revert 应该用 git revert（精确控制），     │
  │  而不是手动编辑（容易引入其他变化）。                    │
  └──────────────────────────────────────────────────────────┘
```

## 因果方向

```
  ┌──────────────────────────────────────────────────────────┐
  │  Step 4 (MUST FAIL) 和 Step 5 (pass) 确认了因果方向：   │
  │                                                          │
  │  方向 1: bug 存在 → 测试失败                             │
  │  证明了什么？                                            │
  │  → 测试对 bug 的存在敏感                                │
  │  → 测试不是恒真断言                                     │
  │  → 测试确实在验证与 bug 相关的行为                      │
  │                                                          │
  │  方向 2: 修复存在 → 测试通过                             │
  │  证明了什么？                                            │
  │  → 测试对 bug 的消除敏感                                │
  │  → 修复确实消除了测试检测到的问题                       │
  │  → 修复和测试之间存在因果关系                           │
  │                                                          │
  │  两个方向合在一起：                                      │
  │  测试通过 ⟺ 修复存在                                    │
  │  （测试通过当且仅当修复存在）                            │
  │                                                          │
  │  这就是回归测试的完整定义：                              │
  │  "一个测试，它在 bug 存在时失败，                       │
  │   在 bug 被修复时通过。"                                 │
  │  → 只有一个方向 = 定义不完整                             │
  │  → 两个方向 = 定义完整                                   │
  └──────────────────────────────────────────────────────────┘
```

## 与科学方法的对应

```
  ┌──────────────────────────────────────────────────────────┐
  │  科学方法         Red-Green 验证                         │
  │  ──────────       ──────────────────                     │
  │  提出假设         Step 1: 写回归测试                     │
  │  → H: 测试能检测 bug                                    │
  │                                                          │
  │  收集数据         Step 2: 运行测试（通过）               │
  │  → 观察: 测试在修复后通过                               │
  │                                                          │
  │  设计对照实验     Step 3: Revert fix                     │
  │  → 创建"bug 存在"的对照组                               │
  │                                                          │
  │  验证假设         Step 4: Run (MUST FAIL)                │
  │  → 对照组结果: 测试失败 = 假设得到支持                  │
  │                                                          │
  │  确认结果可重复   Step 5: Restore → Run (pass)           │
  │  → 实验组结果: 测试通过 = 假设再次得到支持              │
  │                                                          │
  │  结论             假设成立:                              │
  │  → 测试确实能检测 bug X                                 │
  │  → 回归测试有效                                         │
  │  → 可以信任此测试检测未来回归                            │
  │                                                          │
  │  关键洞察：                                              │
  │  Red-Green 验证不是"多余的仪式"——                      │
  │  它是科学方法在软件测试中的应用。                       │
  │  跳过它 = 跳过科学验证 = 基于信仰而非证据。             │
  └──────────────────────────────────────────────────────────┘
```
