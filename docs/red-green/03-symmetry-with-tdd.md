# 03 与 TDD Red-Green 的对称性

回归测试的 Red-Green 验证和 TDD 的 Red-Green 循环是同一个核心原则的两种应用：**如果没看到测试失败，你不知道测试是否有效**。

## 两种 Red-Green 的结构对比

```
  ┌──────────────────────────────────────────────────────────┐
  │  TDD Red-Green：                                        │
  │                                                          │
  │  RED:  写测试 → 功能缺失 → 测试失败                     │
  │        → 证明：测试在检测"功能不存在"                   │
  │                                                          │
  │  GREEN: 写代码 → 功能实现 → 测试通过                    │
  │        → 证明：代码确实让测试从"失败"变为"通过"        │
  │                                                          │
  │  回归测试 Red-Green：                                    │
  │                                                          │
  │  RED:  撤回修复 → bug 存在 → 测试失败                   │
  │        → 证明：测试在检测"bug 存在"                     │
  │                                                          │
  │  GREEN: 恢复修复 → bug 修复 → 测试通过                  │
  │        → 证明：修复确实让测试从"失败"变为"通过"        │
  │                                                          │
  │  对称性：                                                │
  │  ┌──────────────────────────────────────────────┐       │
  │  │              TDD              回归测试        │       │
  │  │  ────────────  ──────────────────────         │       │
  │  │  RED 原因: 功能缺失       bug 存在            │       │
  │  │  RED 手段: 不写功能       Revert 修复         │       │
  │  │  GREEN 原因: 功能实现     bug 修复             │       │
  │  │  GREEN 手段: 写代码       Restore 修复         │       │
  │  │  核心验证: 看到失败       看到失败              │       │
  │  │  核心结论: 测试有效       测试有效              │       │
  │  └──────────────────────────────────────────────┘       │
  └──────────────────────────────────────────────────────────┘
```

## 统一的核心原则

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的核心原则：                                │
  │  "If you didn't watch the test fail,                    │
  │   you don't know if it tests the right thing."          │
  │                                                          │
  │  TDD 如何实现"看到失败"：                               │
  │  → 先写测试（此时功能不存在）                           │
  │  → 运行测试 → 失败（因为功能确实不存在）               │
  │  → 你看到了失败 → 你知道测试有效                       │
  │                                                          │
  │  回归测试如何实现"看到失败"：                           │
  │  → 先写测试（此时 bug 已修复）                          │
  │  → 运行测试 → 通过（但你还不能信任它）                 │
  │  → Revert 修复 → 运行测试 → 失败                       │
  │  → 你看到了失败 → 你知道测试有效                       │
  │                                                          │
  │  两种方式都是"创造一个让测试失败的条件"，               │
  │  然后确认失败 = 证明测试的信息量 > 0。                  │
  │                                                          │
  │  差异在于"如何创造失败条件"：                           │
  │  TDD: 不写功能（功能自然不存在）                        │
  │  回归测试: 撤回修复（人工恢复 bug）                     │
  │                                                          │
  │  TDD 的失败条件更自然（功能还没写当然不存在）           │
  │  回归测试的失败条件更刻意（需要主动 Revert）             │
  │  → 回归测试的验证更容易被跳过                           │
  │  → 这就是为什么 Superpowers 强调 "MUST FAIL"            │
  │  → "MUST" = 强制 = 不能跳过                             │
  └──────────────────────────────────────────────────────────┘
```

## 为什么回归测试的验证更容易被跳过？

```
  ┌──────────────────────────────────────────────────────────┐
  │  TDD 的 "看到失败" 是流程的一部分：                     │
  │  → 你写了测试 → 运行它 → 它自然失败                    │
  │  → 你不需要额外做任何事情                               │
  │  → 失败是"默认状态"——你需要写代码才能让它通过         │
  │                                                          │
  │  回归测试的 "看到失败" 需要额外操作：                   │
  │  → 你写了测试 → 运行它 → 它通过了                      │
  │  → 然后你需要额外做 Revert + 再运行                     │
  │  → 失败不是"默认状态"——你需要主动创造它               │
  │  → 这需要额外的 2-5 分钟                                │
  │                                                          │
  │  心理障碍：                                              │
  │  - "测试已经通过了，为什么要 Revert？"                  │
  │  - "Revert 可能有副作用"                                │
  │  - "我确定测试是对的"                                   │
  │  - "多花 5 分钟不值得"                                  │
  │                                                          │
  │  这些都是合理化——Superpowers 的 Rationalization Table    │
  │  中有对应的条目：                                        │
  │  "I'm confident" → "Confidence ≠ evidence"              │
  │  "Just this once" → "No exceptions"                      │
  │  "Partial check is enough" → "Partial proves nothing"    │
  │                                                          │
  │  → 回归测试的验证更容易被跳过 →                        │
  │    Superpowers 用 "MUST FAIL" 强调其强制性              │
  │  → 不是"建议你做"，是"你必须做"                        │
  └──────────────────────────────────────────────────────────┘
```

## 何时用 TDD Red-Green，何时用回归 Red-Green？

```
  ┌──────────────────────────────────────────────────────────┐
  │  场景                    用哪种 Red-Green                │
  │  ────                    ─────────────────               │
  │  新功能开发              TDD Red-Green                   │
  │  → 先写测试，看到失败，再写代码                         │
  │                                                          │
  │  Bug 修复                两者结合                        │
  │  → TDD: 先写失败测试（描述期望行为）                   │
  │  → 修复代码让测试通过                                   │
  │  → 回归 Red-Green: Revert + MUST FAIL                   │
  │    确认测试确实在检测 bug                               │
  │                                                          │
  │  重构                    只需保持 Green                  │
  │  → 测试已经在 TDD 中写好                                │
  │  → 重构时保持测试通过 = 行为不变                        │
  │  → 不需要额外的 Red-Green 验证                          │
  │                                                          │
  │  添加回归测试到旧代码    回归 Red-Green                  │
  │  → 旧代码没有 TDD 测试                                  │
  │  → 写回归测试后必须 Revert + MUST FAIL                  │
  │  → 因为你没有 TDD 的"先失败后通过"的验证              │
  │                                                          │
  │  Superpowers 的 Debugging Integration：                  │
  │  "Bug found? Write failing test reproducing it.          │
  │   Follow TDD cycle. Test proves fix and                  │
  │   prevents regression. Never fix bugs without a test."   │
  │  → Bug 修复 = TDD + 回归 Red-Green 的结合              │
  └──────────────────────────────────────────────────────────┘
```

## 两种验证的互补

```
  ┌──────────────────────────────────────────────────────────┐
  │  TDD Red-Green 验证了：                                  │
  │  → 测试能检测"功能缺失"                                │
  │  → 实现满足测试的要求                                   │
  │  → 但不验证：测试能检测"bug 引入"（回归）              │
  │                                                          │
  │  回归 Red-Green 验证了：                                 │
  │  → 测试能检测"bug 回归"                                │
  │  → 修复确实消除了 bug                                   │
  │  → 但不验证：测试在最初开发时是否有效                   │
  │                                                          │
  │  互补关系：                                              │
  │  TDD Red-Green: 保证测试在开发时有效                    │
  │  回归 Red-Green: 保证测试在未来回归时有效               │
  │  → 开发时有效 + 回归时有效 = 测试在任何时候都有效      │
  │                                                          │
  │  但 Superpowers 中 TDD 的 Verify RED 步骤               │
  │  实际上已经包含了回归 Red-Green 的核心：                 │
  │  - Verify RED = 确认测试在功能缺失时失败               │
  │  = 确认测试能检测"功能不存在"                          │
  │  = 回归检测的基础（如果功能被意外删除，测试会失败）   │
  │                                                          │
  │  → TDD 的 Verify RED 已经隐含了回归测试的验证          │
  │  → 但对于 bug 修复场景，显式的回归 Red-Green            │
  │    仍然必要——因为 bug 不是"功能缺失"，是"行为错误"   │
  │  → 你需要验证测试对"行为错误"也敏感                   │
  └──────────────────────────────────────────────────────────┘
```
