这是一个非常敏锐的问题。你问：“为什么 TDD 要求看到的是**错误失败**，而不是**测试成功**？”——这触及了 TDD 认知逻辑的核心。

## 简短回答

因为 **在写任何实现代码之前，测试“成功”是一件坏事，而不是好事**。

- **测试失败（RED）** → 证明你的测试**能够**检测到功能的缺失 → 测试本身是有效的。
- **测试成功（GREEN）** → 要么功能已经存在，要么测试写错了（比如永远为真的断言）→ 你无法信任这个测试。

TDD 的目标不是“尽快让测试通过”，而是 **“建立对测试的信任”**。而信任的唯一方式，就是在你知道功能不存在的时候，亲眼看到测试**确实**失败了。

---

## 详细解释：为什么“看到失败”才是成果？

### 1. 测试的两种“成功”

| 情况 | 含义 | 可信度 |
|------|------|--------|
| **测试 RED（失败）** | 测试检测到了缺失功能 | ✅ 有效测试 |
| **测试 GREEN（通过）** | 要么功能已实现，要么测试有 bug | ❓ 不确定 |

如果你先写测试，然后它**立即通过**，可能的原因包括：

- 你要添加的功能已经存在了（那这个测试就没意义）
- 测试断言写反了（比如 `expect(thing).toBe(false)` 而实际是 `true`，但你本意要测 `true`）
- 测试永远返回真（例如 `expect(true).toBe(true)`）
- 异步测试没有正确 `await`，导致断言根本没执行

**如果不先看到失败，你就无法排除上述可能性。**

### 2. 类比：验证烟雾报警器

你买了一颗烟雾报警器，想测试它是否有效。你会怎么做？

- ✅ **正确方法**：人工制造一点烟雾（比如点燃一根香），看它是否报警（**先看到失败——报火警**）。  
- ❌ **错误方法**：没有烟雾时，看到它沉默（**测试成功**），就认为它工作正常。

你必须在**你知道有危险**的情况下，亲眼看到它发出警报，才能相信它。TDD 的 RED 阶段就是那个“人工烟雾”——你在实现代码之前，故意让测试面对一个“没有功能”的环境，观察它是否正确地报错。

### 3. 如果跳过 RED，会怎样？

假设你写了一个测试，没有运行它就直接写实现，然后测试通过了。你会有如下错觉：

> “我的测试通过了，说明代码正确。”

但实际上，你从未验证过测试本身是否“诚实”。可能你写的实现有 bug，但测试恰好也写错了，两者互相抵消。例如：

```javascript
// 实现（本应是加法，实际写成了减法）
function add(a, b) { return a - b; }

// 测试（本应测 1+2=3，但错误地预期为 -1）
test('add', () => { expect(add(1, 2)).toBe(-1); });
```

运行测试 → 通过。你误以为加法正确，实际上错误被隐藏了。  
如果先运行测试（RED），你会看到 `add is not defined`，然后写最小实现 `return a + b`，再运行测试（GREEN），才能保证实现与测试都对。

---

## 总结：TDD 的认知顺序

```
写测试 → 运行测试 → 看到 RED（失败） → 确认测试有效 → 写实现 → 看到 GREEN（成功） → 完成
```

- **RED** 是你对测试的“校准”——证明测试能正确区分“有功能”和“无功能”。
- **GREEN** 是你对实现的“验证”——证明代码满足了测试的要求。

没有 RED 的 GREEN 是脆弱的，因为你不知道绿色是因为代码对了，还是因为测试错了。**看到失败才是成果，因为它给了你信任测试的理由**。
