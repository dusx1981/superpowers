# 04 "失败"即证明

Red-Green 验证的核心洞察：测试的"失败"比"通过"更有信息量。看到失败 = 证明测试有效；没看到失败 = 不知道测试是否有效。

## 信息论视角

```
  ┌──────────────────────────────────────────────────────────┐
  │  一个测试的结果可以是"通过"或"失败"。                   │
  │  两种结果的信息量不同。                                 │
  │                                                          │
  │  "通过"的信息量：                                       │
  │  ──────────────────                                      │
  │  测试通过可能因为：                                      │
  │  A. 代码正确 + 测试正确 → 真通过                        │
  │  B. 代码错误 + 测试错误 → 假通过                        │
  │  C. 测试恒真 → 永远通过                                 │
  │  → "通过"无法区分 A/B/C → 信息量低                     │
  │                                                          │
  │  "失败"的信息量：                                       │
  │  ──────────────────                                      │
  │  测试失败意味着：                                        │
  │  A. 测试不是恒真断言（否则不会失败）                    │
  │  B. 测试确实在验证某个条件                               │
  │  C. 被验证的条件在当前代码中不满足                      │
  │  → "失败"排除了恒真断言 → 信息量高                     │
  │                                                          │
  │  信息量排序：                                            │
  │  "通过"（无先验失败）< "失败" < "失败 + 通过"          │
  │                                                          │
  │  → 只有"先失败后通过"的测试才有最高的信息量             │
  │  → Red-Green = 先失败后通过 = 最高信息量                │
  │  → 只看"通过" = 信息量不足 = 无法信任                  │
  └──────────────────────────────────────────────────────────┘
```

## "MUST FAIL" 的强制性

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的表达：                                    │
  │  "Write → Run (pass) → Revert fix →                    │
  │   Run (MUST FAIL) → Restore → Run (pass)"               │
  │                                                          │
  │  "MUST FAIL" 不是描述性的（"应该会失败"），             │
  │  而是规范性的（"必须失败，否则测试无效"）。              │
  │                                                          │
  │  两种读法的差异：                                        │
  │                                                          │
  │  描述性读法：                                            │
  │  "Revert 后运行测试，看看是否失败"                      │
  │  → 如果没失败 → "嗯，没失败" → 继续前进               │
  │  → 测试可能无效，但你没注意到                           │
  │                                                          │
  │  规范性读法：                                            │
  │  "Revert 后运行测试，必须失败"                          │
  │  → 如果没失败 → "测试无效，必须重写" → 停下来          │
  │  → 你被迫面对测试可能无效的事实                         │
  │                                                          │
  │  "MUST" 的作用：                                        │
  │  - 将"可选验证"变成"强制验证"                          │
  │  - 将"看看"变成"确认"                                  │
  │  - 将"失败是意外"变成"不失败是意外"                   │
  │                                                          │
  │  没通过 ≠ 测试有效                                       │
  │  没失败 ≠ 测试有效                                       │
  │  先失败后通过 = 测试有效                                 │
  │  → MUST FAIL 确认了"失败"这一半的验证                  │
  └──────────────────────────────────────────────────────────┘
```

## 失败的"正确原因"

```
  ┌──────────────────────────────────────────────────────────┐
  │  不仅需要看到失败，还需要失败的原因正确。               │
  │                                                          │
  │  TDD Verify RED 的三个确认项（同源原理）：               │
  │  1. Test fails (not errors)                              │
  │  2. Failure message is expected                          │
  │  3. Fails because feature missing (not typos)            │
  │                                                          │
  │  回归测试的 MUST FAIL 也需要确认失败原因：               │
  │  1. 测试失败是因为 bug 存在（不是其他原因）             │
  │  2. 失败消息和 bug 相关（不是不相关的错误）             │
  │  3. 失败是 bug 导致的（不是 Revert 的副作用）           │
  │                                                          │
  │  错误的失败原因示例：                                    │
  │  - Revert 导致编译错误 → 测试失败是因为编译错误         │
  │    不是因为测试检测到了 bug                              │
  │  - Revert 破坏了其他依赖 → 测试失败是因为依赖断裂      │
  │    不是因为测试检测到了 bug                              │
  │  - 测试和修复代码紧耦合 → 撤回修复导致测试代码无效     │
  │    不是因为测试检测到了行为变化                          │
  │                                                          │
  │  正确的失败原因：                                        │
  │  Revert 后 → bug 的行为恢复了 →                        │
  │  测试检测到行为与期望不符 → 测试失败                    │
  │  → 失败原因 = 行为偏差（不是编译错误或依赖问题）       │
  │  → 这才是有效的 MUST FAIL                               │
  └──────────────────────────────────────────────────────────┘
```

## 失败的可重复性

```
  ┌──────────────────────────────────────────────────────────┐
  │  Red-Green 验证创建了一个可重复的失败-通过模式：        │
  │                                                          │
  │  状态 A (bug 存在)    → 测试失败                         │
  │  状态 B (bug 不存在)  → 测试通过                         │
  │                                                          │
  │  A → B: 测试从"失败"变为"通过"                         │
  │  B → A: 测试从"通过"变为"失败"                         │
  │                                                          │
  │  可重复意味着：                                          │
  │  - 你可以反复切换 A/B，测试每次都给出正确结果           │
  │  - 未来 bug 回归（回到状态 A），测试仍会失败            │
  │  - 这就是"回归测试"的"回归"——                        │
  │    bug 如果再次出现，测试会再次失败                     │
  │                                                          │
  │  不可重复的验证（只做一次"通过"）：                     │
  │  - 你只知道"这一次通过了"                               │
  │  - 你不知道"下一次会不会通过"                           │
  │  - 你不知道"代码变化后会不会通过"                       │
  │  - 单次通过 ≠ 持续有效                                  │
  │                                                          │
  │  Red-Green = 双向可重复                                  │
  │  → bug 存在 → 总是失败                                  │
  │  → bug 不存在 → 总是通过                                │
  │  → 双向可重复 = 回归测试的可靠性保证                   │
  └──────────────────────────────────────────────────────────┘
```

## "失败"作为测试质量的衡量

```
  ┌──────────────────────────────────────────────────────────┐
  │  测试的质量可以用"它失败时的行为"来衡量：              │
  │                                                          │
  │  高质量测试：                                            │
  │  - 在 bug 存在时失败                                     │
  │  - 失败消息清晰，指出问题所在                           │
  │  - 在 bug 不存在时通过                                   │
  │  - 不在无关变化时失败（不脆弱）                         │
  │                                                          │
  │  低质量测试：                                            │
  │  - 在 bug 存在时可能失败也可能不失败                    │
  │  - 失败消息模糊（"Assertion failed"）                   │
  │  - 在无关变化时也失败（脆弱测试）                       │
  │  - 永远通过（恒真断言）                                 │
  │                                                          │
  │  Red-Green 验证区分了高质量和低质量测试：               │
  │  - 高质量测试在 MUST FAIL 步骤确实失败                  │
  │  - 低质量测试在 MUST FAIL 步骤可能不失败                │
  │    （恒真断言、错误行为验证、脆弱测试）                 │
  │                                                          │
  │  → Red-Green 不仅是"验证测试有效"，                    │
  │    也是"衡量测试质量"的工具。                           │
  │  → MUST FAIL 失败了 = 测试质量合格                     │
  │  → MUST FAIL 没失败 = 测试质量不合格                   │
  └──────────────────────────────────────────────────────────┘
```
