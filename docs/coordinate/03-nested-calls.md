# 03 嵌套调用

嵌套调用是协调的第三层——技能内部调用其他技能，形成调用链。每个技能不需要知道整个系统，只需要知道自己的上下游。

## subagent-driven-development 的嵌套结构

```
  ┌──────────────────────────────────────────────────────────┐
  │  subagent-driven-development                             │
  │  │                                                      │
  │  ├── 实现者子智能体                                     │
  │  │   └── 使用 TDD (test-driven-development)              │
  │  │       → 实现者内部遵循 Red-Green-Refactor             │
  │  │       → 子智能体不知道自己在"subagent-driven"流程中  │
  │  │       → 它只知道自己要做 TDD                        │
  │  │                                                      │
  │  ├── 规范审阅者子智能体                                 │
  │  │   └── 使用 spec-reviewer-prompt.md                    │
  │  │       → 内嵌不信任原则                               │
  │  │       → 审阅者不知道有实现者存在                     │
  │  │       → 它只知道要审阅代码是否匹配规范               │
  │  │                                                      │
  │  ├── 代码质量审阅者子智能体                             │
  │  │   └── 使用 requesting-code-review                     │
  │  │       → 审阅模板                                     │
  │  │       → 审阅者不知道有规范审阅者存在                 │
  │  │       → 它只知道要审阅代码质量                       │
  │  │                                                      │
  │  └── 终态                                               │
  │      └── finishing-a-development-branch                  │
  │          → 子智能体流程结束后由 Controller 调用           │
  │                                                          │
  │  嵌套结构：                                             │
  │  ┌─────────────────────────────────────┐                │
  │  │ subagent-driven-development         │                │
  │  │  ┌───────────────────────────────┐  │                │
  │  │  │ 实现者 → TDD                 │  │                │
  │  │  └───────────────────────────────┘  │                │
  │  │  ┌───────────────────────────────┐  │                │
  │  │  │ 规范审阅者 → 不信任原则      │  │                │
  │  │  └───────────────────────────────┘  │                │
  │  │  ┌───────────────────────────────┐  │                │
  │  │  │ 代码审阅者 → 审阅模板        │  │                │
  │  │  └───────────────────────────────┘  │                │
  │  │  ┌───────────────────────────────┐  │                │
  │  │  │ finishing-a-development-branch │  │                │
  │  │  └───────────────────────────────┘  │                │
  │  └─────────────────────────────────────┘                │
  └──────────────────────────────────────────────────────────┘
```

## 嵌套调用的"去中心化"特性

```
  ┌──────────────────────────────────────────────────────────┐
  │  每个技能只需要知道两件事：                             │
  │  1. 我该做什么（自己的职责）                            │
  │  2. 下一步是什么（唯一出口/依赖）                      │
  │                                                          │
  │  不需要知道：                                            │
  │  - 整个系统有多少技能                                   │
  │  - 其他技能的内部实现                                   │
  │  - 自己在整个流程中的位置                               │
  │                                                          │
  │  类比：Unix 管道                                        │
  │  sort | uniq | wc                                        │
  │  → sort 只知道排序，不知道 uniq 和 wc 存在             │
  │  → uniq 只知道去重，不知道 sort 和 wc 存在             │
  │  → wc 只知道计数，不知道 sort 和 uniq 存在             │
  │  → 但组合后实现了"排序去重计数"                       │
  │                                                          │
  │  Superpowers 的技能同理：                               │
  │  → brainstorming 只知道设计和 writing-plans              │
  │  → writing-plans 只知道计划和 subagent/executing         │
  │  → subagent 只知道执行和 finishing                      │
  │  → 组合后实现了"设计→计划→执行→收尾"                 │
  │                                                          │
  │  去中心化的好处：                                       │
  │  - 新增技能不需要修改已有技能                          │
  │  - 修改一个技能不影响其他技能                          │
  │  - 每个技能可以被独立测试和验证                        │
  │  - 系统可以自然扩展（加入新技能）                      │
  └──────────────────────────────────────────────────────────┘
```

## REQUIRED SUB-SKILL 声明

```
  ┌──────────────────────────────────────────────────────────┐
  │  技能通过 "REQUIRED SUB-SKILL" 声明依赖：               │
  │                                                          │
  │  writing-plans 的计划文档头：                            │
  │  "REQUIRED SUB-SKILL: Use                                 │
  │   superpowers:subagent-driven-development                 │
  │   (recommended) or superpowers:executing-plans            │
  │   to implement this plan task-by-task."                   │
  │                                                          │
  │  subagent-driven-development 的 Integration 节：         │
  │  "Required workflow skills:                              │
  │   - superpowers:using-git-worktrees - REQUIRED"           │
  │   - superpowers:writing-plans                             │
  │   - superpowers:requesting-code-review                   │
  │   - superpowers:finishing-a-development-branch"           │
  │                                                          │
  │  "Subagents should use:                                  │
  │   - superpowers:test-driven-development"                  │
  │                                                          │
  │  依赖声明的三个层次：                                   │
  │  1. REQUIRED = 必须使用，不可跳过                       │
  │  2. Recommended = 强烈建议，有备选                      │
  │ 3. Should use = 子智能体内部使用                        │
  │                                                          │
  │  → 每个技能在代码中声明了自己的依赖                     │
  │  → 不需要外部文档来描述技能之间的关系                  │
  │  → 代码即文档，依赖即协调                              │
  └──────────────────────────────────────────────────────────┘
```

## 嵌套调用的信息封装

```
  ┌──────────────────────────────────────────────────────────┐
  │  嵌套调用中的信息封装：                                 │
  │                                                          │
  │  Controller → 实现者子智能体：                           │
  │  传入：任务描述 + 上下文                                 │
  │  不传入：其他任务、用户偏好、Controller 的决策逻辑       │
  │  → 实现者只知道自己要做的这一件事                       │
  │                                                          │
  │  Controller → 规范审阅者子智能体：                       │
  │  传入：规范文本 + 实现者报告                             │
  │  不传入：实现者的思考过程、其他审阅者的结果             │
  │  → 审阅者独立判断，不受实现者影响                       │
  │                                                          │
  │  Controller → 代码审阅者子智能体：                       │
  │  传入：实现代码 + 审阅范围 (BASE_SHA..HEAD_SHA)          │
  │  不传入：规范审阅者的判断、实现者的意图                 │
  │  → 审阅者独立判断，不受规范审阅影响                     │
  │                                                          │
  │  信息封装的好处：                                       │
  │  - 每个子智能体/技能只看到自己需要的信息               │
  │  - 不存在"信息泄漏"导致的偏见                          │
  │  - 各环节独立判断 → 结果更客观                         │
  │  - 类似于微服务的数据隔离                              │
  └──────────────────────────────────────────────────────────┘
```
