# 11 上下文构造的五个关键模式

从三个注入模式中提炼出五个跨模式的关键设计模式。这些模式反复出现在不同技能的上下文构造中，是 Controller 协调能力的核心机制。

## 模式一：信息最小化——只给需要知道的

```
  ┌──────────────────────────────────────────────────────────┐
  │  原则：每个技能/子智能体只获得完成其职责所需的最少信息  │
  │                                                          │
  │  实现者获得：任务全文 + 场景设定 + 行为规则              │
  │  规范审阅者获得：需求全文 + 实现者声称 + 不信任框架      │
  │  代码审阅者获得：git diff + 任务摘要 + 审阅模板          │
  │                                                          │
  │  三者都不获得：                                          │
  │  - 其他技能的完整规则                                    │
  │  - 用户的完整对话历史                                    │
  │  - 其他子智能体的判断结果                                │
  │  - 项目的完整架构知识                                    │
  │                                                          │
  │  为什么不是"给所有信息让 AI 自己判断"？                  │
  │                                                          │
  │  实验对比：                                              │
  │  ┌─ 给所有信息 ─────────────────────────────────────┐   │
  │  │ 子智能体看到：                                    │   │
  │  │ - 完整对话历史（200+ 条消息）                     │   │
  │  │ - 所有技能规则                                    │   │
  │  │ - 其他审阅者的判断                                │   │
  │  │                                                    │   │
  │  │ 结果：                                            │   │
  │  │ - 信息过载 → 忽略关键指令                         │   │
  │  │ - 偏见传染 → 规范审阅者受代码审阅影响             │   │
  │  │ - 上下文窗口溢出 → 截断早期关键指令               │   │
  │  │ - 行为不可预测 → 同一任务不同结果                 │   │
  │  └──────────────────────────────────────────────────┘   │
  │                                                          │
  │  ┌─ 给最小信息 ─────────────────────────────────────┐   │
  │  │ 子智能体看到：                                    │   │
  │  │ - Controller 精选的必要信息                       │   │
  │  │                                                    │   │
  │  │ 结果：                                            │   │
  │  │ - 信息密度高 → 关键指令不被淹没                   │   │
  │  │ - 无偏见 → 各环节独立判断                         │   │
  │  │ - 上下文可控 → 不会溢出                           │   │
  │  │ - 行为可预测 → 同一任务一致结果                   │   │
  │  └──────────────────────────────────────────────────┘   │
  │                                                          │
  │  信息最小化 = Need-to-Know 原则                          │
  │  → 类比：军事机密的"需要知道"原则                      │
  │  → 不是"不能知道"，而是"不需要知道"                   │
  │  → 多余信息是噪音，不是信号                             │
  └──────────────────────────────────────────────────────────┘
```

## 模式二：偏见隔离——每个环节独立判断

```
  ┌──────────────────────────────────────────────────────────┐
  │  原则：不同技能/子智能体之间不共享判断结果              │
  │                                                          │
  │  如果规范审阅者知道代码审阅者已经通过：                  │
  │  → 可能放松自己的标准（"既然代码质量好，规范可能也OK"） │
  │  → 从众心理——人类和 AI 都有                            │
  │                                                          │
  │  如果实现者知道审阅者的判断标准：                        │
  │  → 可能针对性地"通过审阅"而非真正实现需求              │
  │  → Goodhart 法则：当指标成为目标，它就不再是好指标      │
  │                                                          │
  │  Superpowers 的偏见隔离设计：                            │
  │                                                          │
  │  实现者 → 不知道审阅标准（只知道自己要实现什么）        │
  │  规范审阅者 → 不知道代码审阅者的存在和结果              │
  │  代码审阅者 → 不知道规范审阅者的判断                    │
  │  Controller → 综合所有结果，但不做技术判断               │
  │                                                          │
  │  隔离的实现方式：                                        │
  │  1. 子智能体独立上下文（结构性隔离）                    │
  │  2. Controller 不传递其他子智能体的判断（信息隔离）      │
  │  3. "Do Not Trust the Report"（认知隔离——不信任实现者） │
  │                                                          │
  │  三层隔离确保了：                                        │
  │  → 每个环节的判断是独立的                               │
  │  → 错误不会从一个环节传播到另一个环节                  │
  │  → 多个独立判断的交集比单个判断更可靠                  │
  │                                                          │
  │  类比：                                                  │
  │  双盲实验 —— 实验者和被试都不知道谁在实验组             │
  │  → Superpowers 的审阅流程是"三盲"：                    │
  │    实现者不知道审阅标准                                  │
  │    审阅者不知道彼此的判断                                │
  │    Controller 不做技术判断                               │
  └──────────────────────────────────────────────────────────┘
```

## 模式三：预设立场——通过框架引导行为

```
  ┌──────────────────────────────────────────────────────────┐
  │  原则：在给子智能体信息之前，先设定它的立场/态度        │
  │                                                          │
  │  实现者的预设立场：                                      │
  │  "You are implementing Task N"                            │
  │  → 立场：我是实现者，我的工作是实现任务                  │
  │  → "Implement exactly what the task specifies"            │
  │  → 预设：不做额外的事（YAGNI）                           │
  │  → "It is always OK to stop and say                       │
  │     'this is too hard for me.'"                           │
  │  → 预设：升级是合法的，不是失败                          │
  │                                                          │
  │  规范审阅者的预设立场：                                  │
  │  "The implementer finished suspiciously quickly."          │
  │  → 立场：实现者的报告不可信                              │
  │  → "DO NOT take their word for what they implemented"     │
  │  → 预设：独立验证一切                                    │
  │  → "DO NOT accept their interpretation of requirements"   │
  │  → 预设：实现者可能理解错了需求                          │
  │                                                          │
  │  代码审阅者的预设立场：                                  │
  │  通过 code-reviewer.md 模板设定                          │
  │  → 立场：我是审阅者，我评估代码质量                     │
  │  → 审阅维度已预设：Strengths / Issues / Assessment        │
  │  → 不需要"不信任"预设——审阅者天然看代码不看报告        │
  │                                                          │
  │  预设立场 vs. 中立立场：                                 │
  │                                                          │
  │  中立立场：                                              │
  │  "请检查这个实现是否正确"                                │
  │  → AI 可能倾向于信任实现者（锚定效应）                  │
  │  → 可能只做表面检查                                      │
  │  → 可能"不好意思"指出问题                              │
  │                                                          │
  │  预设立场：                                              │
  │  "实现者的报告可能不完整、不准确、或过度乐观"            │
  │  → AI 从怀疑出发，主动找问题                             │
  │  → 会读实际代码，不只看报告                              │
  │  → 会检查是否有遗漏/多余/误解                            │
  │                                                          │
  │  预设立场 = 认知锚点                                     │
  │  → 在 AI 开始处理信息之前，先设定它的"初始位置"         │
  │  → 初始位置决定了后续判断的偏向                          │
  │  → 类比：法官的"无罪推定"vs."有罪推定"               │
  └──────────────────────────────────────────────────────────┘
```

## 模式四：信息转换——Controller 的翻译角色

```
  ┌──────────────────────────────────────────────────────────┐
  │  原则：Controller 在传递信息时做"翻译"，而非"转发"      │
  │                                                          │
  │  翻译 vs. 转发：                                         │
  │                                                          │
  │  转发：                                                  │
  │  Controller 把实现者的完整报告原封不动传给审阅者         │
  │  → 审阅者可能被实现者的思考过程影响                      │
  │  → 审阅者可能被实现者的解释说服                          │
  │  → 信息过载——报告可能很长                                │
  │                                                          │
  │  翻译：                                                  │
  │  Controller 提取报告中的关键信息，重组为审阅者需要的格式 │
  │  → "What Implementer Claims They Built"                   │
  │    = 实现者声称做了什么（不是为什么这样做）              │
  │  → 信息被过滤——只保留审阅者需要的                      │
  │  → 信息被重组——按审阅者的需要排列                      │
  │                                                          │
  │  Controller 的三种翻译动作：                              │
  │                                                          │
  │  翻译 1: 计划 → 任务全文                                 │
  │  → Controller 从计划文件中提取单个任务的全文             │
  │  → 不传整个计划文件（太长，且包含其他任务）              │
  │  → 不传任务摘要（信息不足，可能遗漏细节）               │
  │  → 传任务全文（精确、完整、无歧义）                      │
  │                                                          │
  │  翻译 2: 实现者报告 → 审阅者输入                         │
  │  → Controller 从实现者报告中提取"声称做了什么"          │
  │  → 不传实现者的思考过程                                  │
  │  → 不传实现者的自审结果（审阅者应该自己判断）            │
  │  → 只传"声称"——让审阅者验证"声称"是否为真             │
  │                                                          │
  │  翻译 3: 审阅者反馈 → 实现者修复指令                     │
  │  → Controller 把审阅者的 ✅/❌ 转化为具体修复指令        │
  │  → 如果规范审阅者说"缺少空配置处理"                     │
  │  → Controller 告诉实现者"添加空配置处理"                │
  │  → 不是传审阅者的完整报告——而是翻译为可执行的指令       │
  │                                                          │
  │  翻译的三个原则：                                        │
  │  1. 精确——不丢失关键信息                                │
  │  2. 最小——不传递不必要的信息                            │
  │  3. 重组——按接收者的需要排列信息                        │
  └──────────────────────────────────────────────────────────┘
```

## 模式五：上下文生命周期管理——创建、使用、丢弃

```
  ┌──────────────────────────────────────────────────────────┐
  │  原则：上下文是有限资源，必须主动管理生命周期            │
  │                                                          │
  │  上下文的三个阶段：                                      │
  │                                                          │
  │  ┌─ 创建 ──────────────────────────────────────────┐    │
  │  │  主链技能：Skill("name") 调用 → 追加到对话       │    │
  │  │  子智能体：Task("prompt") 调用 → 全新上下文       │    │
  │  │  启动注入：messages.transform → 插入首条消息      │    │
  │  │                                                    │    │
  │  │  创建时的关键决策：                                │    │
  │  │  - 包含什么信息（信息最小化）                      │    │
  │  │  - 排除什么信息（偏见隔离）                        │    │
  │  │  - 设定什么立场（预设立场）                        │    │
  │  │  - 如何组织信息（信息转换）                        │    │
  │  └──────────────────────────────────────────────────┘    │
  │                                                          │
  │  ┌─ 使用 ──────────────────────────────────────────┐    │
  │  │  主链技能：AI 在对话中持续遵循技能规则             │    │
  │  │  子智能体：AI 在隔离上下文中执行单一任务           │    │
  │  │  启动注入：AI 在每条消息中检查技能匹配             │    │
  │  │                                                    │    │
  │  │  使用时的关键风险：                                │    │
  │  │  - 上下文膨胀（主链）——对话越长，上下文越大        │    │
  │  │  - 上下文遗忘（长对话）——AI 可能忽略早期指令      │    │
  │  │  - 上下文冲突（多技能）——旧技能规则可能干扰新技能  │    │
  │  └──────────────────────────────────────────────────┘    │
  │                                                          │
  │  ┌─ 丢弃 ──────────────────────────────────────────┐    │
  │  │  主链技能：技能完成但内容仍在上下文中               │    │
  │  │    → "软丢弃"——AI 注意力转移，旧规则仍在但    │    │
  │  │      优先级降低                                    │    │
  │  │  子智能体：任务完成后上下文被销毁                   │    │
  │  │    → "硬丢弃"——上下文完全消失，不回传给          │    │
  │  │      Controller                                   │    │
  │  │  启动注入：会话结束时整体销毁                       │    │
  │  │    → 会话是上下文的生命周期边界                    │    │
  │  └──────────────────────────────────────────────────┘    │
  │                                                          │
  │  生命周期管理的关键洞察：                                │
  │                                                          │
  │  子智能体的"硬丢弃"是最有效的上下文管理：              │
  │  → 实现者完成任务 → 上下文销毁                          │
  │  → 规范审阅者完成审阅 → 上下文销毁                      │
  │  → 代码审阅者完成审阅 → 上下文销毁                      │
  │  → 每个子智能体都从零开始，没有历史包袱                 │
  │  → 这就是 subagent-driven 优于 inline 执行的核心原因    │
  │                                                          │
  │  主链的"软丢弃"是妥协：                                │
  │  → brainstorming 完成后内容仍在上下文中                  │
  │  → writing-plans 完成后内容仍在上下文中                  │
  │  → 上下文持续增长，AI 可能遗忘早期指令                  │
  │  → 唯一出口和 "Announce at start" 是缓解手段            │
  │                                                          │
  │  理想 vs. 现实：                                        │
  │  理想：所有上下文都能"硬丢弃"                           │
  │  现实：主链技能共享对话，无法"硬丢弃"                   │
  │  → 子智能体派发 = 在现实约束下尽可能接近理想            │
  └──────────────────────────────────────────────────────────┘
```
