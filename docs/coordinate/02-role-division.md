# 02 角色分工

角色分工是协调的第二层——每个技能只做一件事，不做其他技能的事。无重叠 = 无冲突。

## 技能的唯一职责

```
  ┌──────────────────────────────────────────────────────────┐
  │  技能              唯一职责         不做的事              │
  │  ────              ──────────       ────────              │
  │  brainstorming     定义"做什么"     不写代码、不写计划    │
  │  writing-plans     定义"怎么做"     不写代码、不做设计    │
  │  subagent-driven   执行"做了"       不做设计、不写计划    │
  │  executing-plans   执行"做了"       不做设计、不写计划    │
  │  TDD               保证"做对了"     不做设计、不写计划    │
  │  verification      保证"做完了"     不做设计、不写代码    │
  │  debugging         保证"修对了"     不做设计、不写新功能  │
  │  finishing         保证"收尾了"     不做设计、不写代码    │
  │                                                          │
  │  职责边界清晰：                                         │
  │  brainstorming = What（做什么）                          │
  │  writing-plans = How（怎么做）                           │
  │  subagent/executing = Do（做了）                         │
  │  TDD = Quality（做对了）                                │
  │  verification = Done（做完了）                           │
  │  debugging = Fix（修对了）                              │
  │  finishing = Ship（收尾了）                             │
  └──────────────────────────────────────────────────────────┘
```

## 单一职责的三重价值

### 价值 1: 无冲突

```
  ┌──────────────────────────────────────────────────────────┐
  │  如果 brainstorming 既做设计又做实现：                   │
  │  → 设计过程中 AI 可能开始写代码                        │
  │  → 设计和实现混在一起                                 │
  │  → 设计被实现约束（不是从需求出发）                   │
  │  → 违反了 TDD 的核心原则                              │
  │                                                          │
  │  单一职责 = 无冲突：                                    │
  │  brainstorming 只做设计 → 不可能写代码                 │
  │  TDD 只保证质量 → 不可能做设计                        │
  │  subagent 只执行 → 不可能做设计                        │
  │  → 每个技能在自己的领域内是权威                       │
  │  → 不存在"两个技能要求做不同的事"                    │
  └──────────────────────────────────────────────────────────┘
```

### 价值 2: 可组合

```
  ┌──────────────────────────────────────────────────────────┐
  │  单一职责 = 拼图块                                     │
  │  → 可以按需组合成不同流程                              │
  │                                                          │
  │  流程 A (推荐)：                                        │
  │  brainstorming + writing-plans + subagent-driven          │
  │  + finishing = 完整功能开发                              │
  │                                                          │
  │  流程 B (备选)：                                        │
  │  brainstorming + writing-plans + executing-plans           │
  │  + finishing = 完整功能开发（同会话执行）                │
  │                                                          │
  │  流程 C (Bug 修复)：                                    │
  │  debugging + TDD + verification = 修复并验证             │
  │                                                          │
  │  流程 D (探索)：                                        │
  │  brainstorming alone = 只做设计，不实现                  │
  │                                                          │
  │  → 同样的技能块可以拼出不同的流程                      │
  │  → 不需要为每种流程设计单独的技能                      │
  │  → 类似于 Unix 管道：sort | uniq | wc                   │
  │    每个命令只做一件事，组合后实现复杂功能               │
  └──────────────────────────────────────────────────────────┘
```

### 价值 3: 可替换

```
  ┌──────────────────────────────────────────────────────────┐
  │  单一职责 = 可替换                                      │
  │  → 如果某个技能不适合，可以替换为等效技能              │
  │  → 其他技能不受影响                                    │
  │                                                          │
  │  替换示例 1:                                            │
  │  subagent-driven ↔ executing-plans                       │
  │  → 两者职责相同（执行计划）                            │
  │  → 但执行方式不同（子智能体 vs 同会话）                │
  │  → brainstorming 和 writing-plans 不受替换影响          │
  │                                                          │
  │  替换示例 2 (假设未来)：                                │
  │  如果出现了更好的 TDD 技能                              │
  │  → 替换 TDD 技能文件                                   │
  │  → 其他技能（brainstorming/writing-plans/subagent）      │
  │    引用 TDD 的地方自动生效                             │
  │  → 不需要修改其他技能                                  │
  │                                                          │
  │  Unix 管道类比：                                        │
  │  sort | uniq | wc                                       │
  │  → 把 sort 换成 sort -r 不影响 uniq 和 wc              │
  │  → 模块化 = 可替换性 = 可维护性                       │
  └──────────────────────────────────────────────────────────┘
```

## 职责边界如何被执行

```
  ┌──────────────────────────────────────────────────────────┐
  │  职责边界不只是概念——在代码中有具体执行机制：          │
  │                                                          │
  │  1. HARD-GATE 封锁跨界                                   │
  │     brainstorming: "Do NOT write any code"               │
  │     → 在设计阶段，AI 被禁止写代码                      │
  │     → 职责边界被门禁强制执行                           │
  │                                                          │
  │  2. 唯一出口 防止越权                                   │
  │     brainstorming: "Do NOT invoke any other skill"        │
  │     → 在设计阶段，AI 不能调用实现技能                  │
  │     → 职责边界被出口控制强制执行                       │
  │                                                          │
  │  3. Minimal code 防止过度实现                            │
  │     TDD GREEN: "Don't add features, refactor other        │
  │      code, or 'improve' beyond the test"                  │
  │     → 在实现阶段，AI 不能做设计决策                    │
  │     → 职责边界被最小代码原则强制执行                   │
  │                                                          │
  │  4. REFACTOR 不加功能                                    │
  │     TDD REFACTOR: "Don't add behavior"                   │
  │     → 在重构阶段，AI 不能加新功能                      │
  │     → 职责边界被行为不变原则强制执行                   │
  │                                                          │
  │  四种执行机制覆盖了所有跨界可能：                       │
  │  - 设计阶段跨界到实现 → HARD-GATE 拦截                 │
  │  - 设计阶段越权调用实现技能 → 唯一出口拦截             │
  │  - 实现阶段做设计决策 → Minimal code 拦截               │
  │  - 重构阶段加新功能 → 行为不变原则拦截                 │
  └──────────────────────────────────────────────────────────┘
```
