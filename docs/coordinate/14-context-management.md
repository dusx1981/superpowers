# 14 上下文管理逻辑：状态与策略

Controller 最重要的工作不是"调用技能"，而是"管理上下文"。上下文决定了 AI 的行为——同一个模型，在不同上下文中，输出完全不同。

## 上下文的四种状态

```
  ┌──────────────────────────────────────────────────────────┐
  │  空白 → 初始 → 累积 ↔ 隔离                               │
  │                                                          │
  │  ┌─ 空白 ───────────────────────────────────────────┐   │
  │  │  会话启动前，没有上下文                            │   │
  │  │  → 插件注入 using-superpowers → 进入初始状态      │   │
  │  └──────────────────────────────────────────────────┘   │
  │         ↓                                                │
  │  ┌─ 初始 ───────────────────────────────────────────┐   │
  │  │  using-superpowers 已注入                          │   │
  │  │  → Controller 身份已建立、技能匹配规则已生效       │   │
  │  └──────────────────────────────────────────────────┘   │
  │         ↓                                                │
  │  ┌─ 累积 ───────────────────────────────────────────┐   │
  │  │  技能逐个加载，上下文持续增长                      │   │
  │  │  → 旧技能仍在上下文中但不再是焦点                  │   │
  │  │  → 风险：注意力稀释、指令冲突、窗口溢出            │   │
  │  └──────────────────────────────────────────────────┘   │
  │         ↓ 派发子智能体                                   │
  │  ┌─ 隔离 ───────────────────────────────────────────┐   │
  │  │  子智能体获得全新的精简上下文                      │   │
  │  │  → 完成后上下文被销毁                              │   │
  │  └──────────────────────────────────────────────────┘   │
  │                                                          │
  │  "累积"是主链的常态，"隔离"是子智能体的常态              │
  └──────────────────────────────────────────────────────────┘
```

## 累积状态的上下文膨胀

```
  ┌──────────────────────────────────────────────────────────┐
  │  上下文增长轨迹：                                        │
  │  T0: [using-superpowers] ≈ 1K tokens                     │
  │  T1: + [brainstorming 交互] ≈ 4K tokens                  │
  │  T2: + [writing-plans] ≈ 6K tokens                       │
  │  T3: + [subagent-driven] ≈ 8K tokens                     │
  │  T4: + [N 个子智能体报告] ≈ 8K + 5K*N tokens             │
  │                                                          │
  │  后果 1: 注意力稀释                                      │
  │  → using-superpowers 的 1% 原则可能被"遗忘"              │
  │  → AI 注意力集中在最近消息                               │
  │                                                          │
  │  后果 2: 指令冲突                                        │
  │  → brainstorming 说"不要写代码"                          │
  │  → subagent-driven 说"派发实现者子智能体"                │
  │  → 两者同时存在于上下文中                                │
  │                                                          │
  │  后果 3: 上下文窗口溢出                                  │
  │  → 早期消息可能被截断                                    │
  │  → 截断可能丢失关键指令                                  │
  └──────────────────────────────────────────────────────────┘
```

## Controller 的五种上下文管理策略

```
  ┌──────────────────────────────────────────────────────────┐
  │  策略 1: "Announce at start" 注意力重置                   │
  │  ──────────────────────────────────────────────          │
  │  每个技能要求 AI 在开始时宣布：                          │
  │  "I'm using the writing-plans skill to create            │
  │   the implementation plan."                              │
  │  → AI 说出当前技能 = 主动确认进入新阶段                  │
  │  → 宣布行为重新锚定注意力焦点                            │
  │  → 类比：新官上任三把火——新指令权重被提升               │
  │                                                          │
  │  策略 2: 唯一出口的"自动清理"效果                        │
  │  ──────────────────────────────────────────────          │
  │  brainstorming 的唯一出口说"Invoke writing-plans"         │
  │  → AI 注意力从"设计"切换到"计划"                        │
  │  → 旧规则仍在上下文中但不再执行                          │
  │  → 类比：到达目的地后 GPS 导航自动停止                   │
  │                                                          │
  │  策略 3: 子智能体隔离的"垃圾回收"                        │
  │  ──────────────────────────────────────────────          │
  │  子智能体完成后上下文被销毁：                             │
  │  → 实现者 ~3K tokens → 销毁                              │
  │  → 规范审阅者 ~2K tokens → 销毁                          │
  │  → 代码审阅者 ~2K tokens → 销毁                          │
  │  → 只有结果摘要回主会话（~200 tokens）                   │
  │  → 净节省：~7K - 200 = 6.8K tokens per task              │
  │  → 这是 subagent-driven 优于 inline 的核心原因           │
  │                                                          │
  │  策略 4: TodoWrite 的外部记忆                             │
  │  ──────────────────────────────────────────────          │
  │  任务列表存储在 TodoWrite 工具状态中：                    │
  │  → 不在对话历史中永久占用空间                            │
  │  → AI 需要时通过工具读取                                │
  │  → 类比：笔记放在桌上，不背在脑子里                     │
  │                                                          │
  │  策略 5: 文件系统的持久化存储                             │
  │  ──────────────────────────────────────────────          │
  │  设计文档 → 写入文件系统                                 │
  │  计划文档 → 写入文件系统                                 │
  │  代码提交 → 写入 git                                     │
  │  → 文件系统 = 无限容量的外部记忆                        │
  │  → 需要时读取，不占用上下文窗口                          │
  │                                                          │
  │  五种策略的协同：                                        │
  │  策略 1 → 解决注意力问题（重置焦点）                     │
  │  策略 2 → 解决指令冲突（出口指向消歧）                   │
  │  策略 3 → 解决上下文膨胀（垃圾回收）                     │
  │  策略 4 → 解决状态追踪（外部记忆）                       │
  │  策略 5 → 解决持久化需求（文件系统）                     │
  └──────────────────────────────────────────────────────────┘
```

## 跨会话边界

```
  ┌──────────────────────────────────────────────────────────┐
  │  Controller 的上下文管理有一个硬性边界：会话。           │
  │                                                          │
  │  会话内：上下文累积，Controller 持续工作                 │
  │  会话间：上下文清零，Controller 从零开始                 │
  │                                                          │
  │  跨会话的延续机制：                                      │
  │  ┌──────────────────────────────────────────────────┐    │
  │  │  持久化层        内容                恢复方式     │    │
  │  │  ─────────       ────                ────────     │    │
  │  │  文件系统        设计文档            读取文件     │    │
  │  │  文件系统        实现计划            读取文件     │    │
  │  │  git             代码提交           git log      │    │
  │  │  git worktree    工作目录            路径指向     │    │
  │  │  CLAUDE.md       用户偏好           自动读取     │    │
  │  │                                                     │    │
  │  │  不跨越会话的：                                    │    │
  │  │  ✗ 对话历史（技能加载顺序、用户交互）              │    │
  │  │  ✗ 子智能体报告                                    │    │
  │  │  ✗ 审阅反馈                                        │    │
  │  │  ✗ Controller 的当前状态（在哪个技能阶段）         │    │
  │  └──────────────────────────────────────────────────┘    │
  │                                                          │
  │  跨会话恢复的挑战：                                      │
  │  → 新会话的 Controller 不知道上次停在哪里               │
  │  → 需要从文件系统和 git 历史推断当前阶段               │
  │  → 如果设计文档存在但计划不存在 → 写计划阶段            │
  │  → 如果计划存在且任务部分完成 → 继续执行阶段            │
  │  → 这是一种"隐式状态恢复"——没有显式的会话状态存储       │
  │  → 依赖 AI 的推理能力来推断上下文                      │
  │  → 这是当前设计最脆弱的环节                             │
  └──────────────────────────────────────────────────────────┘
```
