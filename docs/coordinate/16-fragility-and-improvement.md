# 16 Controller 逻辑的脆弱性与改进方向

任何设计都有边界条件。Controller 逻辑的精妙之处在于它用纯协议实现了复杂控制，但其脆弱性同样源于"纯协议"——没有代码保证，只有提示词约束。

## 六个已知的脆弱点

```
  ┌──────────────────────────────────────────────────────────┐
  │  脆弱点 1: 技能匹配的 1% 原则是 AI 自判                 │
  │  ──────────────────────────────────────────────          │
  │  "Might any skill apply?" → AI 自己判断                  │
  │  → AI 可能判断"不需要技能" → 跳过技能检查               │
  │  → 红旗表是反制措施，但仍是提示词层面的约束             │
  │  → 没有代码级别的强制执行                                │
  │                                                          │
  │  真实场景：                                              │
  │  用户说："帮我看看这个文件"                              │
  │  → AI 可能认为这是"简单查看"不需要技能                  │
  │  → 但如果文件包含 bug → 应该触发 debugging               │
  │  → 如果文件是新功能 → 应该触发 brainstorming             │
  │  → 1% 原则要求 AI 即使只有 1% 可能也要检查               │
  │  → 但"1%"本身是模糊的——AI 的 1% 和人类的 1% 不同      │
  │                                                          │
  │  脆弱点 2: 上下文膨胀导致早期指令遗忘                    │
  │  ──────────────────────────────────────────────          │
  │  主链的上下文是累积式的                                  │
  │  → 10+ 轮对话后，using-superpowers 的规则可能远离焦点    │
  │  → AI 可能不再遵循 1% 原则                              │
  │  → "Announce at start" 部分缓解但无法根治                │
  │  → 没有机制定期"提醒"AI 遵守全局规则                   │
  │                                                          │
  │  脆弱点 3: 跨会话状态丢失                                │
  │  ──────────────────────────────────────────────          │
  │  会话结束后上下文清零                                    │
  │  → 新会话不知道上次停在哪里                             │
  │  → 依赖 AI 从文件系统和 git 历史推断当前阶段            │
  │  → 推断可能出错——特别是在复杂项目中                    │
  │  → 没有"会话恢复"的协议                                │
  │                                                          │
  │  脆弱点 4: 子智能体报告的信任问题                        │
  │  ──────────────────────────────────────────────          │
  │  子智能体返回 DONE 但实际实现可能有问题                  │
  │  → 规范审阅者缓解了这个问题（"Do Not Trust"）           │
  │  → 但如果规范审阅者也不仔细呢？                          │
  │  → 如果代码审阅者只做表面检查呢？                        │
  │  → 信任链的每一环都有断裂的可能                         │
  │  → verification-before-completion 是最后一道防线         │
  │  → 但它依赖 AI 主动触发——又是提示词约束                 │
  │                                                          │
  │  脆弱点 5: 并发安全                                      │
  │  ──────────────────────────────────────────────          │
  │  SKILL.md 明确禁止并行派发实现者子智能体                  │
  │  → "Dispatch multiple implementation subagents            │
  │     in parallel (conflicts)" 是 Red Flag                  │
  │  → 但如果 AI 无视这个约束呢？                            │
  │  → 两个子智能体同时修改同一文件 → 冲突                   │
  │  → 代码层面的互斥锁可以防止，但提示词不能               │
  │                                                          │
  │  脆弱点 6: 技能描述的匹配精度                            │
  │  ──────────────────────────────────────────────          │
  │  技能匹配依赖 description 字段的文本匹配                  │
  │  → 新技能的 description 可能与现有技能重叠               │
  │  → AI 可能匹配到错误的技能                              │
  │  → 例如 "implementing" 可能匹配 TDD 也可能匹配           │
  │    subagent-driven-development                            │
  │  → Skill Priority 规则部分缓解但无法覆盖所有情况         │
  └──────────────────────────────────────────────────────────┘
```

## 脆弱性的结构性根源

```
  ┌──────────────────────────────────────────────────────────┐
  │  所有六个脆弱点有一个共同的结构性根源：                  │
  │                                                          │
  │  提示词约束 vs. 代码保证                                  │
  │                                                          │
  │  ┌─ 提示词约束 ─────────────────────────────────────┐   │
  │  │  "Do NOT invoke any other skill"                   │   │
  │  │  → AI 可能遵守，也可能不遵守                       │   │
  │  │  → 遵守率 ≈ 95%（经验值，非保证）                 │   │
  │  │  → 每次对话都有小概率违约                          │   │
  │  └──────────────────────────────────────────────────┘   │
  │                                                          │
  │  ┌─ 代码保证 ───────────────────────────────────────┐   │
  │  │  if (skill !== "writing-plans") throw Error()      │   │
  │  │  → 100% 保证                                       │   │
  │  │  → 不可能违约                                      │   │
  │  └──────────────────────────────────────────────────┘   │
  │                                                          │
  │  Superpowers 选择提示词约束而非代码保证，                 │
  │  因为：                                                  │
  │  1. 代码保证需要平台集成（每个平台不同）                 │
  │  2. 代码保证需要维护（技能变化时代码也要变）            │
  │  3. 提示词约束更灵活（可以适应未预见的场景）            │
  │  4. 5% 的违约率可以通过冗余设计降低到可接受水平         │
  │                                                          │
  │  这是深思熟虑的权衡，不是疏忽。                          │
  │  → 95% 遵守率 × 多重冗余 ≈ 99.9% 有效约束              │
  │  → 代价：偶尔需要人工纠正                               │
  │  → 收益：跨平台兼容、低维护成本、高灵活性               │
  └──────────────────────────────────────────────────────────┘
```

## 冗余设计如何弥补脆弱性

```
  ┌──────────────────────────────────────────────────────────┐
  │  以 brainstorming 的 HARD-GATE 为例：                    │
  │                                                          │
  │  单一约束：                                              │
  │  "Do NOT write any code"                                 │
  │  → 遵守率 ≈ 95%                                         │
  │  → 5% 的对话中 AI 可能写代码                            │
  │                                                          │
  │  冗余约束 1: Anti-Pattern                                │
  │  "This Is Too Simple To Need A Design"                   │
  │  → 拦截"太简单不需要设计"的理性化                       │
  │  → 覆盖了 HARD-GATE 的最大绕过路径                      │
  │  → 联合遵守率 ≈ 95% + 4% = 99%                          │
  │                                                          │
  │  冗余约束 2: 唯一出口                                    │
  │  "The ONLY skill you invoke after brainstorming           │
  │   is writing-plans."                                     │
  │  → 拦截"跳过设计直接实现"的路径                         │
  │  → 即使 HARD-GATE 被绕过，AI 也找不到实现技能的入口    │
  │  → 联合遵守率 ≈ 99% + 0.9% = 99.9%                      │
  │                                                          │
  │  冗余约束 3: Checklist 第 9 条                           │
  │  "Transition to implementation — invoke writing-plans"   │
  │  → AI 必须完成整个清单才能离开 brainstorming             │
  │  → 清单的最后一步就是调用 writing-plans                  │
  │  → 即使其他约束全部被绕过，清单仍导向正确出口          │
  │  → 联合遵守率 ≈ 99.9% + 0.09% = 99.99%                  │
  │                                                          │
  │  四重冗余的数学模型：                                    │
  │  P(违约) = P(违反 HARD-GATE)                             │
  │           × P(违反 Anti-Pattern | 违反 GATE)             │
  │           × P(违反出口 | 违反前两者)                     │
  │           × P(违反清单 | 违反前三者)                     │
  │  = 0.05 × 0.20 × 0.10 × 0.10                            │
  │  = 0.0001 = 0.01%                                        │
  │                                                          │
  │  → 单一约束 5% 的违约率                                 │
  │  → 四重冗余降到 0.01%                                   │
  │  → 这是 Superpowers "冗余 > DRY" 原则的数学基础         │
  └──────────────────────────────────────────────────────────┘
```

## 改进方向：代码保证的可能性

```
  ┌──────────────────────────────────────────────────────────┐
  │  如果要引入代码保证来弥补最关键的脆弱点：                │
  │                                                          │
  │  方向 1: 技能匹配的硬门禁                                │
  │  → 在插件层拦截"创建性"用户消息                         │
  │  → 自动触发 brainstorming，不给 AI 跳过的机会            │
  │  → 实现位置：superpowers.js 的 messages.transform        │
  │  → 代价：需要准确识别"创建性"消息（不简单）             │
  │                                                          │
  │  方向 2: 并发互斥锁                                      │
  │  → 在 Task 工具的包装层添加互斥检查                      │
  │  → 同一时刻只允许一个实现者子智能体运行                  │
  │  → 实现位置：插件的工具拦截层                            │
  │  → 代价：需要平台支持工具拦截                            │
  │                                                          │
  │  方向 3: 会话状态持久化                                   │
  │  → 在 .opencode/ 目录中保存当前技能阶段                  │
  │  → 新会话启动时读取状态并自动恢复                        │
  │  → 实现位置：superpowers.js 的 config + transform        │
  │  → 代价：需要定义状态格式和恢复协议                      │
  │                                                          │
  │  方向 4: 验证步骤的强制执行                              │
  │  → 在 finishing 技能中，通过代码验证测试是否真的通过     │
  │  → 不依赖 AI 主动运行验证命令                            │
  │  → 实现位置：finishing 的 Bash 命令包装                  │
  │  → 代价：需要知道项目的测试命令                          │
  │                                                          │
  │  关键洞察：                                              │
  │  → 代码保证不是全有或全无                                │
  │  → 可以在最关键的脆弱点引入代码保证                     │
  │  → 其他点仍然依赖提示词约束 + 冗余设计                  │
  │  → 混合策略：代码保证关键路径 + 提示词约束灵活路径      │
  │  → 这可能是 Superpowers 演进的下一阶段                  │
  └──────────────────────────────────────────────────────────┘
```
