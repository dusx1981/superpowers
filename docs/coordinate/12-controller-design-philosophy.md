# 12 Controller 逻辑的设计思想

Controller 不是一个对象、一个类、一个函数。它是一种**逻辑角色**——由 using-superpowers 技能注入后，AI 在对话中扮演的协调者身份。它的设计思想可以浓缩为一个悖论：**一个没有代码的控制层，如何可靠地控制数十个技能的协作？**

## 核心设计思想：协议即架构

```
  ┌──────────────────────────────────────────────────────────┐
  │  传统控制器的实现：                                      │
  │  ┌────────────────────────────────────────────────────┐  │
  │  │ class Controller {                                 │  │
  │  │   skills: Map<string, Skill>                       │  │
  │  │   dispatch(skillName: string, ctx: Context)         │  │
  │  │   route(message: Message): Skill                   │  │
  │  │ }                                                  │  │
  │  └────────────────────────────────────────────────────┘  │
  │  → 代码定义了控制逻辑                                    │
  │  → 代码保证了执行路径                                    │
  │  → 代码就是架构                                          │
  │                                                          │
  │  Superpowers 的控制器：                                   │
  │  ┌────────────────────────────────────────────────────┐  │
  │  │ （没有代码）                                        │  │
  │  │ using-superpowers SKILL.md = Controller 的"宪法"    │  │
  │  │ 每个技能的 SKILL.md = Controller 的"法律"           │  │
  │  │ 插件 superpowers.js = Controller 的"出生证明"       │  │
  │  │                                                     │  │
  │  │ 协议 = 架构                                        │  │
  │  │ 技能文件就是代码                                    │  │
  │  │ AI 的行为就是执行                                    │  │
  │  └────────────────────────────────────────────────────┘  │
  │                                                          │
  │  这意味着：                                              │
  │  → 修改 SKILL.md = 修改 Controller 的行为               │
  │  → 增加 SKILL.md = 增加 Controller 的能力               │
  │  → 不需要编译、不需要部署、不需要重启                    │
  │  → 协议的修改即架构的修改                                │
  └──────────────────────────────────────────────────────────┘
```

## 五个设计原则

```
  ┌──────────────────────────────────────────────────────────┐
  │  原则 1: 声明式 > 命令式                                 │
  │  → 技能声明"我要什么"，不命令"怎么做"                   │
  │  → brainstorming 声明"HARD-GATE: 不要写代码"             │
  │  → 不写"if user asks for code, refuse"                  │
  │  → AI 自己决定如何遵守声明                               │
  │                                                          │
  │  原则 2: 约束 > 指导                                     │
  │  → 先告诉 AI 不能做什么，再告诉能做什么                  │
  │  → HARD-GATE / Iron Law / Red Flags 都是约束             │
  │  → 约束是"墙"，指导是"路"                              │
  │  → 墙比路更可靠——AI 不会穿墙，但可能走错路             │
  │                                                          │
  │  原则 3: 协议 > 实现                                     │
  │  → 技能之间通过协议通信，不通过代码调用                  │
  │  → 协议 = 唯一出口声明 + REQUIRED SUB-SKILL 声明         │
  │  → 不需要 import、不需要依赖注入                        │
  │  → 协议的修改不影响其他技能的实现                        │
  │                                                          │
  │  原则 4: 隔离 > 共享                                     │
  │  → 子智能体上下文完全隔离                                │
  │  → 主链技能上下文部分共享（妥协）                        │
  │  → 隔离 = 可预测性                                      │
  │  → 共享 = 灵活性但不可预测                               │
  │                                                          │
  │  原则 5: 冗余 > DRY                                      │
  │  → 关键约束在多处重复出现                                │
  │  → HARD-GATE 在 SKILL.md 开头声明                       │
  │  → Anti-Pattern 在中间重申                               │
  │  → Process Flow 在结尾强化                               │
  │  → 不是冗余的浪费——是冗余的保险                         │
  │  → AI 可能在不同位置读到同一约束，任一处生效即可          │
  └──────────────────────────────────────────────────────────┘
```
