# 05 Controller 的协调角色

Controller 是技能系统的管道连接器——它连接技能但不是中央调度器。它不决定"做什么"，只决定"下一步是什么"。

## Controller 的定位

```
  ┌──────────────────────────────────────────────────────────┐
  │  误解：Controller 是中央调度器                           │
  │  → 所有技能向 Controller 注册                           │
  │  → Controller 决定调用哪个技能                          │
  │  → Controller 拥有全局视图                              │
  │  → 修改一个技能需要通知 Controller                     │
  │                                                          │
  │  事实：Controller 是管道连接器                           │
  │  → 每个技能声明自己的出口                               │
  │  → Controller 只是把出口连到入口                        │
  │  → Controller 没有全局视图——它只看当前消息              │
  │  → 修改一个技能不需要通知 Controller                    │
  │                                                          │
  │  类比：                                                  │
  │  中央调度器 = 交通警察 → 指挥每辆车往哪走              │
  │  管道连接器 = 管道工 → 把管道连好，水自己流            │
  │                                                          │
  │  Superpowers 的 Controller 是管道工：                    │
  │  → 它把 brainstorming 的出口连到 writing-plans 的入口    │
  │  → 它把 writing-plans 的出口连到 subagent 的入口        │
  │  → 水自己流——AI 不需要被指挥                          │
  │  → Controller 只需要确保管道没堵住                      │
  └──────────────────────────────────────────────────────────┘
```

## Controller 做的三件事

```
  ┌──────────────────────────────────────────────────────────┐
  │  1. 技能匹配                                            │
  │     "Might any skill apply?"                             │
  │     → 在每个用户消息后检查                              │
  │     → 1% 可能性就触发加载                               │
  │     → 不是"决定调用"，而是"发现可能适用"              │
  │                                                          │
  │  2. 技能加载                                            │
  │     Invoke Skill tool → 注入技能内容到对话上下文          │
  │     → 加载后技能自己告诉 AI 该做什么                    │
  │     → Controller 不解释技能——技能自己说话               │
  │     → 类似于加载 DLL：OS 加载但不解释代码              │
  │                                                          │
  │  3. 上下文传递                                          │
  │     在子智能体场景中，Controller 传递任务上下文           │
  │     → 子智能体描述 + 相关文件 + BASE_SHA                │
  │     → 传递"最小必要信息"而非"全部上下文"              │
  │     → 信息封装（参见 03-nested-calls.md）               │
  │                                                          │
  │  Controller 不做的事：                                   │
  │  ✗ 不决定调用顺序——唯一出口决定                       │
  │  ✗ 不解释技能内容——技能自己解释                        │
  │  ✗ 不做设计决策——brainstorming 做                      │
  │  ✗ 不做实现决策——subagent 做                           │
  │  ✗ 不验证结果——verification 做                         │
  └──────────────────────────────────────────────────────────┘
```

## Controller 在子智能体场景中的协调

```
  ┌──────────────────────────────────────────────────────────┐
  │  subagent-driven-development 中 Controller 的角色：      │
  │                                                          │
  │  ┌─────────┐                                            │
  │  │Controller│                                            │
  │  └────┬────┘                                            │
  │       │                                                  │
  │       ├──1. 分配任务给实现者子智能体                     │
  │       │   传入：任务描述 + 上下文                        │
  │       │   实现者内部使用 TDD                              │
  │       │   实现者报告结果给 Controller                    │
  │       │                                                  │
  │       ├──2. 分配审阅给规范审阅者子智能体                 │
  │       │   传入：规范文本 + 实现者报告                    │
  │       │   审阅者独立判断                                  │
  │       │   审阅者报告结果给 Controller                    │
  │       │                                                  │
  │       ├──3. 分配审阅给代码审阅者子智能体                 │
  │       │   传入：实现代码 + 审阅范围                      │
  │       │   审阅者独立判断                                  │
  │       │   审阅者报告结果给 Controller                    │
  │       │                                                  │
  │       └──4. 综合结果，决定下一步                         │
  │           → 所有审阅通过 → finishing                     │
  │           → 审阅有反馈 → 反馈回实现者                   │
  │           → 实现失败 → debugging                        │
  │                                                          │
  │  Controller = 项目经理（不做技术，只做协调）              │
  │  → 不写代码，不审阅代码，不调试代码                    │
  │  → 只分配任务、收集结果、做流程决策                     │
  │  → 技术决策由子智能体自主做出                           │
  └──────────────────────────────────────────────────────────┘
```

## Controller 的决策点

```
  ┌──────────────────────────────────────────────────────────┐
  │  Controller 在整个流程中只有少数决策点：                 │
  │                                                          │
  │  决策点 1: 首次技能选择                                  │
  │  "Might any skill apply?"                                │
  │  → using-superpowers 流程图                              │
  │  → 通常触发 brainstorming（创建性任务）                  │
  │  → 或 debugging（修理性任务）                           │
  │  → 1% 原则：宁可误触发，不可遗漏                       │
  │                                                          │
  │  决策点 2: 执行方式选择                                  │
  │  writing-plans 完成后                                    │
  │  → subagent-driven-development（推荐）                   │
  │  → executing-plans（备选）                               │
  │  → 两个选项在技能中已声明，Controller 只做选择           │
  │  → 推荐标记降低了决策难度                               │
  │                                                          │
  │  决策点 3: 审阅结果处理                                  │
  │  审阅者返回反馈                                          │
  │  → 反馈有效 → 触发 receiving-code-review                 │
  │  → 反馈无效 → 忽略，继续                                │
  │  → 判断标准：不信任原则 + 技术严谨性                   │
  │                                                          │
  │  决策点 4: 终态选择                                      │
  │  finishing-a-development-branch                           │
  │  → merge / PR / cleanup                                  │
  │  → 用户选择，Controller 不做主                           │
  │  → 提供选项，让用户决策                                 │
  │                                                          │
  │  关键洞察：                                              │
  │  Controller 的决策点很少——因为大部分决策                 │
  │  已经被"唯一出口"和"角色分工"预定义了                │
  │  → 裁量空间被技能系统压缩到最小                        │
  │  → Controller 只在"分叉点"做决策                      │
  │  → 分叉点越少，系统越可靠                              │
  └──────────────────────────────────────────────────────────┘
```

## Controller 的局限与风险

```
  ┌──────────────────────────────────────────────────────────┐
  │  局限 1: 依赖 AI 的技能匹配判断                         │
  │  "Might any skill apply?" 是 AI 自己判断                │
  │  → AI 可能判断"不需要技能" → 跳过技能                │
  │  → 对策：1% 原则 + 红旗表格                            │
  │  → 但仍然是软约束，不是硬约束                          │
  │                                                          │
  │  局限 2: 上下文窗口限制                                  │
  │  技能内容被注入到对话上下文                              │
  │  → 多个技能叠加可能超出窗口                             │
  │  → 对策：子智能体隔离（每个子智能体独立上下文）        │
  │  → 但主链上的技能共享上下文                             │
  │                                                          │
  │  局限 3: 无运行时状态                                    │
  │  Controller 没有"记住"之前调用过哪个技能                │
  │  → 每个消息都是独立的技能匹配                           │
  │  → 可能重复触发已完成的技能                             │
  │  → 对策：技能自己的 HARD-GATE 防止重复                 │
  │    （如 brainstorming 的 "already brainstormed?" 检查）  │
  │                                                          │
  │  风险缓解：                                              │
  │  每个局限都有对应的技能层面对策                         │
  │  → Controller 的局限不靠 Controller 自己解决           │
  │  → 靠技能的 HARD-GATE、唯一出口、子智能体隔离解决      │
  │  → 这正是"管道连接器"设计的好处：                     │
  │    管道工不需要保证水质——水厂负责                       │
  └──────────────────────────────────────────────────────────┘
```
