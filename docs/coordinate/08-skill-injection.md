# 08 模式一：技能注入——主链的上下文构造

主链上的技能（brainstorming → writing-plans → subagent-driven → finishing）共享同一个对话上下文。Controller 通过 Skill 工具加载技能内容，技能的 SKILL.md 被追加到当前对话中，AI 开始遵循新技能的指令。

## 注入机制

```
  ┌──────────────────────────────────────────────────────────┐
  │  注入前的对话上下文：                                    │
  │  ┌────────────────────────────────────────────────────┐  │
  │  │ System: [平台默认系统提示词]                       │  │
  │  │ User: [第一个用户消息 + using-superpowers 注入]    │  │
  │  │ Assistant: [brainstorming 的交互历史]               │  │
  │  │ User: [用户的设计批准]                              │  │
  │  └────────────────────────────────────────────────────┘  │
  │                                                          │
  │  Controller 调用 Skill("writing-plans")                   │
  │  ↓                                                       │
  │  注入后的对话上下文：                                    │
  │  ┌────────────────────────────────────────────────────┐  │
  │  │ System: [平台默认系统提示词]                       │  │
  │  │ User: [第一个用户消息 + using-superpowers 注入]    │  │
  │  │ Assistant: [brainstorming 的交互历史]               │  │
  │  │ User: [用户的设计批准]                              │  │
  │  │ + [writing-plans SKILL.md 全文]  ← 新注入         │  │
  │  └────────────────────────────────────────────────────┘  │
  │                                                          │
  │  关键：                                                  │
  │  - writing-plans 看到了 brainstorming 的全部历史          │
  │  - writing-plans 看到了 using-superpowers 的规则         │
  │  - writing-plans 看到了用户的设计批准                    │
  │  → 信息是累积的，不是替换的                             │
  └──────────────────────────────────────────────────────────┘
```

## 注入的七个维度

### 维度 1: 内容选择——SKILL.md 包含什么

```
  ┌──────────────────────────────────────────────────────────┐
  │  每个 SKILL.md 的内容结构：                              │
  │                                                          │
  │  ┌─ frontmatter ─────────────────────────────────────┐  │
  │  │ name: writing-plans                                │  │
  │  │ description: Use when you have a spec or           │  │
  │  │   requirements for a multi-step task,              │  │
  │  │   before touching code                             │  │
  │  └───────────────────────────────────────────────────┘  │
  │  → name + description = 技能的身份卡                    │
  │  → Controller 用 description 做 1% 匹配判断             │
  │                                                          │
  │  ┌─ body ────────────────────────────────────────────┐  │
  │  │ Overview / Process / Checklist / Anti-patterns     │  │
  │  │ / Red Flags / Integration / Example                │  │
  │  └───────────────────────────────────────────────────┘  │
  │  → body = 技能的完整行为规则                            │
  │  → AI 遵循 body 中的所有指令                            │
  │                                                          │
  │  内容选择的约束：                                        │
  │  - SKILL.md 不能太长（受上下文窗口限制）                │
  │  - 必须自包含（不能依赖其他文件来理解）                 │
  │  - 必须有明确的入口和出口                               │
  └──────────────────────────────────────────────────────────┘
```

### 维度 2: 内容排除——SKILL.md 不包含什么

```
  ┌──────────────────────────────────────────────────────────┐
  │  SKILL.md 故意排除的信息：                               │
  │                                                          │
  │  排除 1: 其他技能的实现细节                              │
  │  → writing-plans 不知道 subagent-driven 如何派发子智能体 │
  │  → 它只知道"调用 subagent-driven-development"           │
  │  → 不需要知道内部实现                                   │
  │                                                          │
  │  排除 2: 用户的上下文历史（跨会话）                      │
  │  → 技能只看到当前会话的对话历史                         │
  │  → 不包含之前会话的任何信息                             │
  │  → 会话是上下文的生命周期边界                           │
  │                                                          │
  │  排除 3: 技能的内部引用文件（除非必要）                  │
  │  → SKILL.md 引用了 implementer-prompt.md                │
  │  → 但 SKILL.md 本身不包含 prompt 的全文                 │
  │  → 引用文件在使用时才加载                               │
  │                                                          │
  │  排除 = 选择性无知                                      │
  │  → 技能知道的越少，行为越可预测                         │
  │  → 技能知道的越少，越不会越权                           │
  └──────────────────────────────────────────────────────────┘
```

### 维度 3: 内容排序——信息的呈现顺序

```
  ┌──────────────────────────────────────────────────────────┐
  │  SKILL.md 的信息排序遵循"先约束后自由"原则：            │
  │                                                          │
  │  brainstorming 的排序：                                  │
  │  1. HARD-GATE (最严格的约束——先读)                      │
  │     "Do NOT write any code, scaffold any project..."      │
  │  2. Anti-Pattern (次严格——认知疫苗)                      │
  │     "This Is Too Simple To Need A Design"                │
  │  3. Checklist (流程——逐步执行)                           │
  │     9 个步骤，从探索到转实现                             │
  │  4. Process Flow (可视化——理解流程)                      │
  │     dot 流程图                                           │
  │  5. Process Details (细节——按需参考)                     │
  │     理解想法、探索方法、呈现设计                         │
  │  6. Key Principles (原则——灵活遵循)                      │
  │     一次一问、多选优先、YAGNI...                         │
  │                                                          │
  │  排序的心理学效应：                                      │
  │  → AI 读到 HARD-GATE 时，还没看到任何可做的事          │
  │  → 约束在行动之前植入，形成"先否后是"的框架效应        │
  │  → 类似于法律条文：先定义禁止行为，再定义允许行为       │
  │                                                          │
  │  如果排序反了（先 Process 后 HARD-GATE）：               │
  │  → AI 可能先读到"写设计文档"就动手了                   │
  │  → 然后才看到"不要写代码"——但已经在写了              │
  │  → 顺序错误 = 约束失效                                  │
  └──────────────────────────────────────────────────────────┘
```

### 维度 4: 内容框架——角色和视角的设定

```
  ┌──────────────────────────────────────────────────────────┐
  │  每个 SKILL.md 都在第一句话设定了 AI 的角色：            │
  │                                                          │
  │  brainstorming:                                          │
  │  "Help turn ideas into fully formed designs               │
  │   through natural collaborative dialogue."                │
  │  → AI 是"协作设计师"——提问、探索、不做决定             │
  │                                                          │
  │  writing-plans:                                          │
  │  "Write comprehensive implementation plans                │
  │   assuming the engineer has zero context                  │
  │   for our codebase and questionable taste."               │
  │  → AI 是"计划编写者"——为无知的工程师写详细计划         │
  │  → "questionable taste" 是一个关键框架——                 │
  │    它告诉 AI：不要假设实现者会做好的设计决策             │
  │    → 每一步都要写明代码，不留裁量空间                   │
  │                                                          │
  │  subagent-driven-development:                             │
  │  "Execute plan by dispatching fresh subagent              │
  │   per task, with two-stage review after each."            │
  │  → AI 是"项目协调者"——派发、审阅、不写代码             │
  │                                                          │
  │  systematic-debugging:                                    │
  │  "ALWAYS find root cause before attempting fixes.          │
  │   Symptom fixes are failure."                             │
  │  → AI 是"调查员"——找根因，不做猜测                     │
  │                                                          │
  │  框架效应：                                              │
  │  → 同一个 AI，在"协作设计师"框架下会提问               │
  │  → 同一个 AI，在"调查员"框架下会拒绝猜测               │
  │  → 角色设定是上下文构造中最轻量但最有效的工具           │
  └──────────────────────────────────────────────────────────┘
```

### 维度 5: 内容隔离——共享 vs 隔离

```
  ┌──────────────────────────────────────────────────────────┐
  │  主链技能的上下文是共享的（累积式）：                    │
  │                                                          │
  │  时刻 T1: brainstorming 加载                             │
  │  上下文 = [using-superpowers] + [brainstorming]           │
  │  → AI 同时看到两个技能的规则                             │
  │  → brainstorming 的 HARD-GATE + using-superpowers 的     │
  │    流程图同时生效                                       │
  │                                                          │
  │  时刻 T2: writing-plans 加载                             │
  │  上下文 = [using-superpowers] + [brainstorming 历史式]    │
  │         + [writing-plans]                                 │
  │  → brainstorming 的内容仍在上下文中                      │
  │  → 但 AI 的注意力被 writing-plans 的新指令吸引           │
  │  → brainstorming 的 HARD-GATE 仍有效但不再是焦点         │
  │                                                          │
  │  时刻 T3: subagent-driven 加载                           │
  │  上下文 = [using-superpowers] + [brainstorming 历史]      │
  │         + [writing-plans 历史] + [subagent-driven]        │
  │  → 上下文持续增长                                       │
  │  → 新技能的指令优先级高于旧技能                         │
  │                                                          │
  │  共享上下文的风险：                                      │
  │  - 旧技能的指令可能干扰新技能                           │
  │  - 上下文过长导致 AI 忽略远处的指令                     │
  │  - 多个技能的约束可能冲突                               │
  │                                                          │
  │  风险缓解：                                              │
  │  - 唯一出口防止了技能冲突（同一时刻只有一个活跃技能）   │
  │  - 新技能的"Announce at start"仪式重置了 AI 的焦点     │
  │  - 技能的独立性设计（单一职责）减少了干扰               │
  └──────────────────────────────────────────────────────────┘
```

### 维度 6: 内容变换——信息在技能间如何转换

```
  ┌──────────────────────────────────────────────────────────┐
  │  brainstorming 的产出 → writing-plans 的输入              │
  │  ────────────────────────────────────────────            │
  │                                                          │
  │  brainstorming 产出：                                    │
  │  → 用户意图（通过问答提取）                              │
  │  → 设计方案（2-3 方案 + 推荐）                          │
  │  → 设计文档（docs/superpowers/specs/...-design.md）      │
  │                                                          │
  │  变换方式：                                              │
  │  → 设计文档写入文件系统                                  │
  │  → writing-plans 读取设计文档                            │
  │  → 文件系统 = 技能间的"共享内存"                        │
  │                                                          │
  │  writing-plans 的产出 → subagent-driven 的输入            │
  │  ────────────────────────────────────────────            │
  │                                                          │
  │  writing-plans 产出：                                    │
  │  → 实现计划（docs/superpowers/plans/...-.md）            │
  │  → 每个任务有完整步骤、代码、命令                        │
  │  → 计划头声明了 REQUIRED SUB-SKILL                       │
  │                                                          │
  │  变换方式：                                              │
  │  → 计划文档写入文件系统                                  │
  │  → subagent-driven 读取计划，提取所有任务                │
  │  → 每个任务的文本被复制到子智能体的 Prompt 中            │
  │  → 关键：子智能体不读文件——Controller 提取全文传入       │
  │                                                          │
  │  subagent-driven 的产出 → finishing 的输入                │
  │  ────────────────────────────────────────────            │
  │                                                          │
  │  subagent-driven 产出：                                  │
  │  → 所有任务完成（git commits）                           │
  │  → 两次审阅通过（spec + quality）                        │
  │  → 最终代码审阅通过                                     │
  │                                                          │
  │  变换方式：                                              │
  │  → git 提交历史 + 当前工作树状态                         │
  │  → finishing 读取 git 状态来决定下一步                   │
  │  → 不需要 subagent-driven 的内部报告                     │
  │                                                          │
  │  变换的三种介质：                                        │
  │  1. 文件系统 = 持久化共享内存                            │
  │  2. git 历史 = 不可变的审计日志                          │
  │  3. Prompt 文本 = 实时传递的上下文                       │
  └──────────────────────────────────────────────────────────┘
```

### 维度 7: 内容生命周期——上下文何时创建、使用、丢弃

```
  ┌──────────────────────────────────────────────────────────┐
  │  上下文的生命周期：                                      │
  │                                                          │
  │  ┌─ using-superpowers ──────────────────────────────┐    │
  │  │  创建：会话启动时（插件注入第一个用户消息）       │    │
  │  │  使用：整个会话期间（每条消息都检查技能匹配）     │    │
  │  │  丢弃：会话结束时                                 │    │
  │  │  生命周期 = 会话级别                              │    │
  │  └──────────────────────────────────────────────────┘    │
  │                                                          │
  │  ┌─ brainstorming ──────────────────────────────────┐    │
  │  │  创建：Skill("brainstorming") 调用时              │    │
  │  │  使用：设计阶段（可能多轮对话）                   │    │
  │  │  丢弃：设计文档提交后，AI 注意力转移到新技能     │    │
  │  │  生命周期 = 设计阶段                              │    │
  │  │  注意：内容仍在上下文中，但不再是焦点             │    │
  │  └──────────────────────────────────────────────────┘    │
  │                                                          │
  │  ┌─ 实现者子智能体 ────────────────────────────────┐    │
  │  │  创建：Task 工具派发时                             │    │
  │  │  使用：实现一个任务                                │    │
  │  │  丢弃：任务完成后子智能体上下文被销毁             │    │
  │  │  生命周期 = 单个任务                              │    │
  │  │  注意：真正的丢弃——上下文不回传给 Controller      │    │
  │  └──────────────────────────────────────────────────┘    │
  │                                                          │
  │  ┌─ spec-reviewer 子智能体 ─────────────────────────┐    │
  │  │  创建：Task 工具派发时                             │    │
  │  │  使用：审阅一个任务                                │    │
  │  │  丢弃：审阅完成后上下文被销毁                     │    │
  │  │  生命周期 = 单次审阅                              │    │
  │  └──────────────────────────────────────────────────┘    │
  │                                                          │
  │  生命周期递减：                                          │
  │  会话 > 设计阶段 > 单个任务 > 单次审阅                  │
  │  → 生命周期越短，上下文越干净                           │
  │  → 上下文越干净，行为越可预测                           │
  └──────────────────────────────────────────────────────────┘
```
