# 04 裁量空间平移：从代码铁律到电商风控铁律

Superpowers 的裁量空间控制有一个核心洞察：**同一意图不同表达 → AI 服从概率截然不同**。这个洞察在电商中同样成立——而且后果更严重。软件开发中裁量空间过大的代价是代码质量下降；电商中裁量空间过大的代价是资金损失和合规风险。

## 裁量空间的电商映射

```
  ┌──────────────────────────────────────────────────────────┐
  │  场景              开发裁量空间    电商裁量空间           │
  │  ────              ────────────    ────────────           │
  │  普通文本          ~50%            ~50%                   │
  │  粗体强调          ~40%            ~40%                   │
  │  清单格式          ~30%            ~30%                   │
  │  XML 标签          ~10%            ~15%（电商更复杂）     │
  │  铁律全大写        ~5%             ~10%（电商有灰色地带） │
  │                                                          │
  │  为什么电商的服从率更低？                                │
  │  → 电商场景比开发更"灵活"——每个订单都是独特的          │
  │  → AI 更容易认为"这次不一样"                          │
  │  → TDD 中"代码要么测试要么没测试"是二元的              │
  │  → 电商中"这个订单是否可疑"是连续的灰色地带            │
  │  → 连续谱比二元判断更容易产生裁量空间                   │
  └──────────────────────────────────────────────────────────┘
```

## 电商特有的裁量空间陷阱

```
  ┌──────────────────────────────────────────────────────────┐
  │  陷阱 1: "特殊情况"裁量                                 │
  │  ────────────────────────                                │
  │  TDD: "这个函数太简单不需要测试"                        │
  │  电商: "这个订单金额太小不需要风控"                      │
  │  → 两者逻辑相同——用"特殊情况"绕过规则                  │
  │  → 但电商中"小额订单"正是刷单测试的常用手法            │
  │  → "特殊情况"裁量在电商中比开发中更危险                │
  │                                                          │
  │  陷阱 2: "先做后查"裁量                                 │
  │  TDD: "先写代码再补测试"                                │
  │  电商: "先审批订单再做风控复查"                          │
  │  → 两者逻辑相同——改变执行顺序绕过规则                  │
  │  → 但电商中"先审批"= 资金已流出                        │
  │  → 事后风控 = 马后炮                                    │
  │                                                          │
  │  陷阱 3: "信任来源"裁量                                 │
  │  TDD: "这个代码已经很完善了"                             │
  │  电商: "这个供应商的货一直没问题"                        │
  │  → 两者逻辑相同——用历史信任代替当前验证                │
  │  → 但供应商质量可能突然下降                             │
  │  → "一直没问题" ≠ "这次没问题"                         │
  │                                                          │
  │  陷阱 4: "效率优先"裁量                                 │
  │  TDD: "TDD 太慢了，先发布再说"                          │
  │  电商: "风控太严了，先放行再说"                          │
  │  → 两者逻辑相同——用效率理由绕过质量/安全               │
  │  → 但电商中"先放行"= 可能放走欺诈                     │
  │  → 效率 > 安全 是电商最常见的裁量空间陷阱               │
  └──────────────────────────────────────────────────────────┘
```

## 电商裁量空间的压缩策略

```
  ┌──────────────────────────────────────────────────────────┐
  │  策略 1: 将连续判断二元化                                │
  │  ──────────────────────────────                          │
  │  电商中很多判断是连续的（风险等级 1-100）                │
  │  → AI 可以在灰色地带做裁量                              │
  │  → 压缩方法：设定硬阈值，将连续判断变成二元             │
  │  → "风险分 > 70 → 自动拒绝"（铁律，无裁量）             │
  │  → "风险分 30-70 → 需人工审核"（转移裁量给人）          │
  │  → "风险分 < 30 → 自动通过"（铁律，无裁量）             │
  │  → 二元化 = 将裁量空间从 [0,100] 压缩到 {0,1}           │
  │  → 灰色地带的裁量转移给人类                             │
  │                                                          │
  │  策略 2: 用"NO X WITHOUT Y"句式代替"建议"              │
  │  ────────────────────────────────────────────            │
  │  弱版本："建议对大额订单进行风控验证"                    │
  │  → AI 裁量：什么是"大额"？是否需要验证？               │
  │  → 裁量空间 ~40%                                        │
  │                                                          │
  │  强版本："NO ORDER APPROVAL WITHOUT RISK VERIFICATION"    │
  │  → AI 裁量：0%——没有验证就不能审批                     │
  │  → 裁量空间从 40% 压缩到 0%                             │
  │                                                          │
  │  策略 3: 用"不信任"预设压缩裁量空间                     │
  │  ──────────────────────────────────                      │
  │  没有"不信任"预设：                                    │
  │  → AI 看到供应商报告"质量合格"→ 倾向于信任             │
  │  → 裁量空间：AI 可以选择信任或不信任                    │
  │  → 裁量空间 ~50%                                        │
  │                                                          │
  │  有"不信任"预设：                                      │
  │  → "供应商报告可能不完整或不准确"                        │
  │  → AI 倾向于独立验证                                    │
  │  → 裁量空间：AI 只能选择验证                            │
  │  → 裁量空间从 50% 压缩到 ~10%                           │
  │                                                          │
  │  策略 4: 红旗表拦截内心独白                              │
  │  ──────────────────────────                              │
  │  AI 内心："这个订单看起来正常"                           │
  │  红旗表拦截："看起来正常的订单也可能是欺诈"              │
  │  → AI 的偏航想法被直接反驳                              │
  │  → 裁量空间从"偏向放行"被拉回"必须验证"               │
  │  → 红旗表 = 裁量空间的实时校准                          │
  └──────────────────────────────────────────────────────────┘
```

## 电商裁量空间的特殊考量：何时保留裁量

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 允许在某些场景保留裁量空间                  │
  │  （07-tactical-choice: 何时应保留裁量空间）               │
  │  → 电商中保留裁量空间的场景更多：                        │
  │                                                          │
  │  场景 1: VIP 客户服务                                    │
  │  → 铁律太严可能得罪大客户                               │
  │  → 保留裁量：AI 可以根据客户等级调整服务标准             │
  │  → 但风控铁律仍不可裁量（即使是 VIP 也可能被盗号）       │
  │  → 部分裁量：服务标准可调，风控标准不可调               │
  │                                                          │
  │  场景 2: 新品试错                                        │
  │  → 选品没有绝对对错——需要试探市场                       │
  │  → 保留裁量：AI 可以根据市场信号调整选品建议             │
  │  → 但合规铁律不可裁量（即使是试错也不能卖违禁品）       │
  │  → 部分裁量：选品策略可调，合规底线不可调               │
  │                                                          │
  │  场景 3: 大促期间的风控                                  │
  │  → 大促时风控过严 = 拒绝大量真实订单                    │
  │  → 需要临时放宽阈值——但不是取消风控                    │
  │  → 保留裁量：阈值可调，风控流程不可跳过                 │
  │  → 类比：TDD 中"throwaway prototypes"是 TDD 的例外       │
  │  → 电商中"大促临时阈值"是风控的例外                   │
  │  → 例外需要用户明确授权                                 │
  │                                                          │
  │  裁量空间预算模型：                                      │
  │  → 合规：0% 裁量（铁律）                                │
  │  → 风控：0-10% 裁量（阈值可调但流程不可跳）              │
  │  → 选品：20-40% 裁量（需要市场判断）                     │
  │  → 营销：30-50% 裁量（创意性决策）                       │
  │  → 客服：10-20% 裁量（有 SOP 但需要同理心）              │
  │  → 裁量空间预算 = 电商 AI 的"风险预算"                 │
  └──────────────────────────────────────────────────────────┘
```
