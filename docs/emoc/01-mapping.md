## A 类详解：通用方法论（可直接平移）

### 1. 唯一出口协议 → ★★★★★

```
  ┌──────────────────────────────────────────────────────────┐
  │  开发：brainstorming → writing-plans → subagent → finish  │
  │  电商：选品分析 → 供应链对接 → 商品上架 → 营销投放       │
  │         → 订单处理 → 售后服务                            │
  │                                                          │
  │  平移方案：每个运营技能声明唯一出口                       │
  │  → "选品完成后，唯一出口是供应链对接"                    │
  │  → 防止 AI 跳过环节（如跳过选品直接上架）               │
  │                                                          │
  │  电商特有挑战：流程有回环（售后→重新选品→重新上架）      │
  │  → 需要扩展唯一出口为"条件出口"                         │
  │  → 售后完成后 → if 退货率>30% → 选品分析                 │
  │                   else → 订单处理                        │
  └──────────────────────────────────────────────────────────┘
```

### 2. 角色分工 → ★★★★★

```
  ┌──────────────────────────────────────────────────────────┐
  │  电商团队天然分工适配：                                  │
  │  选品经理 → 决定卖什么    运营专员 → 决定怎么卖          │
  │  客服团队 → 处理售后      财务团队 → 处理结算            │
  │                                                          │
  │  平移方案：                                              │
  │  → 选品技能只做选品分析，不做营销决策                    │
  │  → 客服技能只做售后处理，不做退款审批（审批是财务技能）  │
  │  → 角色分工 = 权限边界 = 越权防护                       │
  │                                                          │
  │  粒度调整：小团队一人多角色→技能合并；大团队→技能拆分   │
  └──────────────────────────────────────────────────────────┘
```

### 3. 嵌套调用+偏见隔离 → ★★★★☆

```
  ┌──────────────────────────────────────────────────────────┐
  │  电商偏见场景：                                          │
  │  → 选品分析被营销预算影响 → "因为有钱就选贵品"           │
  │  → 客服被销售指标影响 → "为冲量拒绝退换"               │
  │  → 质检被供应链压力影响 → "赶工期放行次品"              │
  │                                                          │
  │  平移方案：                                              │
  │  → 选品技能不传入营销预算                                │
  │  → 质检技能不传入供应链时效                              │
  │  → 客服技能不传入销售 KPI                                │
  │                                                          │
  │  关键调整：传数据不传结论                                │
  │  → 传"退货率 15%"但不传"选品经理认为可以接受"          │
  │  → 传"库存 500 件"但不传"仓储建议尽快清仓"             │
  └──────────────────────────────────────────────────────────┘
```

### 4. 预设立场 → ★★★★★

```
  ┌──────────────────────────────────────────────────────────┐
  │  电商"不信任"场景（最高价值平移）：                     │
  │  → 质检：不信任供应商自检报告                            │
  │  → 风控：不信任用户自述信息                              │
  │  → 财务：不信任前端销售数据（需独立对账）                │
  │  → 合规：不信任平台自动审核结果                          │
  │                                                          │
  │  平移方案：                                              │
  │  质检技能："供应商的自检报告可能不完整。                  │
  │    你必须独立验证每一项指标。"                           │
  │  风控技能："用户身份信息可能伪造。                        │
  │    你必须交叉验证至少两个独立来源。"                     │
  │                                                          │
  │  电商特有价值：欺诈风险高（刷单/假货/退款欺诈）          │
  │  → "不信任"预设可显著降低 AI 辅助决策的欺诈风险         │
  └──────────────────────────────────────────────────────────┘
```

### 5. 信息最小化 → ★★★★★

```
  ┌──────────────────────────────────────────────────────────┐
  │  电商数据特点：PB 级数据量 + 强合规要求                  │
  │  → AI 客服不需全部用户历史（隐私+效率）                  │
  │  → 选品 AI 不需全部财务数据（只需品类预算）              │
  │  → 营销 AI 不需供应链成本（只需 SKU 和售价）             │
  │                                                          │
  │  双重价值：                                              │
  │  → token 节省 = 成本降低（数据量大）                    │
  │  → 数据最小化 = 合规对齐（GDPR/个保法已是法律要求）     │
  └──────────────────────────────────────────────────────────┘
```

### 6-7. 铁律+红旗表 → ★★★★☆/★★★★★

```
  ┌──────────────────────────────────────────────────────────┐
  │  电商铁律候选：                                          │
  │  风控：NO ORDER APPROVAL WITHOUT RISK VERIFICATION        │
  │  质检：NO PRODUCT LISTING WITHOUT QUALITY CHECK            │
  │  财务：NO PAYOUT WITHOUT RECONCILIATION                    │
  │  退款：NO REFUND WITHOUT EVIDENCE VERIFICATION             │
  │                                                          │
  │  电商红旗表候选（风控）：                                │
  │  | Thought | Reality |                                   │
  │  | "This order looks normal"                              │
  │  |   Normal-looking orders can be fraud. Verify. |        │
  │  | "The user has a good history"                          │
  │  |   Account takeover changes everything. Re-verify. |    │
  │  | "It's just a small amount"                             │
  │  |   Small amounts test fraud detection. Flag. |          │
  │  | "We'll catch it later"                                │
  │  |   Later = after money is gone. Block now. |            │
  │                                                          │
  │  适配挑战：电商铁律的后果比 TDD 更复杂                  │
  │  → TDD：写代码前没测试？删除重来。后果明确。             │
  │  → 电商：没风控验证就审批？拒绝订单。                   │
  │    → 但拒绝真实订单 = 损失收入                          │
  │  → 需要平衡：过严铁律 = 误杀，过松铁律 = 漏杀          │
  └──────────────────────────────────────────────────────────┘
```

## B 类详解：领域适配型

### 8-10. 宣布仪式/流程图/清单

```
  ┌──────────────────────────────────────────────────────────┐
  │  宣布仪式 → ★★★★☆                                      │
  │  → AI 先声明角色和目标："我正在使用风控技能验证此订单"   │
  │  → 声明角色 = 后续行为对齐角色                          │
  │  → 电商场景切换频繁，需要简化格式                        │
  │                                                          │
  │  流程图 → ★★★★☆                                        │
  │  → 电商运营路径可用 dot 图定义                           │
  │  → 选品：市场扫描→竞品分析→利润测算→选品决策            │
  │  → 退款：用户申请→证据验证→审批决策→执行退款            │
  │  → 电商流程图可能有更多回环（退款→退货→重新入库）        │
  │                                                          │
  │  清单 → ★★★★★                                          │
  │  → 电商天然依赖 SOP                                     │
  │  → 新品上架清单：标题/主图/详情/价格/库存/物流模板       │
  │  → 大促准备清单：备货/页面/推广/客服/仓储/系统压测       │
  │  → 几乎不需要适配——电商已有 SOP 传统                    │
  └──────────────────────────────────────────────────────────┘
```

### 11-12. 子智能体/Good-Bad 对比

```
  ┌──────────────────────────────────────────────────────────┐
  │  子智能体派发 → ★★★☆☆                                  │
  │  → 电商场景中"子智能体"= 专用 AI Agent                  │
  │  → 选品 Agent / 客服 Agent / 风控 Agent / 财务 Agent     │
  │  → 隔离原则可复用：每个 Agent 独立上下文                 │
  │  → 挑战：电商 Agent 通常访问数据库/API，                │
  │    不像开发 Agent 只读文件                               │
  │  → 垃圾回收概念可复用：Agent 完成后销毁上下文            │
  │                                                          │
  │  Good/Bad 对比 → ★★★★☆                                 │
  │  → 电商大量场景需要对比学习                              │
  │  Good：退款申请含完整订单号、问题描述、图片证据            │
  │  Bad：退款申请只有"东西不好"四个字                      │
  │  → AI 学会区分有效和无效的输入                           │
  │  → 电商客服、质检、风控都可使用                          │
  └──────────────────────────────────────────────────────────┘
```

## C 类详解：不可平移

```
  ┌──────────────────────────────────────────────────────────┐
  │  TDD / 代码审阅 / git worktree → ★☆☆☆☆                  │
  │                                                          │
  │  这些实践深度耦合于软件工程：                            │
  │  → TDD 的 "Red-Green-Refactor" 没有电商对应             │
  │  → git worktree 的隔离开发没有电商对应                   │
  │  → 代码审阅的对象是源代码，电商审阅的对象是业务决策      │
  │                                                          │
  │  但精神可以保留：                                        │
  │  → TDD 的 "先验证后执行" → 先验证业务规则再执行操作      │
  │  → 代码审阅的 "独立验证" → 业务决策的独立复核            │
  │  → git worktree 的 "隔离" → 业务实验的隔离环境           │
  │  → 精神可平移，具体实践不可平移                          │
  └──────────────────────────────────────────────────────────┘
```
