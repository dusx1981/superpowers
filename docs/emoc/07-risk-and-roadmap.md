# 07 风险分析与路线图

## 五大风险

```
  ┌──────────────────────────────────────────────────────────┐
  │  风险 1: 铁律过严导致业务损失                            │
  │  ────────────────────────────────                        │
  │  Superpowers 的铁律设计假设：违反的后果可逆              │
  │  → TDD: 写代码前没测试？删除代码重新开始                 │
  │  → 后果：浪费几小时，但没有资金损失                     │
  │                                                          │
  │  电商铁律的后果可能不可逆：                              │
  │  → 风控铁律太严 → 拒绝真实订单 → 损失收入+客户流失      │
  │  → 退款铁律太严 → 拒绝合理退款 → 投诉+差评+平台处罚     │
  │  → 质检铁律太严 → 延迟上架 → 错过市场窗口              │
  │                                                          │
  │  缓解措施：                                              │
  │  → 铁律只约束"流程不可跳过"，不约束"结果必须拒绝"       │
  │  → "NO ORDER APPROVAL WITHOUT RISK VERIFICATION"          │
  │    = 必须做风控验证（不可跳过）                           │
  │    ≠ 必须拒绝高风险订单（结果由验证决定）               │
  │  → 铁律控制流程，不控制结果                              │
  │  → 结果由人类或更高级的 AI 决策                          │
  │                                                          │
  │  风险 2: 电商灰色地带比开发多                            │
  │  ────────────────────────────────                        │
  │  开发决策大部分是二元的：                                │
  │  → 测试通过/不通过                                       │
  │  → 代码编译/不编译                                       │
  │  → 规范匹配/不匹配                                       │
  │                                                          │
  │  电商决策大量是灰色的：                                  │
  │  → "这个退款是否合理？"——可能部分合理                   │
  │  → "这个选品是否有潜力？"——可能有点潜力                 │
  │  → "这个投诉是否需要升级？"——可能需要也可能不需要       │
  │  → AI 在灰色地带有更大的裁量空间                        │
  │  → 裁量空间大 = 行为不可预测                             │
  │                                                          │
  │  缓解措施：                                              │
  │  → 二元化阈值（风控分>70=拒绝，<30=通过，30-70=人工）    │
  │  → 灰色地带转移给人类决策                               │
  │  → AI 只做清晰的决策，人类做模糊的决策                   │
  │                                                          │
  │  风险 3: 电商数据访问的复杂性                            │
  │  ──────────────────────────────                          │
  │  开发子智能体：本地文件系统，确定性访问                   │
  │  电商子智能体：数据库+API+第三方平台，不确定性访问        │
  │  → API 可能超时、限流、返回错误数据                     │
  │  → 数据库可能有延迟、主从不一致                         │
  │  → 第三方平台数据格式不统一                             │
  │  → 子智能体的上下文构造更复杂                           │
  │                                                          │
  │  缓解措施：                                              │
  │  → Controller 负责数据获取和清洗                         │
  │  → 子智能体只看到清洗后的数据                            │
  │  → 数据不确定性由 Controller 处理，不传递给子智能体       │
  │  → 信息最小化 + 数据清洗 = 子智能体看到干净数据          │
  │                                                          │
  │  风险 4: 电商法规的地域差异                              │
  │  ──────────────────────────────                          │
  │  Superpowers 的规则是通用的（不依赖地域）                 │
  │  → TDD 铁律在任何国家都一样                             │
  │  → 代码审阅标准在任何团队都类似                         │
  │                                                          │
  │  电商法规因地域差异巨大：                                │
  │  → 中国：电商法、消费者权益保护法、个保法                │
  │  → 美国：各州消费者保护法不同                           │
  │  → 欧盟：GDPR、消费者指令                               │
  │  → 同一退款请求在不同地区可能适用不同规则                │
  │  → 合规技能需要地域感知                                 │
  │                                                          │
  │  缓解措施：                                              │
  │  → 合规技能按地域拆分为子技能                           │
  │  → Controller 根据订单地域选择合规子技能                 │
  │  → 类似 Superpowers 的技能路由机制                      │
  │                                                          │
  │  风险 5: 电商操作的实时性要求                            │
  │  ──────────────────────────────                          │
  │  Superpowers 的子智能体派发有延迟（秒级）                 │
  │  → 开发中几秒延迟可接受                                 │
  │  → 电商客服中几秒延迟可能影响体验                       │
  │  → 电商秒杀/抢购场景中毫秒级响应是必需                  │
  │  → 子智能体架构可能无法满足实时性要求                   │
  │                                                          │
  │  缓解措施：                                              │
  │  → 区分实时技能和批处理技能                              │
  │  → 实时技能（客服响应）：预加载，不派发子智能体          │
  │  → 批处理技能（选品分析、财务对账）：可派发子智能体       │
  │  → 混合架构：关键路径实时，非关键路径批处理              │
  └──────────────────────────────────────────────────────────┘
```

## 路线图

```
  ┌──────────────────────────────────────────────────────────┐
  │  Phase 1: 方法论验证（2-4 周）                           │
  │  ────────────────────────────────                        │
  │  目标：验证 Superpowers 方法论在电商中的有效性            │
  │  范围：1-2 个最简单的技能                                │
  │  交付：                                                  │
  │  → 退款审核技能（铁律+红旗表+预设立场）                  │
  │  → 订单风控技能（铁律+Gate Function）                    │
  │  验证指标：                                              │
  │  → AI 服从铁律的概率（目标 > 90%）                      │
  │  → 红旗表拦截偏航的频率                                  │
  │  → 误杀率 vs. 漏杀率                                    │
  │                                                          │
  │  Phase 2: 流程构建（4-8 周）                             │
  │  ────────────────────────────────                        │
  │  目标：构建完整的技能流程链                               │
  │  范围：8 个核心技能                                      │
  │  交付：                                                  │
  │  → using-emoc 入口技能                                   │
  │  → 运营分析→选品→上架→营销→复盘 主链                    │
  │  → 订单风控+退款审核 横切触发                            │
  │  验证指标：                                              │
  │  → 技能遗漏率（目标 < 5%）                               │
  │  → 唯一出口遵循率                                       │
  │  → 端到端流程完成率                                     │
  │                                                          │
  │  Phase 3: 子智能体架构（8-12 周）                        │
  │  ────────────────────────────────                        │
  │  目标：实现子智能体协调                                   │
  │  范围：退款审核+选品决策的子智能体流程                    │
  │  交付：                                                  │
  │  → 执行者+合规审阅者+质量审阅者 协调流程                 │
  │  → 信息最小化的上下文构造                                │
  │  → 偏见隔离验证                                         │
  │  验证指标：                                              │
  │  → 偏见隔离效果（审阅者独立性测试）                      │
  │  → 上下文膨胀控制（token 消耗对比）                      │
  │  → 决策质量（人工评估）                                  │
  │                                                          │
  │  Phase 4: 平台适配（12-16 周）                           │
  │  ────────────────────────────────                        │
  │  目标：适配多电商平台的差异                               │
  │  范围：淘宝/京东/拼多多/抖音/SHEIN                       │
  │  交付：                                                  │
  │  → 平台特定技能变体                                     │
  │  → 地域合规子技能                                       │
  │  → 实时+批处理混合架构                                  │
  │  验证指标：                                              │
  │  → 跨平台一致性                                         │
  │  → 合规覆盖率                                           │
  │  → 响应时间（实时场景 < 2s，批处理 < 30s）              │
  └──────────────────────────────────────────────────────────┘
```

## 可行性总结

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的 15 项最佳实践在电商中的可行性：          │
  │                                                          │
  │  高可行性（★★★★★，7 项）：                              │
  │  唯一出口 / 角色分工 / 预设立场 / 信息最小化               │
  │  / 红旗表 / 清单 / 冗余                                  │
  │  → 这些是纯方法论，可直接平移                            │
  │  → 实施成本低，收益确定性高                              │
  │  → 建议在 Phase 1 优先实施                               │
  │                                                          │
  │  中可行性（★★★☆，6 项）：                               │
  │  嵌套调用+偏见隔离 / 铁律 / 宣布仪式                     │
  │  / 流程图 / Good/Bad / 子智能体派发                      │
  │  → 框架可复用，内容需重新设计                            │
  │  → 实施成本中等，需要电商领域知识                        │
  │  → 建议在 Phase 2-3 逐步实施                             │
  │                                                          │
  │  不可平移（★☆，1 项）：                                  │
  │  TDD / 代码审阅 / git worktree                            │
  │  → 深度耦合于软件工程                                    │
  │  → 但精神可保留：先验证后执行、独立复核、隔离实验        │
  │                                                          │
  │  最高价值平移：预设立场（不信任框架）                    │
  │  → 电商欺诈风险高，"不信任"预设可显著降低风险           │
  │  → 实施成本最低（纯语言框架）                            │
  │  → 收益最高（风控/质检/财务全场景适用）                  │
  │  → 建议作为第一个验证实验                                │
  └──────────────────────────────────────────────────────────┘
```
