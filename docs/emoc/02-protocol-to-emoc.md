# 02 协议设计平移：四层协议栈 → 电商工作流协议

Superpowers 的四层协议栈（会话/路由/切换/子智能体通信）是一个通用框架。电商需要重新定义每一层的内容，但架构可以复用。

## 四层协议栈的电商映射

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers         → 电商                               │
  │  ──────────            ────                               │
  │  Layer 4: 会话协议     Layer 4: 运营会话协议              │
  │  → 插件注入            → 电商 AI 入口注入                 │
  │  → using-superpowers   → using-emoc（电商技能调度规则）    │
  │                                                          │
  │  Layer 3: 技能路由     Layer 3: 业务技能路由              │
  │  → 1% 匹配原则         → 业务场景匹配原则                │
  │  → Process > Implement  → 合规 > 效率（风控优先）         │
  │                                                          │
  │  Layer 2: 技能切换     Layer 2: 业务流程切换              │
  │  → 唯一出口            → 流程唯一出口                     │
  │  → 横切触发            → 异常业务触发                    │
  │                                                          │
  │  Layer 1: 子智能体     Layer 1: 业务 Agent 通信           │
  │  → 请求/响应格式       → 任务/结果格式                   │
  │  → 状态枚举            → 业务状态枚举                     │
  └──────────────────────────────────────────────────────────┘
```

## Layer 4: 运营会话协议

```
  ┌──────────────────────────────────────────────────────────┐
  │  电商 AI 的入口注入：                                    │
  │                                                          │
  │  <EXTREMELY_IMPORTANT>                                   │
  │  你是电商运营 AI 助手。                                  │
  │                                                          │
  │  <EXTREMELY-IMPORTANT>                                   │
  │  如果当前场景可能涉及任何电商技能，你绝对必须调用技能。    │
  │  合规和风控技能优先于运营效率技能。                       │
  │  </EXTREMELY-IMPORTANT>                                  │
  │                                                          │
  │  ## 指令优先级                                           │
  │  1. 法律法规（消费者权益保护法、电商法）                  │
  │  2. 平台规则（各电商平台政策）                            │
  │  3. 电商技能规则                                         │
  │  4. 默认行为                                             │
  │  </EXTREMELY_IMPORTANT>                                  │
  │                                                          │
  │  与 Superpowers 的差异：                                 │
  │  → 优先级最高层不是"用户指令"而是"法律法规"             │
  │  → 电商 AI 不能因为用户要求就违反法律                   │
  │  → 这是电商特有的合规约束                                │
  │  → 类比：Superpowers 中"用户 > 技能 > 默认"             │
  │  → 电商中"法律 > 平台规则 > 技能 > 用户 > 默认"         │
  └──────────────────────────────────────────────────────────┘
```

## Layer 3: 业务技能路由

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的 1% 原则在电商中的适配：                  │
  │                                                          │
  │  开发场景：1% 可能匹配就调用 → 宁可误触发               │
  │  电商场景：1% 可能合规风险就触发 → 宁可误拦截           │
  │  → 逻辑相同，但方向相反                                  │
  │  → 开发：宁可多做（多触发技能）                          │
  │  → 电商：宁可多查（多触发风控/合规技能）                 │
  │                                                          │
  │  路由优先级：                                            │
  │  Superpowers: Process skills > Implementation skills      │
  │  电商: 合规/风控技能 > 运营效率技能                      │
  │                                                          │
  │  具体规则：                                              │
  │  1. 涉及退款/赔付 → 先触发风控技能                       │
  │  2. 涉及商品上架 → 先触发合规技能                        │
  │  3. 涉及定价/促销 → 先触发财务技能                       │
  │  4. 涉及营销投放 → 先触发预算技能                        │
  │  5. 涉及客服响应 → 先触发知识库技能                      │
  │  → 风控/合规总是优先——因为电商的错误成本不对称：         │
  │    漏杀欺诈 = 损失金钱                                   │
  │    误杀正常订单 = 损失一单收入                           │
  │    → 不对称性决定了风控优先                              │
  └──────────────────────────────────────────────────────────┘
```

## Layer 2: 业务流程切换

```
  ┌──────────────────────────────────────────────────────────┐
  │  电商主链的技能切换：                                    │
  │                                                          │
  │  选品分析 → 供应链对接 → 商品上架 → 营销投放              │
  │  → 订单处理 → 物流跟踪 → 售后服务 → 数据复盘            │
  │                                                          │
  │  唯一出口设计：                                          │
  │  选品分析：                                              │
  │    "选品完成后，唯一出口是供应链对接技能。                │
  │     不要跳过供应链直接上架商品。"                         │
  │                                                          │
  │  商品上架：                                              │
  │    "商品上架完成后，唯一出口是营销投放技能。              │
  │     不要在上架前就投放广告。"                              │
  │                                                          │
  │  横切触发：                                              │
  │  质量问题触发质检技能（任何环节）                         │
  │  → 在选品中发现质量投诉 → 触发质检                      │
  │  → 在上架中发现差评 → 触发质检                          │
  │  → 在物流中发现破损 → 触发质检                          │
  │                                                          │
  │  风控异常触发风控技能（任何环节）                         │
  │  → 选品环节发现刷单数据 → 触发风控                      │
  │  → 营销环节发现异常流量 → 触发风控                      │
  │  → 退款环节发现欺诈模式 → 触发风控                      │
  │                                                          │
  │  条件出口（电商特有）：                                   │
  │  售后完成 → if 退货率 > 阈值 → 选品分析                 │
  │            else → 数据复盘                                │
  │  → 条件出口 = 电商流程的"回环"机制                      │
  └──────────────────────────────────────────────────────────┘
```

## Layer 1: 业务 Agent 通信协议

```
  ┌──────────────────────────────────────────────────────────┐
  │  电商 Agent 的请求消息格式：                             │
  │                                                          │
  │  Task tool:                                              │
  │    description: "处理退款申请 #12345"                     │
  │    prompt: |                                             │
  │      你正在使用退款处理技能审核退款申请 #12345。          │
  │                                                          │
  │      ## 订单信息                                         │
  │      [订单号、商品、金额、下单时间]                       │
  │                                                          │
  │      ## 退款申请                                         │
  │      [用户原因、图片证据、申请时间]                       │
  │                                                          │
  │      ## CRITICAL: 不信任用户描述                         │
  │      用户描述可能不完整或不准确。                         │
  │      你必须独立验证订单状态和物流状态。                   │
  │                                                          │
  │      ## 你的工作                                         │
  │      1. 验证订单状态（已发货？已签收？）                  │
  │      2. 验证物流状态（在途？已退回？）                    │
  │      3. 检查用户历史（退款频率？欺诈记录？）              │
  │      4. 对照退款政策做决策                               │
  │      5. 报告决策和依据                                   │
  │                                                          │
  │  电商 Agent 的响应格式：                                 │
  │  Status: APPROVED | DENIED | NEEDS_REVIEW | ESCALATED     │
  │  Decision: [批准/拒绝/需人工/升级]                       │
  │  Evidence: [决策依据列表]                                 │
  │  Risk level: LOW | MEDIUM | HIGH                          │
  │                                                          │
  │  与 Superpowers 的差异：                                 │
  │  → 增加了 Risk level 字段（电商特有）                    │
  │  → 状态枚举不同（不是 DONE/BLOCKED 而是                 │
  │    APPROVED/DENIED/NEEDS_REVIEW/ESCALATED）              │
  │  → ESCALATED = 升级到人工处理（电商特有）               │
  │  → 电商某些决策需要人工确认（法律要求）                 │
  └──────────────────────────────────────────────────────────┘
```
