# 05 子智能体架构平移：开发协调 → 电商运营协调

Superpowers 的子智能体架构（实现者+规范审阅者+代码审阅者）在电商中对应为"执行者+合规审阅者+质量审阅者"。框架可复用，但角色和交互逻辑需要重新设计。

## 开发协调 → 电商运营协调

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的协调架构：                                │
  │  Controller → 实现者子智能体 → 规范审阅者 → 代码审阅者   │
  │  → 设计→计划→执行→验证→收尾                            │
  │                                                          │
  │  电商的协调架构：                                        │
  │  Controller → 运营执行者 → 合规审阅者 → 质量审阅者       │
  │  → 分析→执行→合规检查→质量检查                          │
  │                                                          │
  │  对照：                                                  │
  │  开发角色        电商角色           职责                  │
  │  ────────        ────────         ────                   │
  │  Controller      运营调度器        分配任务、传递上下文   │
  │  实现者          运营执行者        执行业务操作           │
  │  规范审阅者      合规审阅者        检查是否合规           │
  │  代码审阅者      质量审阅者        检查是否优质           │
  │  验证技能        结果验证          确认操作真正完成       │
  │  finishing       收尾归档          流程收尾和记录         │
  └──────────────────────────────────────────────────────────┘
```

## 电商场景：退款审核流程的子智能体协调

```
  ┌──────────────────────────────────────────────────────────┐
  │  场景：用户申请退款，AI 需要审核并决定                  │
  │                                                          │
  │  Step 1: Controller 派发退款审核者                       │
  │  ──────────────────────────────────                      │
  │  传入：                                                  │
  │  - 订单信息（商品、金额、下单时间）                      │
  │  - 退款申请（用户原因、图片证据）                        │
  │  - 退款政策（平台规则、店铺政策）                        │
  │  不传入：                                                │
  │  - 客户的浏览历史（隐私）                                │
  │  - 其他客户的退款数据（不相关）                          │
  │  - 销售指标（偏见隔离）                                  │
  │                                                          │
  │  Step 2: 退款审核者报告                                  │
  │  Status: APPROVED / DENIED / NEEDS_REVIEW / ESCALATED     │
  │  Decision: 批准退款                                      │
  │  Evidence: 证据充分 + 订单已签收 + 用户首次退款          │
  │  Risk level: LOW                                         │
  │                                                          │
  │  Step 3: Controller 派发合规审阅者                       │
  │  ──────────────────────────────────                      │
  │  传入：                                                  │
  │  - 退款审核者的决策和依据                                │
  │  - 合规要求（消费者权益保护法、平台规则）                │
  │  预设立场："审核者的决策可能不完全符合合规要求。          │
  │    你必须独立验证每条合规要求。"                         │
  │                                                          │
  │  Step 4: 合规审阅者报告                                  │
  │  ✅ 合规通过 / ❌ 合规问题：[具体问题列表]               │
  │                                                          │
  │  Step 5: Controller 综合决策                              │
  │  → 审核通过 + 合规通过 → 执行退款                       │
  │  → 审核通过 + 合规不通过 → 拒绝或升级                   │
  │  → 审核不通过 → 不执行退款                               │
  │  → NEEDS_REVIEW → 转人工                                 │
  │                                                          │
  │  关键差异：电商比开发多了一个"升级人工"出口             │
  │  → TDD 中没有"人工介入"——代码要么对要么错              │
  │  → 电商中很多决策需要人工判断                           │
  │  → "升级人工"是电商 AI 的安全阀                        │
  └──────────────────────────────────────────────────────────┘
```

## 电商场景：选品决策的子智能体协调

```
  ┌──────────────────────────────────────────────────────────┐
  │  场景：决定是否引入一个新品类                            │
  │                                                          │
  │  Step 1: Controller 派发选品分析者                        │
  │  传入：                                                  │
  │  - 市场数据（品类规模、增速、竞争度）                    │
  │  - 竞品分析（主要竞品、价格带、差异化空间）              │
  │  - 供应链信息（供应商、成本、起订量）                    │
  │  不传入：                                                │
  │  - 营销预算（偏见隔离——不因"有钱"就选贵品）             │
  │  - 其他品类的历史表现（不相关——每个品类是独立决策）     │
  │                                                          │
  │  选品分析者报告：                                        │
  │  Status: RECOMMENDED / NOT_RECOMMENDED / NEEDS_MORE_DATA   │
  │  Analysis: 品类增速 25%、竞争度中、毛利 35%、差异化可行   │
  │  Recommendation: 建议引入，优先 A 供应商                  │
  │                                                          │
  │  Step 2: Controller 派发合规审阅者                        │
  │  传入：                                                  │
  │  - 选品分析者的推荐和依据                                │
  │  - 合规要求（产品认证、进口规定、标签法规）              │
  │  预设立场："选品推荐可能忽略了合规风险。                  │
  │    你必须独立验证所有合规要求。"                         │
  │                                                          │
  │  Step 3: 合规审阅者报告                                  │
  │  ✅ 合规通过 / ❌ 合规问题                               │
  │                                                          │
  │  特殊情况：选品 + 合规 + 财务三方协调                    │
  │  → 如果选品推荐但合规有问题 → 需要评估修复成本           │
  │  → 如果修复成本过高 → 财务技能介入做 ROI 分析            │
  │  → 三方协调 = 电商比开发更常见的模式                    │
  │  → Superpowers 只有两阶段审阅（规范+质量）               │
  │  → 电商可能需要三阶段审阅（合规+质量+财务）             │
  └──────────────────────────────────────────────────────────┘
```

## 子智能体隔离的电商挑战

```
  ┌──────────────────────────────────────────────────────────┐
  │  挑战 1: 数据访问方式不同                                │
  │  开发子智能体：读写文件系统 + git                         │
  │  电商子智能体：访问数据库 + API + 第三方平台             │
  │  → 需要为电商子智能体定义数据访问接口                   │
  │  → 文件系统是本地的，API 是远程的                       │
  │  → 远程访问有延迟、失败、限流等问题                    │
  │  → 开发中不需要考虑 API 调用失败                        │
  │  → 电商中 API 失败是常态（平台限流、网络超时）           │
  │  → 需要增加"重试"和"降级"机制                         │
  │                                                          │
  │  挑战 2: 操作的不可逆性                                  │
  │  开发操作：大部分可逆（git reset、删除文件可恢复）        │
  │  电商操作：很多不可逆（退款已执行=钱已转出）             │
  │  → 电商子智能体需要更强的"确认机制"                     │
  │  → 开发中"提交代码"可以 revert                           │
  │  → 电商中"执行退款"不能 undo                             │
  │  → 需要增加"两步确认"或"审批流"机制                    │
  │                                                          │
  │  挑战 3: 实时性要求                                      │
  │  开发流程：分钟级响应可接受                               │
  │  电商流程：秒级响应是标配                                │
  │  → 子智能体派发有延迟（上下文构造 + 模型推理）           │
  │  → 客服场景中用户等不了 30 秒                           │
  │  → 需要区分"实时技能"和"批处理技能"                    │
  │  → 实时技能（客服）：预加载技能，不派发子智能体          │
  │  → 批处理技能（选品/财务）：可以派发子智能体             │
  │  → 电商的子智能体策略 = 实时 vs. 批处理的混合架构        │
  └──────────────────────────────────────────────────────────┘
```
