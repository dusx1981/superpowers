# 04 红旗表格

红旗表格是结构性权重中**最精妙**的类型——它不直接告诉 AI 做什么，而是预判 AI 内心可能出现的合理化想法，逐一批驳。

## 4 套红旗表格

### 4.1 using-superpowers 红旗（12 项）

**来源**：`skills/using-superpowers/SKILL.md:78-95`

| 想法 | 现实 |
|------|------|
| "这只是个简单的问题" | 问题也是任务。检查技能。 |
| "我需要先更多上下文" | 技能检查在澄清问题之前。 |
| "让我先探索代码库" | 技能告诉你如何探索。先检查。 |
| "我可以快速检查文件" | 文件缺少对话上下文。检查技能。 |
| "让我先收集信息" | 技能告诉你如何收集信息。 |
| "这不需要正式技能" | 如果技能存在，就使用它。 |
| "我记得这个技能" | 技能会演进。读当前版本。 |
| "这不算任务" | 行动 = 任务。检查技能。 |
| "技能杀鸡用牛刀了" | 简单的事会变复杂。使用它。 |
| "我先做这一件事" | 在做任何事之前检查。 |
| "这感觉很高产" | 不受纪律约束的行动浪费时间。 |
| "我知道那是什么意思" | 知道概念 ≠ 使用技能。调用它。 |

**结构特征**：
- 表格格式——左右对比，视觉冲击强
- 左列用引号——模拟 AI 的内心独白
- 右列简短反驳——不解释为什么，直接给现实
- 覆盖 12 种合理化——最全面的防御

### 4.2 TDD 合理化表格（12 项）

**来源**：`skills/test-driven-development/SKILL.md:245-270`

| 借口 | 现实 |
|------|------|
| "太简单不需要测试" | 简单代码也会坏。测试花 30 秒。 |
| "我之后再测" | 之后通过的测试什么也证明不了。 |
| "之后测试达到同样目的" | 之后测试 = "这做了什么？" 之前测试 = "这应该做什么？" |
| "已经手动测了" | 临时测试 ≠ 系统测试。 |
| "删 X 小时工作是浪费" | 沉没成本谬误。保留未验证代码是技术债。 |
| "留作参考，先写测试" | 你会适配它。那是测试后写。删除=删除。 |
| "需要先探索" | 可以。扔掉探索代码，从 TDD 开始。 |
| "测试难写 = 设计有误" | 听测试的。难测试 = 难使用。 |
| "TDD 会拖慢我" | TDD 比调试快。实用 = 测试先行。 |
| "手动测试更快" | 手动不证明边缘情况。 |
| "现有代码没测试" | 你在改善它。为现有代码加测试。 |

**独特设计**：
- "之后测试 vs 之前测试"的对比——不是简单反驳，而是指出本质差异
- "删除=删除"——极简表达，消除误解空间
- 使用认知偏差名称（"沉没成本谬误"）——用心理学语言增强说服力

### 4.3 验证合理化表格（8 项）

**来源**：`skills/verification-before-completion/SKILL.md:64-74`

| 借口 | 现实 |
|------|------|
| "应该没问题了" | 运行验证命令。 |
| "我很自信" | 自信 ≠ 证据。 |
| "就这一次" | 没有例外。 |
| "Linter 通过了" | Linter ≠ 编译器。 |
| "Agent 说成功了" | 独立验证。 |
| "我累了" | 疲惫 ≠ 借口。 |
| "部分检查够了" | 部分证明不了什么。 |
| "换了个说法所以规则不适用" | 精神超越字面。 |

**独特设计**：
- "精神超越字面"——预判 AI 可能用不同措辞绕过规则
- 反驳极短（1-5 个词）——不给 AI 思考反驳的空间
- "没有例外"——绝对式回应

### 4.4 调试合理化表格（8 项）

**来源**：`skills/systematic-debugging/SKILL.md:245-256`

| 借口 | 现实 |
|------|------|
| "问题简单，不需要流程" | 简单问题也有根因。流程对简单 bug 很快。 |
| "紧急，没时间走流程" | 系统调试比猜着修更快。 |
| "先试这个，再调查" | 第一个修复定调。从开始就做对。 |
| "确认修复后再写测试" | 未测试的修复不持久。测试先行证明它。 |
| "一次修多个节省时间" | 无法隔离什么起效了。造成新 bug。 |
| "参考太长，我适配模式" | 部分理解保证有 bug。完整阅读。 |
| "我看到问题了，修它" | 看到症状 ≠ 理解根因。 |
| "再试一次修复"（2+次后） | 3+ 次失败 = 架构问题。质疑模式。 |

**独特设计**：
- "第一个修复定调"——暗示修复会形成模式，第一个错了后面都错
- "看到症状 ≠ 理解根因"——用 ≠ 符号强化差异
- 有升级机制（2+ 次 → 质疑架构）

## 红旗表格的三层结构

```
  ┌──────────────────────────────────────────────────────────┐
  │  每个红旗条目的三层结构：                                │
  │                                                          │
  │  Layer 1: 识别（左列）                                   │
  │  ──────────────────                                      │
  │  用引号括起 AI 可能的想法                                │
  │  → "这太简单了"                                         │
  │  → "我之后再测"                                         │
  │  → "应该没问题了"                                       │
  │  引号模拟内心独白——AI 读到时会产生"被看穿"的感觉        │
  │                                                          │
  │  Layer 2: 反驳（右列）                                   │
  │  ──────────────────                                      │
  │  用事实直接反驳                                          │
  │  → "简单代码也会坏"                                     │
  │  → "之后通过的测试什么也证明不了"                        │
  │  → "运行验证命令"                                       │
  │  不解释为什么——直接给现实，消除讨论空间                  │
  │                                                          │
  │  Layer 3: 指令（隐含）                                   │
  │  ──────────────────                                      │
  │  反驳本身隐含了必须执行的行动                            │
  │  → "测试花 30 秒" → 去测试                              │
  │  → "从测试先行开始" → 用 TDD                            │
  │  → "没有例外" → 必须验证                                 │
  │  不显式说"你应该做 X"——因为反驳已经暗示了行动          │
  └──────────────────────────────────────────────────────────┘
```

## 表格格式 vs 列表格式的权重差异

```
  列表格式：
  ────────
  - "太简单不需要测试"：简单代码也会坏
  - "我之后再测"：之后通过的测试什么也证明不了
  - "应该没问题了"：运行验证命令

  → 线性阅读，AI 可能跳过某些条目
  → 借口和反驳混在一起，视觉权重相同

  表格格式：
  ────────
  | 借口 | 现实 |
  |------|------|
  | "太简单不需要测试" | 简单代码也会坏 |
  | "我之后再测" | 之后通过的测试什么也证明不了 |
  | "应该没问题了" | 运行验证命令 |

  → 左右对比，视觉冲击强
  → 借口在左、反驳在右——因果关系视觉化
  → 表格结构本身暗示"这些借口都有反驳"
  → AI 读到左列时，右列的否决已经等待
```

## 红旗 + Anti-Pattern 专节的协同

brainstorming 技能中，红旗表格和 Anti-Pattern 专节形成协同：

```
  红旗表格（通用防御）：
  "This is just a simple question" → 检查技能

  Anti-Pattern 专节（针对性防御）：
  "This Is Too Simple To Need A Design"
  Every project goes through this process.
  "Simple" projects are where unexamined assumptions
  cause the most wasted work.

  协同效果：
  - 红旗表格覆盖通用场景（任何任务都可能"太简单"）
  - Anti-Pattern 专节深入特定场景（设计前的"太简单"）
  - 双重防御——即使红旗没拦住，Anti-Pattern 专节再次拦截
```
