# 11 子智能体隔离

子智能体隔离是结构性权重中**最根本**的类型——它不依赖语言或格式，而是依赖架构：全新实例无法绕过 Controller 的控制，因为它根本不知道 Controller 的存在。

## 隔离的架构基础

```
  ┌──────────────────────────────────────────────────────────┐
  │  主会话 (Controller)                                     │
  │  ┌────────────────────────────────────────────────────┐  │
  │  │  上下文：用户对话 + 全部技能 + 注入的 bootstrap   │  │
  │  │  角色：协调者，调度子智能体                        │  │
  │  │  可见信息：计划全文 + 用户偏好 + 历史对话          │  │
  │  └────────────────────────────────────────────────────┘  │
  │         │ dispatch                                       │
  │         ▼                                                │
  │  ┌────────────────────────────────────────────────────┐  │
  │  │  子智能体 A (实现者)                               │  │
  │  │  上下文：仅 Prompt 模板注入的内容                  │  │
  │  │  角色：实现者，完成任务并报告                      │  │
  │  │  可见信息：任务描述 + 上下文 + 工作目录            │  │
  │  │  不可见：其他任务、用户偏好、历史对话              │  │
  │  └────────────────────────────────────────────────────┘  │
  │         │ report                                         │
  │         ▼                                                │
  │  Controller 评估报告 → 调度审阅者                         │
  │         │ dispatch                                       │
  │         ▼                                                │
  │  ┌────────────────────────────────────────────────────┐  │
  │  │  子智能体 B (规范审阅者)                           │  │
  │  │  上下文：仅审阅者模板注入的内容                    │  │
  │  │  角色：审阅者，验证实现符合规范                    │  │
  │  │  可见信息：规范 + 实现者报告 + 代码                │  │
  │  │  不可见：实现者的上下文、其他任务、用户偏好        │  │
  │  └────────────────────────────────────────────────────┘  │
  └──────────────────────────────────────────────────────────┘
```

## 为什么隔离是权重？

```
  ┌──────────────────────────────────────────────────────────┐
  │  隔离之所以是结构性权重，是因为它消除了                   │
  │  "子智能体自行其是"的可能性：                            │
  │                                                          │
  │  1. 无法绕过流程                                         │
  │     子智能体不知道"流程"的存在                           │
  │     它只知道自己被要求做一件事并报告                     │
  │     → 不可能"跳过"它不知道的步骤                        │
  │                                                          │
  │  2. 无法篡改上下文                                       │
  │     子智能体看到的信息由 Controller 精确控制              │
  │     → 不可能"参考"其他任务或用户偏好                    │
  │     → 不可能"受影响"于之前任务的结果                    │
  │                                                          │
  │  3. 无法跳过审阅                                         │
  │     审阅者是独立的子智能体                               │
  │     实现者无法"说服"审阅者                              │
  │     → 审阅者只能读代码，不能读实现者的"心里话"          │
  │                                                          │
  │  4. 无法改变报告格式                                     │
  │     Prompt 模板固定了 4 种状态码                         │
  │     子智能体必须选择其中一个                             │
  │     → 不可能用模糊文本回避判断                          │
  └──────────────────────────────────────────────────────────┘
```

## SUBAGENT-STOP 标签

**来源**：`skills/using-superpowers/SKILL.md:6-8`

```html
<SUBAGENT-STOP>
If you were dispatched as a subagent to execute a specific task,
skip this skill.
</SUBAGENT-STOP>
```

**双重保护**：

```
  ┌──────────────────────────────────────────────────────────┐
  │  SUBAGENT-STOP 的两个作用：                              │
  │                                                          │
  │  1. 防止子智能体加载全套技能                             │
  │     如果子智能体也加载 using-superpowers，               │
  │     它会试图调度更多子智能体 → 无限递归                  │
  │     SUBAGENT-STOP 让子智能体跳过技能加载                │
  │                                                          │
  │  2. 保持子智能体聚焦                                     │
  │     技能体系是为 Controller 设计的协调框架               │
  │     子智能体只需要"做一件事"，不需要"协调一切"          │
  │     跳过技能加载 = 子智能体只遵循 Prompt 模板            │
  │                                                          │
  │  权重来源：                                              │
  │  - 标签名 "SUBAGENT-STOP" = "停！"                      │
  │  - 位于 using-superpowers 的最前面                       │
  │  - 条件判断 "If you were dispatched as a subagent"       │
  │  - 子智能体读到这条 → 跳过整个技能体系                  │
  └──────────────────────────────────────────────────────────┘
```

## "Fresh subagent per task" 原则

```
  ┌──────────────────────────────────────────────────────────┐
  │  subagent-driven-development 的核心原则：                │
  │                                                          │
  │  "Fresh subagent per task + two-stage review             │
  │   = high quality, fast iteration"                        │
  │                                                          │
  │  为什么每个任务用新的子智能体？                          │
  │                                                          │
  │  1. 无上下文污染                                         │
  │     上一个任务的假设、误解、捷径不会带入下一个           │
  │                                                          │
  │  2. 无确认偏误                                           │
  │     实现者不会因为"之前这样做的"而重复同样模式          │
  │                                                          │
  │  3. 精确上下文控制                                       │
  │     Controller 为每个任务精确构造上下文                  │
  │     子智能体只看到"需要看到的"                           │
  │                                                          │
  │  4. 可重现性                                             │
  │     同样的 Prompt 模板 + 同样的上下文 = 可重现的行为     │
  │                                                          │
  │  代价：                                                  │
  │  - 更多子智能体调用（实现者 + 2 个审阅者/任务）         │
  │  - Controller 更多准备工作（提取所有任务全文）           │
  │  - 审阅循环增加迭代次数                                  │
  │  - 但问题发现更早（比事后调试更便宜）                   │
  └──────────────────────────────────────────────────────────┘
```

## 子智能体不能继承什么？

```
  子智能体隔离意味着以下信息不可继承：

  ┌────────────────────────────────────────┐
  │  不可继承的信息                         │
  │  ────────────────                       │
  │  ❌ 用户的对话历史                      │
  │  ❌ 用户的偏好和约束                    │
  │  ❌ 其他任务的内容和结果                │
  │  ❌ Controller 的决策逻辑               │
  │  ❌ 技能体系的完整内容                  │
  │  ❌ 之前审阅者的反馈                    │
  │                                        │
  │  可以继承的信息（由 Controller 注入）   │
  │  ──────────────────────────────         │
  │  ✅ 当前任务的完整描述                  │
  │  ✅ 场景上下文（任务在哪里、为什么）    │
  │  ✅ 工作目录                            │
  │  ✅ Prompt 模板定义的角色和约束          │
  │  ✅ 实现者的报告（给审阅者看）          │
  └────────────────────────────────────────┘

  Controller 的"构造"职责：
  - 决定子智能体需要看到什么
  - 粘贴任务全文（不让子智能体读文件）
  - 提供场景上下文（让子智能体知道"为什么"）
  - 选择模型（根据任务复杂度）
```

## 隔离 vs 非隔离的对比

```
  非隔离模式（单会话执行所有任务）：
  ────────────────────────────────
  任务 1 → 任务 2 → 任务 3 → 审阅

  问题：
  - 任务 2 受任务 1 的上下文影响
  - 任务 3 可能"参考"任务 2 的实现
  - 审阅时 AI 已经"太熟悉"代码，难以客观
  - 错误在任务 1 中引入，传播到所有后续任务

  隔离模式（子智能体驱动开发）：
  ────────────────────────────────
  Controller → [实现者1] → [审阅者1] → 完成
           → [实现者2] → [审阅者2] → 完成
           → [实现者3] → [审阅者3] → 完成
           → [最终审阅者]

  优势：
  - 每个实现者从零开始，无上下文污染
  - 每个审阅者独立评估，不受实现者影响
  - 错误不会跨任务传播
  - Controller 全局视角，协调所有子智能体
```
