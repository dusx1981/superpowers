# 14 模型选择

模型选择是结构性权重中**最易被忽视**的类型——它不控制"做什么"，而是控制"谁来做"。弱模型执行强任务 = 失败，强模型执行弱任务 = 浪费。

## 模型选择策略

**来源**：`skills/subagent-driven-development/SKILL.md:87-101`

```
  ┌──────────────────────────────────────────────────────────┐
  │  核心原则：用能完成任务的最弱模型                         │
  │                                                          │
  │  任务类型             推荐模型层级         理由           │
  │  ──────────           ──────────           ────           │
  │  机械实现任务         快速/廉价模型        规范明确，      │
  │  (1-2 文件, 完整规范)                       只需执行      │
  │                                                          │
  │  集成和判断任务       标准模型             多文件协调，    │
  │  (多文件, 模式匹配)                         需要判断      │
  │                                                          │
  │  架构/设计/审阅任务   最强模型             需要广泛理解    │
  │                                              和深度判断    │
  └──────────────────────────────────────────────────────────┘
```

## 任务复杂度信号

```
  ┌──────────────────────────────────────────────────────────┐
  │  复杂度判断标准：                                        │
  │                                                          │
  │  信号                            → 模型选择              │
  │  ─────                           ──────────              │
  │  触及 1-2 文件 + 完整规范        → 廉价模型              │
  │  触及多个文件 + 集成考量         → 标准模型              │
  │  需要设计判断 + 广泛代码理解     → 最强模型              │
  │                                                          │
  │  关键洞察：                                              │
  │  "Most implementation tasks are mechanical                │
  │   when the plan is well-specified."                      │
  │                                                          │
  │  → 如果计划写得好，大多数任务不需要强模型                │
  │  → 强模型应该用在"计划不好写"的地方                     │
  │  → 这也是为什么 writing-plans 技能那么重要               │
  └──────────────────────────────────────────────────────────┘
```

## BLOCKED 时的模型升级

**来源**：`skills/subagent-driven-development/SKILL.md:113-118`

```
  Controller 收到 BLOCKED 后的决策树中，模型选择是关键：

  1. 上下文问题 → 提供更多上下文，用相同模型重新调度
  2. 能力问题 → 用更强模型重新调度
  3. 任务太大 → 拆分为更小的任务
  4. 计划有误 → 升级给人类

  关键约束：
  "Never ignore an escalation or force the same model
   to retry without changes."

  如果子智能体说卡住了：
  - 不能什么都不改就重试
  - 如果是能力不足，必须升级模型
  - 强制同一个弱模型重试 = 浪费时间和 token
```

## 模型选择为什么是权重？

```
  ┌──────────────────────────────────────────────────────────┐
  │  模型选择的权重逻辑：                                    │
  │                                                          │
  │  1. 执行能力 = 权重的基础                                │
  │     再强的指令，给一个理解不了的模型 = 零权重            │
  │     模型选择保证了 AI 有能力执行指令                     │
  │                                                          │
  │  2. 匹配防止"低能高用"                                  │
  │     廉价模型执行审阅任务 → 审阅质量差                    │
  │     → 审阅循环失效 → 权重体系崩溃                       │
  │                                                          │
  │  3. 匹配防止"高能低用"                                  │
  │     强模型执行机械任务 → 浪费 token/成本                 │
  │     → 系统不可持续 → 限制使用频率                       │
  │     → 技能体系被"太贵"而绕过                            │
  │                                                          │
  │  4. 升级机制防止"死循环"                                │
  │     弱模型 BLOCKED → 不升级就重试 → 再次 BLOCKED        │
  │     升级模型 → 成功 → 流程继续                          │
  │     没有"强制升级"约束 = 可能无限重试                   │
  └──────────────────────────────────────────────────────────┘
```

## 各角色的推荐模型

```
  ┌──────────────────────────────────────────────────────────┐
  │  角色与模型匹配：                                        │
  │                                                          │
  │  Controller (主会话)                                     │
  │  → 最强模型                                              │
  │  → 需要理解全局、调度子智能体、评估报告                  │
  │                                                          │
  │  实现者 (子智能体)                                       │
  │  → 根据任务复杂度选择                                    │
  │  → 大多数任务是廉价模型（规范已完整）                    │
  │  → 集成任务用标准模型                                    │
  │                                                          │
  │  规范审阅者 (子智能体)                                   │
  │  → 标准或强模型                                          │
  │  → 需要精确比对规范和代码                                │
  │  → 判断力不足 = 漏过不符合规范的代码                    │
  │                                                          │
  │  代码质量审阅者 (子智能体)                               │
  │  → 强模型                                                │
  │  → 需要广泛代码理解 + 设计判断                           │
  │  → 低质量审阅 = 技术债                                  │
  │                                                          │
  │  头脑风暴 (主会话)                                       │
  │  → 最强模型                                              │
  │  → 需要提出 2-3 种方案 + 权衡取舍                       │
  │  → 创造力和判断力都需要                                  │
  └──────────────────────────────────────────────────────────┘
```

## 模型选择与成本/质量的平衡

```
  ┌──────────────────────────────────────────────────────────┐
  │  subagent-driven-development 的成本分析：                │
  │                                                          │
  │  每个任务的子智能体调用：                                │
  │  - 1 次实现者 (通常廉价模型)                             │
  │  - 1-2 次规范审阅者 (标准模型)                           │
  │  - 1-2 次代码质量审阅者 (强模型)                         │
  │  → 每个任务 3-5 次子智能体调用                           │
  │                                                          │
  │  5 个任务的项目：15-25 次子智能体调用                    │
  │                                                          │
  │  如果不区分模型：                                        │
  │  - 全用强模型 → 成本 3-5 倍                             │
  │  - 全用弱模型 → 审阅质量不足                            │
  │                                                          │
  │  模型选择的价值：                                        │
  │  - 实现任务（占 60-70% 调用）用廉价模型                  │
  │  - 审阅任务（占 30-40% 调用）用标准/强模型               │
  │  - 整体成本降低 40-60%，质量不降                        │
  │                                                          │
  │  "But catches issues early (cheaper than debugging       │
  │   later)" — 前期审阅比后期调试便宜                      │
  └──────────────────────────────────────────────────────────┘
```
