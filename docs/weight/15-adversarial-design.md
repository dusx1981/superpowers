# 15 对抗性设计

对抗性设计是结构性权重中**最具结构性**的类型——它不依赖语言或格式，而是依赖角色间的利益对立：实现者想通过，审阅者想拦住。只要两者利益对立，任何一方都无法单方面降低标准。

## 对抗性的角色架构

```
  ┌──────────────────────────────────────────────────────────┐
  │  角色利益对立：                                          │
  │                                                          │
  │  实现者：                                                │
  │  "实现规范要求的内容，尽快完成"                          │
  │  → 利益：通过审阅，标记 DONE                             │
  │  → 倾向：可能偷懒、过度工程、忽略边缘情况               │
  │                                                          │
  │  规范审阅者：                                            │
  │  "验证代码是否精确符合规范"                              │
  │  → 利益：找出偏差                                       │
  │  → 倾向：严格比对，不信任实现者报告                     │
  │                                                          │
  │  代码质量审阅者：                                        │
  │  "验证代码质量是否达标"                                  │
  │  → 利益：找出质量问题                                    │
  │  → 倾向：严格评估，不接受"差不多"                       │
  │                                                          │
  │  对抗关系：                                              │
  │  实现者 ←→ 规范审阅者 （做了没有？）                    │
  │  实现者 ←→ 质量审阅者 （做得好不好？）                  │
  │  审阅者之间：顺序锁定（先规范后质量）                   │
  └──────────────────────────────────────────────────────────┘
```

## "Do Not Trust the Report" 原则

**来源**：`skills/subagent-driven-development/spec-reviewer-prompt.md:21-36`

```
  ## CRITICAL: Do Not Trust the Report

  The implementer finished suspiciously quickly.
  Their report may be incomplete, inaccurate, or optimistic.
  You MUST verify everything independently.

  **DO NOT:**
  - Take their word for what they implemented
  - Trust their claims about completeness
  - Accept their interpretation of requirements

  **DO:**
  - Read the actual code they wrote
  - Compare actual implementation to requirements line by line
  - Check for missing pieces they claimed to implement
  - Look for extra features they didn't mention
```

**对抗性的权重来源**：

| 维度 | 分析 |
|------|------|
| "suspiciously quickly" | 预设不信任——审阅者被指示怀疑实现者 |
| "DO NOT" / "DO" 对称 | 同时禁止信任和命令独立验证 |
| "line by line" | 审查粒度——不允许粗略比对 |
| "extra features" | 不仅查缺失，还查多余——防止过度工程 |
| "incomplete, inaccurate, or optimistic" | 三种不信任维度覆盖所有可能 |

## 审阅者检查的三个维度

```
  ┌──────────────────────────────────────────────────────────┐
  │  规范审阅者的三维度对抗：                                │
  │                                                          │
  │  维度 1: 缺失需求 (Missing)                              │
  │  ──────────────────────────                              │
  │  "Did they implement everything that was requested?"     │
  │  对抗实现者的"偷懒"倾向——跳过难做或模糊的需求          │
  │                                                          │
  │  维度 2: 多余工作 (Extra)                                │
  │  ────────────────────────                                │
  │  "Did they build things that weren't requested?"         │
  │  对抗实现者的"过度工程"倾向——添加"nice to have"        │
  │                                                          │
  │  维度 3: 误解需求 (Misunderstandings)                    │
  │  ───────────────────────────────────────                 │
  │  "Did they interpret requirements differently?"          │
  │  对抗实现者的"自作主张"倾向——用自己的理解替代规范      │
  │                                                          │
  │  三维度覆盖了所有"实现与规范不符"的可能：              │
  │  - 少做了 → 缺失                                        │
  │  - 多做了 → 多余                                        │
  │  - 做错了 → 误解                                        │
  │  除了"精确符合规范"以外，所有情况都是 ❌                │
  └──────────────────────────────────────────────────────────┘
```

## 为什么自审不能替代对抗性审阅？

```
  ┌──────────────────────────────────────────────────────────┐
  │  自审的局限性：                                          │
  │                                                          │
  │  1. 确认偏误                                             │
  │     实现者知道自己的意图                                 │
  │     → 读代码时"看到"自己想表达的东西                    │
  │     → 不容易发现遗漏和误解                              │
  │                                                          │
  │  2. 沉没成本                                             │
  │     实现者花了时间写代码                                 │
  │     → 倾向于认为"做的东西是有价值的"                    │
  │     → 不容易发现"多余工作"                              │
  │                                                          │
  │  3. 利益一致                                             │
  │     实现者和自己的利益一致                               │
  │     → "差不多就行"的心态                                │
  │     → 不愿意给自己找麻烦                                │
  │                                                          │
  │  对抗性审阅的优势：                                      │
  │                                                          │
  │  1. 新鲜视角                                             │
  │     审阅者是全新实例，没有实现者的思维包袱               │
  │     → 容易发现遗漏和误解                                │
  │                                                          │
  │  2. 利益对立                                             │
  │     审阅者的目标是"找出问题"                            │
  │     → 主动寻找遗漏和多余                                │
  │     → 不受"沉没成本"影响                               │
  │                                                          │
  │  3. 被指示不信任                                         │
  │     "DO NOT trust their claims"                          │
  │     → 审阅者被结构性地要求怀疑实现者                    │
  │     → 不是"可选的谨慎"，而是"必须的不信任"             │
  └──────────────────────────────────────────────────────────┘
```

## 对抗性设计的代码级实现

```
  对抗性不是概念性的——它在代码和模板中有具体实现：

  1. 规范审阅者 Prompt 中嵌入不信任原则
     → "Do Not Trust the Report" 是模板的固定部分
     → 不是 Controller 在调度时决定"这次要不要信任"
     → 而是模板本身就包含不信任指令

  2. 代码质量审阅者使用独立模板
     → requesting-code-review/code-reviewer.md
     → 不是实现者模板的变体，而是完全独立的模板
     → 角色定义、检查维度、输出格式都不同

  3. 审阅者不继承实现者的上下文
     → 全新实例，看不到实现者的思考过程
     → 只能看到代码和规范
     → 不受实现者"我本意是..."的影响

  4. 审阅顺序强制锁定
     → 先规范后质量
     → 实现者不能跳过规范审阅直接做质量审阅
     → 两个审阅者检查不同维度，不可替代
```

## 对抗性 vs 协作性的平衡

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 的设计选择：                                │
  │                                                          │
  │  实现阶段：协作性                                        │
  │  - 实现者可以在开始前提问                               │
  │  - Controller 回答问题、提供上下文                       │
  │  - "While you work: ask questions. It's always OK        │
  │     to pause and clarify."                               │
  │                                                          │
  │  审阅阶段：对抗性                                        │
  │  - 审阅者被指示不信任实现者                             │
  │  - 实现者不能"说服"审阅者                              │
  │  - 只有代码是共同事实                                   │
  │                                                          │
  │  修复阶段：协作性                                        │
  │  - 实现者修复问题后重新提交                             │
  │  - Controller 协调修复循环                               │
  │  - 修复不是"惩罚"，是"改进"                            │
  │                                                          │
  │  升级阶段：对抗性                                        │
  │  - 实现者 BLOCKED → Controller 必须改变条件              │
  │  - "Never force the same model to retry                  │
  │     without changes"                                     │
  │  - 不能假装问题不存在                                   │
  │                                                          │
  │  平衡点：                                                │
  │  - 在"创造"时协作（实现、提问、修复）                   │
  │  - 在"验证"时对抗（审阅、不信任、强制修复）             │
  │  - 协作提高效率，对抗保证质量                            │
  └──────────────────────────────────────────────────────────┘
```

## 对抗性设计与其他权重的协同

```
  对抗性设计不是孤立起效的——它与其他权重类型形成协同：

  + 结构化状态 (08):
    实现者必须报告 DONE/DONE_WITH_CONCERNS/BLOCKED/NEEDS_CONTEXT
    → 审阅者看到实现者的状态码，知道该期望什么
    → DONE_WITH_CONCERNS → 审阅者更警惕

  + 审阅循环 (12):
    审阅不通过 → 实现者修复 → 重新审阅
    → 对抗性在循环中持续——不是一次性的
    → 直到审阅者批准，对抗才结束

  + 子智能体隔离 (11):
    审阅者是全新实例，不继承实现者上下文
    → 对抗性由隔离保证——审阅者不可能"同情"实现者
    → 因为它根本不知道实现者是谁

  + 反模式列表 (07):
    审阅者知道常见反模式
    → 对抗性有具体的检查清单
    → 不是泛泛的"仔细审查"，而是"检查这些具体模式"

  + 否定式语言 (03):
    "DO NOT trust" / "Never accept" / "Do NOT take their word"
    → 对抗性用否定式强化——不是"建议不信任"而是"禁止信任"
```
