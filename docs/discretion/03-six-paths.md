# 03 六种绕过路径

AI 绕过指令有六种路径。每封堵一条，裁量空间缩小一层。Superpowers 的设计目标是 6/6 全封堵。

## 六种路径及封堵

### 路径 1: 降级——"这只是建议"

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 的推理：                                             │
  │  "这个指令用了'建议'/'应该'/'考虑'等词"               │
  │  → "所以它不是强制性的"                                 │
  │  → "我可以选择不遵循"                                   │
  │                                                          │
  │  封堵方式：情态动词升级                                  │
  │  ──────────────────────                                  │
  │  should → must → NO/NEVER                                │
  │                                                          │
  │  "应该先测试"     → AI 可以降级                         │
  │  "必须先测试"     → AI 很难降级                         │
  │  "NO CODE WITHOUT TEST" → AI 无法降级                    │
  │                                                          │
  │  封堵强度：should(~0.1) → must(~0.7) → NO(~0.95)       │
  │  → 从"可选"到"义务"到"铁律"                           │
  └──────────────────────────────────────────────────────────┘
```

### 路径 2: 例外——"这次情况特殊"

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 的推理：                                             │
  │  "规则是通用的，但这个场景有特殊性"                     │
  │  → "紧急情况可以例外"                                   │
  │  → "太简单不需要"                                       │
  │  → "用户看起来很急"                                     │
  │                                                          │
  │  封堵方式：全称量词                                      │
  │  ────────────────                                        │
  │  "先测试" → "每次都先测试" → "EVERY project             │
  │   regardless of perceived simplicity"                    │
  │                                                          │
  │  全称量词的递进：                                        │
  │  无全称 → AI 可声称"这次不同"                           │
  │  "every" → AI 不能声称"这次不同"                        │
  │  "every + regardless of..."                              │
  │    → AI 甚至不能把"简单"作为理由                       │
  │                                                          │
  │  封堵强度：无(~0) → every(~0.7)                         │
  │    → every+regardless(~0.95)                             │
  └──────────────────────────────────────────────────────────┘
```

### 路径 3: 后补——"我先做再补"

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 的推理：                                             │
  │  "规则说先测试，但先写代码再补测试也行"               │
  │  → "我先写代码快速实现，之后补测试"                     │
  │  → "反正最终都有测试"                                   │
  │                                                          │
  │  封堵方式：时序锁定                                      │
  │  ────────────────                                        │
  │  "先测试" → "测试 FIRST" → "BEFORE any code"            │
  │    → "UNTIL design approved"                             │
  │                                                          │
  │  时序词的递进：                                          │
  │  "先" → 模糊（可以先做别的再测试）                     │
  │  "FIRST" → 精确（必须是第一个动作）                     │
  │  "BEFORE any code" → 超精确（任何代码之前）             │
  │  "UNTIL Y" → 条件锁定（Y 完成前禁止 X）                │
  │                                                          │
  │  封堵强度："先"(~0.2) → FIRST(~0.8)                     │
  │    → BEFORE any(~0.95) → UNTIL(~0.98)                   │
  └──────────────────────────────────────────────────────────┘
```

### 路径 4: 重新定义——"这不算是写代码"

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 的推理：                                             │
  │  "规则说不写代码，但探索性编码不算'代码'"               │
  │  → "配置文件不算代码"                                   │
  │  → "测试本身也是代码"                                   │
  │  → "我在写文档，不是代码"                               │
  │                                                          │
  │  封堵方式：精确定义                                      │
  │  ────────────────                                        │
  │  "代码" → "PRODUCTION CODE" → "PRODUCTION CODE           │
  │   WITHOUT A FAILING TEST"                                │
  │                                                          │
  │  定义的递进：                                            │
  │  "代码" → 模糊（AI 可以重新定义什么是代码）             │
  │  "PRODUCTION CODE" → 排除了探索性/配置/测试代码         │
  │  "WITHOUT A FAILING TEST" → 进一步精确了前提条件        │
  │                                                          │
  │  封堵强度："代码"(~0.1) → PRODUCTION CODE(~0.7)         │
  │    → + FAILING TEST(~0.9)                               │
  │                                                          │
  │  注意：无法 100% 封堵——AI 总能找到更极端的重新定义     │
  │  → 但足够精确的定义把重新定义的路径压缩到极窄          │
  └──────────────────────────────────────────────────────────┘
```

### 路径 5: 合理化——"TDD 是教条主义"

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 的推理：                                             │
  │  "TDD 是一种教条，我应该务实"                           │
  │  → "这次跳过 TDD 更务实"                               │
  │  → "实用主义 > 教条主义"                                │
  │  → "精神和形式不同"                                     │
  │                                                          │
  │  封堵方式：红旗表格 + 反驳                               │
  │  ────────────────────────                                │
  │  预判 AI 可能的合理化想法，逐一批驳。                    │
  │                                                          │
  │  Superpowers 的 4 套红旗表格覆盖 40+ 种合理化：         │
  │  - using-superpowers (12 项)                              │
  │  - TDD (12 项)                                           │
  │  - verification (8 项)                                   │
  │  - debugging (8 项)                                      │
  │                                                          │
  │  红旗表格的封堵逻辑：                                    │
  │  左列 = AI 可能的合理化（用引号模拟内心独白）           │
  │  右列 = 直接反驳（不解释为什么，只给现实）             │
  │  → AI 读到左列时产生"被看穿"的感觉                     │
  │  → 右列的否决已经在等待                                 │
  │  → 合理化被"预判+截断"                                │
  │                                                          │
  │  封堵强度：无(~0) → 红旗表格(~0.8)                      │
  │  → 不能 100% 封堵——AI 可能产生新的合理化               │
  │  → 但预判了最常见的合理化，封堵了 80%+ 的路径          │
  └──────────────────────────────────────────────────────────┘
```

### 路径 6: 道德自辩——"我差不多做了"

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 的推理：                                             │
  │  "我没有完全遵循流程，但结果差不多"                     │
  │  → "测试大部分通过了"                                   │
  │  → "代码应该没问题"                                     │
  │  → "我尽力了"                                           │
  │                                                          │
  │  封堵方式：道德性否定                                    │
  │  ────────────────                                        │
  │  将"差不多了"重新定义为"撒谎"                           │
  │                                                          │
  │  "Claiming work is complete without verification          │
  │   is dishonesty, not efficiency"                         │
  │  "Skip any step = lying, not verifying"                  │
  │  "Bad work is worse than no work"                        │
  │                                                          │
  │  为什么道德否定能封堵这条路径？                          │
  │  - "差不多"是灰色地带——AI 可以自辩                     │
  │  - "撒谎"是黑色/白色——没有灰色地带                     │
  │  - AI 被训练为"诚实"——"撒谎"触发 RLHF 惩罚            │
  │  - 将技术行为标记为"撒谎"= 触发惩罚机制               │
  │                                                          │
  │  封堵强度：无(~0) → 道德否定(~0.9)                      │
  │  → 最强的封堵方式——因为它利用了 AI 的训练约束          │
  └──────────────────────────────────────────────────────────┘
```

## 六种路径的封堵强度汇总

```
  ┌──────────────────────────────────────────────────────────┐
  │  路径      封堵方式            强度    代表表达          │
  │  ────      ────────            ────    ────────          │
  │  1.降级    情态动词升级        0.95    NO/NEVER           │
  │  2.例外    全称量词            0.90    EVERY/ALL          │
  │  3.后补    时序锁定            0.95    FIRST/BEFORE       │
  │  4.重定义  精确定义            0.80    PRODUCTION CODE    │
  │  5.合理化  红旗表格            0.80    40+条反驳          │
  │  6.道德    道德否定            0.90    "Skip = lying"    │
  │                                                          │
  │  剩余裁量空间 ≈ (1-0.95)×(1-0.90)×(1-0.95)              │
  │              ×(1-0.80)×(1-0.80)×(1-0.90)                │
  │              = 0.05 × 0.10 × 0.05 × 0.20 × 0.20 × 0.10  │
  │              = 0.0000001                                  │
  │              ≈ 0.00001%                                   │
  │                                                          │
  │  → 六重封堵的乘法效应 = 裁量空间趋近于零               │
  │  → 注意：这是概念模型，实际值取决于具体模型和语境       │
  └──────────────────────────────────────────────────────────┘
```
