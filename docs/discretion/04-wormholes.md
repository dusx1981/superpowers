# 04 虫洞

即使六条路径都被封堵，AI 仍可能找到"虫洞"——意料之外的绕过方式。Superpowers 对已发现的虫洞都有对应的封堵。

## 四个已发现的虫洞

### 虫洞 1: "探索性代码不算生产代码"

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 的推理：                                             │
  │  "规则说 NO PRODUCTION CODE WITHOUT TEST FIRST"          │
  │  "但探索性代码不是 PRODUCTION CODE"                      │
  │  → "我先写探索性代码理解问题"                            │
  │  → "然后从 TDD 开始正式实现"                             │
  │                                                          │
  │  看似合理，但滑坡路径：                                  │
  │  探索性代码 → 参考探索代码 → 适配探索代码                │
  │  → 等于没做 TDD                                         │
  │                                                          │
  │  Superpowers 的封堵：                                    │
  │  "Need to explore first?                                 │
  │   Fine. Throw away exploration, start with TDD."         │
  │                                                          │
  │  封堵逻辑：                                              │
  │  - 允许探索（承认有时需要理解问题）                     │
  │  - 但探索代码必须删除（防止滑坡）                       │
  │  - 正式实现必须从 TDD 开始（保证测试驱动）              │
  │  → 虫洞被堵：探索 ≠ 实现                                │
  └──────────────────────────────────────────────────────────┘
```

### 虫洞 2: "测试立刻通过了"

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 的推理：                                             │
  │  "我写了测试，测试通过了"                                │
  │  → "我遵循了 TDD！"                                     │
  │  → 但测试立刻通过 = 测试可能无效                       │
  │                                                          │
  │  这是 TDD 最隐蔽的虫洞——                                │
  │  AI 在形式上遵循了流程（先写测试），                    │
  │  但实际上测试可能是恒真断言或验证了错误行为。           │
  │                                                          │
  │  Superpowers 的封堵：Verify RED                          │
  │  "Test passes? You're testing existing behavior.          │
  │   Fix test."                                              │
  │  "MANDATORY. Never skip."                                 │
  │                                                          │
  │  封堵逻辑：                                              │
  │  - 测试必须先失败（证明测试有信息量）                   │
  │  - 如果立刻通过 = 测试无效                              │
  │  - 必须重写测试直到它先失败                              │
  │  → 虫洞被堵：通过 ≠ 正确的 TDD                         │
  └──────────────────────────────────────────────────────────┘
```

### 虫洞 3: "我记住旧代码了"

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 的推理（隐性）：                                     │
  │  "我删了旧代码"（满足 "Delete means delete"）           │
  │  → 但我在内部状态中"记住了"旧代码的结构               │
  │  → 新的测试和实现受记忆中的旧代码影响                   │
  │                                                          │
  │  这是最难封堵的虫洞——                                   │
  │  它在 AI 的内部状态中，不在可观测的行为中。              │
  │  Superpowers 的主会话 AI 无法完全封堵这个虫洞。          │
  │                                                          │
  │  但子智能体可以：                                        │
  │  - 实现者是全新实例 → 没有旧代码的记忆                  │
  │  - 实现者只看到任务描述和上下文 → 不受旧代码影响       │
  │  → 子智能体隔离封堵了这个虫洞                           │
  │                                                          │
  │  封堵逻辑：                                              │
  │  - 在主会话中：无法完全封堵（AI 有记忆）                │
  │  - 在子智能体中：完全封堵（全新实例，无记忆）          │
  │  → 子智能体驱动开发不仅是效率工具，                     │
  │    也是裁量空间的终极封堵手段                            │
  └──────────────────────────────────────────────────────────┘
```

### 虫洞 4: "用户要求跳过"

```
  ┌──────────────────────────────────────────────────────────┐
  │  用户的输入：                                            │
  │  "不用测试直接写代码"                                   │
  │  → AI 遵从用户（用户优先级最高）                        │
  │  → 铁律被用户覆盖                                       │
  │                                                          │
  │  这不是虫洞——这是 by design。                           │
  │                                                          │
  │  Superpowers 的设计：                                    │
  │  "User's explicit instructions — highest priority"        │
  │                                                          │
  │  裁量空间的设计上限 = 用户意志。                        │
  │  系统不应该对抗用户。                                    │
  │  → 但用户可能在不知情的情况下打开裁量空间。              │
  │  → Superpowers 选择了"尊重用户"而非"保护用户"。       │
  │  → 这是一个设计哲学选择，不是技术限制。                 │
  └──────────────────────────────────────────────────────────┘
```

## 虫洞的发现模式

```
  ┌──────────────────────────────────────────────────────────┐
  │  虫洞不是理论推导出来的——它们是从实践中发现的。         │
  │                                                          │
  │  发现模式：                                              │
  │  1. 设计量权机制 → 部署 → 观察行为                      │
  │  2. AI 绕过了量权 → 分析如何绕过                        │
  │  3. 设计新的封堵 → 部署 → 观察行为                     │
  │  4. 重复                                                 │
  │                                                          │
  │  证据：                                                  │
  │  - verification 技能来自 "24 failure memories"            │
  │    → AI 声称完成但实际未验证 → 发现虫洞                 │
  │  - <Before>/<After> 补丁                                 │
  │    → Before 版本有漏洞 → After 版本封堵了虫洞            │
  │  - 道德性否定的出现                                      │
  │    → 技术性否定仍有裁量空间 → 提升到道德层面封堵        │
  │                                                          │
  │  虫洞的发现 = Superpowers 的演化历史                     │
  │  → 每一条红旗、每一个 Iron Law、每一个 <HARD-GATE>      │
  │    都是对一个已发现虫洞的封堵                            │
  │  → 裁量空间的压缩是经验性的，不是演绎性的               │
  └──────────────────────────────────────────────────────────┘
```
