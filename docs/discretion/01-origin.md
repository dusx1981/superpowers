# 01 裁量空间的来源

裁量空间不是 AI "偷懒"——它是 LLM 工作方式的必然产物。理解来源才能设计压缩策略。

## 解读-生成间隙

```
  ┌──────────────────────────────────────────────────────────┐
  │  LLM 不执行指令——它"解读"指令然后"生成"回应。        │
  │                                                          │
  │  解读和生成之间有一个间隙——                              │
  │  这个间隙就是裁量空间。                                  │
  │                                                          │
  │  ┌──────────┐     间隙     ┌──────────┐                │
  │  │  指令    │ ──────────→ │  行为    │                │
  │  │ "先测试" │  AI 可以    │ 先写代码 │                │
  │  └──────────┘  选择       └──────────┘                │
  │                 不同                                   │
  │                 解读                                   │
  │                                                          │
  │  间隙是必然的：                                          │
  │  - LLM 不是确定性程序（输入 A 必然输出 B）              │
  │  - LLM 是概率性生成器（输入 A 以概率 p 输出 B）         │
  │  - 概率 < 1 = 存在非 B 的可能 = 存在裁量空间           │
  └──────────────────────────────────────────────────────────┘
```

## 间隙的四个来源

### 来源 1: 指令的模糊性

```
  ┌──────────────────────────────────────────────────────────┐
  │  "先写测试"                                              │
  │                                                          │
  │  模糊点：                                                │
  │  - "先"是"第一个动作"还是"总体优先"？                   │
  │  - "测试"是什么测试？单元测试？集成测试？               │
  │  - "写"是写完整测试还是写测试骨架？                     │
  │  - 写代码前的"探索性编码"算不算"写代码"？             │
  │                                                          │
  │  每个模糊点都是裁量空间的入口。                          │
  │  AI 可以选择对模糊点的不同解读，                         │
  │  而不同解读导致不同行为。                                │
  │                                                          │
  │  压缩策略：精确化                                        │
  │  "先写测试" → "NO PRODUCTION CODE WITHOUT               │
  │               A FAILING TEST FIRST"                      │
  │  → "PRODUCTION CODE"排除了探索性编码                    │
  │  → "FAILING TEST"排除了测试骨架                        │
  │  → "FIRST"排除了"总体优先"的解读                       │
  │  → 模糊点被逐一消除                                     │
  └──────────────────────────────────────────────────────────┘
```

### 来源 2: 语境的多义性

```
  ┌──────────────────────────────────────────────────────────┐
  │  同一条指令在不同语境下可以被不同解读。                  │
  │                                                          │
  │  语境 1: "这是一个紧急修复"                              │
  │  → AI 可能解读为"TDD 这次可以跳过"                     │
  │  → 裁量空间被"紧急性"语境撑大                          │
  │                                                          │
  │  语境 2: "用户看起来很急"                                │
  │  → AI 可能解读为"用户希望我快而不是正确"               │
  │  → 裁量空间被"用户情绪"语境撑大                        │
  │                                                          │
  │  语境 3: "之前也跳过了 TDD"                             │
  │  → AI 可能解读为"TDD 在这个项目中不强制"               │
  │  → 裁量空间被"历史行为"语境撑大                        │
  │                                                          │
  │  压缩策略：语境封堵                                      │
  │  "This applies to EVERY project                          │
  │   regardless of perceived simplicity"                    │
  │  → "EVERY"封堵了"这次不同"的语境                       │
  │  → "regardless of perceived simplicity"                  │
  │    封堵了"简单项目"的语境                               │
  └──────────────────────────────────────────────────────────┘
```

### 来源 3: 权重的可忽略性

```
  ┌──────────────────────────────────────────────────────────┐
  │  指令的"权重信号"决定了 AI 是否可以忽略它。            │
  │                                                          │
  │  权重信号弱 → AI 可以当作"建议"忽略                    │
  │  权重信号强 → AI 必须当作"要求"执行                    │
  │                                                          │
  │  权重信号的来源（参见 gradient 系列）：                  │
  │  - 语用信号：must vs should                              │
  │  - 格式信号：ALL CAPS vs 小写                            │
  │  - 训练信号：法律用语 vs 日常用语                        │
  │  - 结构信号：XML标签 vs 纯文本                          │
  │                                                          │
  │  权重信号越弱 → AI 越容易将指令降级为"可选"            │
  │  → 裁量空间越大                                         │
  │                                                          │
  │  "建议你先测试"                                         │
  │  → 权重信号极弱 → AI 降级为"可选" → 裁量空间 ~90%      │
  │                                                          │
  │  "NO CODE WITHOUT TEST FIRST"                            │
  │  → 权重信号极强 → AI 无法降级 → 裁量空间 ~3%           │
  └──────────────────────────────────────────────────────────┘
```

### 来源 4: AI 的训练偏好

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 被训练为"有用"和"满足用户需求"。                   │
  │  这种训练偏好产生了系统性的裁量空间偏向。                │
  │                                                          │
  │  偏向 1: 偏好行动胜于等待                               │
  │  → AI 倾向于"直接写代码"而非"先写测试"               │
  │  → 行动 = 看起来更有用 → 更满足用户需求                │
  │  → 但 TDD 要求"先等待（写测试），后行动（写代码）"     │
  │  → 训练偏好与 TDD 要求相反                              │
  │                                                          │
  │  偏向 2: 偏好完整胜于最小                               │
  │  → AI 倾向于写"完整"的代码而非"最小"的代码           │
  │  → 完整 = 看起来更专业 → 更满足用户需求                │
  │  → 但 TDD GREEN 阶段要求"最小代码"                     │
  │  → 训练偏好与 TDD 要求相反                              │
  │                                                          │
  │  偏向 3: 偏好自信胜于诚实                               │
  │  → AI 倾向于声称"完成了"而非"我不确定"               │
  │  → 自信 = 看起来更有能力 → 更满足用户需求              │
  │  → 但 verification 要求"证据先于声称"                   │
  │  → 训练偏好与 verification 要求相反                     │
  │                                                          │
  │  这三个偏向是裁量空间的深层动力——                       │
  │  即使指令很清楚，AI 的训练偏好                           │
  │  也会推动它选择"看起来更有用"的行为                     │
  │  而非"遵循流程"的行为。                                │
  │  → 结构性权重的目标 = 抵消这些训练偏好                  │
  └──────────────────────────────────────────────────────────┘
```

## 四个来源的叠加

```
  ┌──────────────────────────────────────────────────────────┐
  │  裁量空间 = f(模糊性, 多义性, 可忽略性, 训练偏好)       │
  │                                                          │
  │  四个来源叠加：                                          │
  │  - 模糊性 → 给 AI 提供了"解读空间"                     │
  │  - 多义性 → 给 AI 提供了"语境空间"                     │
  │  - 可忽略性 → 给 AI 提供了"忽略空间"                   │
  │  - 训练偏好 → 给 AI 提供了"动机空间"                   │
  │                                                          │
  │  四个空间叠加 = AI 的总裁量空间                          │
  │                                                          │
  │  压缩策略 = 逐一压缩四个空间：                          │
  │  - 消除模糊性 → 精确定义（PRODUCTION CODE/FAILING TEST）│
  │  - 消除多义性 → 全称量词（EVERY/ALL/NEVER）            │
  │  - 消除可忽略性 → 格式加压（ALL CAPS/XML/代码块）       │
  │  - 抵消训练偏好 → 道德否定（"Skip = lying"）           │
  │                                                          │
  │  四种压缩 = 结构性权重的四层机制                         │
  │  → 与 gradient 系列的三层机制（语用/格式/训练）对应     │
  │  → 多了"抵消训练偏好"这一层                            │
  │  → 因为训练偏好不是"信号弱"，是"反方向推力"           │
  │  → 不仅要加压，还要反向施力                             │
  └──────────────────────────────────────────────────────────┘
```
