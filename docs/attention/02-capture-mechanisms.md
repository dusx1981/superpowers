# 02 注意力捕获：7 种机制确保关键指令被"看见"

Superpowers 使用了 7 种注意力捕获机制。目标相同——让关键指令获得高于平均的注意力权重——但原理不同。

## 机制总览

```
  ┌──────────────────────────────────────────────────────────┐
  │  机制           原理               代表模式              │
  │  ────           ────               ────────              │
  │  1. XML 标签    结构性信号增强     HARD-GATE              │
  │  2. 铁律        全大写+代码块      NO CODE WITHOUT TEST   │
  │  3. 宣布仪式    主动输出确认       "I'm using [skill]"    │
  │  4. 流程图      视觉路径锚定       dot/graphviz 图        │
  │  5. 清单        顺序执行约束       9 步 checklist          │
  │  6. Good/Bad    对比学习           正反例对比              │
  │  7. 红旗表      自检触发           "STOP and Start Over"   │
  │                                                          │
  │  7 种机制覆盖注意力的三个阶段：                          │
  │  捕获阶段：1-4（让指令被"看见"）                        │
  │  执行阶段：5-6（让指令被"遵循"）                        │
  │  校验阶段：7  （让指令被"验证"）                        │
  └──────────────────────────────────────────────────────────┘
```

## 机制 1: XML 标签——结构性信号增强

```
  ┌──────────────────────────────────────────────────────────┐
  │  Superpowers 使用的 5 种 XML 标签：                      │
  │                                                          │
  │  EXTREMELY-IMPORTANT → using-superpowers                 │
  │  HARD-GATE           → brainstorming                     │
  │  SUBAGENT-STOP       → using-superpowers                 │
  │  Good / Bad          → TDD, writing-skills               │
  │                                                          │
  │  为什么 XML 标签有效？                                   │
  │  → XML 标签在训练数据中与"结构化数据"关联               │
  │  → LLM 学会了：标签内内容比普通文本更"重要"             │
  │  → 标签名称本身就是语义信号：                            │
  │    "HARD-GATE" = 硬性门禁                                │
  │    "EXTREMELY-IMPORTANT" = 极端重要                      │
  │    "Good" / "Bad" = 正误判断                             │
  │  → 信号强度排序：                                        │
  │    EXTREMELY-IMPORTANT > HARD-GATE > Good/Bad             │
  │  → 标签名称的语义强度 = 注意力捕获强度                  │
  └──────────────────────────────────────────────────────────┘
```

## 机制 2: 铁律——全大写 + 代码块

```
  ┌──────────────────────────────────────────────────────────┐
  │  四个技能的铁律：                                        │
  │  TDD:        NO PRODUCTION CODE WITHOUT A FAILING TEST    │
  │  debugging:  NO FIXES WITHOUT ROOT CAUSE INVESTIGATION    │
  │  verification: NO COMPLETION CLAIMS WITHOUT FRESH EVIDENCE│
  │  writing-skills: NO SKILL WITHOUT A FAILING TEST FIRST    │
  │                                                          │
  │  铁律的三个注意力增强技术：                              │
  │  1. 全大写 = 视觉上的"喊叫"效果                        │
  │     → 训练数据中大写文本常与警告/法律条款关联            │
  │  2. 代码块包裹 = 结构性隔离                              │
  │     → 代码块在训练数据中与"精确、不可修改"关联          │
  │  3. "NO X WITHOUT Y" 句式 = 逻辑上的绝对性              │
  │     → 条件句式比祈使句式更有约束力                      │
  │                                                          │
  │  铁律的三个伴随要素：                                    │
  │  - "Violating the letter is violating the spirit"         │
  │    → 预防"钻空子"的理性化                              │
  │  - "No exceptions:" 清单                                 │
  │    → 逐条关闭可能的例外                                 │
  │  - 即时后果陈述                                          │
  │    → "Write code before the test? Delete it. Start over." │
  │    → 不只说"不行"——还说"违反后的后果"                 │
  └──────────────────────────────────────────────────────────┘
```

## 机制 3: 宣布仪式——主动输出确认

```
  ┌──────────────────────────────────────────────────────────┐
  │  4 个技能要求 AI 在开始时宣布：                           │
  │  writing-plans:  "I'm using the writing-plans skill..."    │
  │  using-git-worktrees: "I'm using the worktrees skill..."  │
  │  finishing:  "I'm using the finishing-a-development..."    │
  │  executing-plans: "I'm using the executing-plans..."       │
  │                                                          │
  │  为什么宣布局能捕获注意力？                               │
  │  → AI 说出一句话 → 进入对话历史 → 获得最高注意力权重    │
  │  → 宣布内容成为注意力的锚点                              │
  │  → 后续行为会"对齐"自己宣布的目标                      │
  │                                                          │
  │  承诺一致性效应：                                        │
  │  → AI 公开声明"我在使用 writing-plans"                   │
  │  → 后续行为更倾向于遵循 writing-plans 规则               │
  │  → 违反规则 = 违反自己的声明 = 认知失调                 │
  │                                                          │
  │  注意力切换功能：                                        │
  │  → 两套规则共存时，宣布"我在使用 writing-plans"          │
  │  → = 明确当前活跃规则集                                 │
  │  → = 压抑旧规则权重                                     │
  │  → = 注意力焦点的显式切换                                │
  └──────────────────────────────────────────────────────────┘
```

## 机制 4: 流程图——视觉路径锚定

```
  ┌──────────────────────────────────────────────────────────┐
  │  7 个技能使用了 dot/graphviz 流程图：                     │
  │  using-superpowers: skill_flow                            │
  │  brainstorming: brainstorming                             │
  │  TDD: tdd_cycle                                          │
  │  subagent-driven: when_to_use + process（两个图）         │
  │  dispatching: when_to_use                                 │
  │  writing-skills: when_flowchart                           │
  │                                                          │
  │  流程图为什么能捕获注意力？                               │
  │  → 流程图 = "可执行的视觉模型"                           │
  │  → 节点和边映射到 AI 的推理图                            │
  │  → 决策节点（菱形）= 注意力的分叉点                     │
  │  → 终态节点（双圆）= 注意力的锚定点                     │
  │                                                          │
  │  流程图的注意力控制功能：                                │
  │  → 定义"合法路径"——AI 只能沿图的边走                   │
  │  → 定义"非法路径"——不存在的边 = 禁止行为               │
  │  → 定义"终态"——图的终点 = 技能出口                     │
  │  → 流程图 = 注意力的轨道                                 │
  │  → AI 不需要"决定"走哪条路，只需"跟随"路径            │
  └──────────────────────────────────────────────────────────┘
```

## 机制 5: 清单——顺序执行约束

```
  ┌──────────────────────────────────────────────────────────┐
  │  清单出现在多个技能中：                                  │
  │  brainstorming: 9 步清单（MUST create task for each）     │
  │  TDD: 8 项验证清单（checkbox 格式）                      │
  │  writing-skills: 20+ 项创建清单（TDD 适配）              │
  │  verification: 5 步 Gate Function（伪代码格式）           │
  │                                                          │
  │  清单 = 外部化的工作记忆                                 │
  │  → AI 不需要"记住"所有步骤——清单替它记住               │
  │  → 每完成一步，注意力自然移到下一步                     │
  │  → 清单的顺序 = 注意力的路径                            │
  │                                                          │
  │  清单 + TodoWrite：                                      │
  │  → 清单转化为 TodoWrite 任务                             │
  │  → TodoWrite 提供外部状态追踪                            │
  │  → AI 不需要把清单保持在注意力中                         │
  │  → 释放注意力带宽用于执行                               │
  │                                                          │
  │  有清单 vs. 没有清单：                                   │
  │  没有 → AI 自己决定下一步 → 可能跳过/改序/遗漏          │
  │  有   → 下一步由清单决定 → 注意力集中在执行当前步骤     │
  └──────────────────────────────────────────────────────────┘
```

## 机制 6: Good/Bad 对比学习

```
  ┌──────────────────────────────────────────────────────────┐
  │  TDD 技能中的 Good/Bad 对比：                            │
  │  Good: test('retries failed operations 3 times', ...)      │
  │        → Clear name, tests real behavior, one thing        │
  │  Bad:  test('retry works', ...)                            │
  │        → Vague name, tests mock not code                   │
  │                                                          │
  │  对比学习为什么有效？                                    │
  │  → 两个极端并置 = 差异被放大                            │
  │  → AI 不只学"应该怎么做"——还学"不应该怎么做"          │
  │  → 注意力在两者间跳转 → 发现关键差异                   │
  │  → 关键差异 = "tests real behavior" vs. "tests mock"      │
  │  → 这是单靠正面规则无法传达的细微差别                   │
  │                                                          │
  │  对比学习 vs. 纯正面规则：                               │
  │  纯正面规则 → AI 理解规则但不知道边界                   │
  │  对比学习   → AI 理解规则和边界                         │
  └──────────────────────────────────────────────────────────┘
```

## 机制 7: 红旗表——自检触发

```
  ┌──────────────────────────────────────────────────────────┐
  │  红旗表出现在 9 个技能中，三种变体：                     │
  │  "STOP and Start Over"（最严重——需要完全重置）           │
  │  "STOP and Follow Process"（回到流程起点）                │
  │  "STOP"（直接停止）                                      │
  │                                                          │
  │  红旗表 = AI 的"良心"                                   │
  │  → 当 AI 想做某事时，红旗表触发自检                    │
  │  → "我现在的想法是不是红旗表中的一条？"                 │
  │  → 如果是 → STOP                                        │
  │  → 如果不是 → 继续                                      │
  │                                                          │
  │  红旗表 vs. 正面规则：                                   │
  │  正面规则告诉 AI "该做什么"                              │
  │  红旗表告诉 AI "不该想什么"                              │
  │  → 正面规则是路标 → 引导方向                            │
  │  → 红旗表是护栏 → 防止偏航                              │
  │  → 两者互补：路标让 AI 知道方向，                       │
  │    护栏让 AI 知道边界                                    │
  │                                                          │
  │  红旗表的自检机制：                                      │
  │  → 红旗表列出的是"AI 内心可能的想法"                    │
  │  → 不是外部行为——是内心独白                            │
  │  → 例如："This is just a simple question"                │
  │  → AI 读到这条 → 意识到"我正在这么想"                  │
  │  → 立即 STOP → 检查技能 → 转向正确行为                  │
  │  → 红旗表 = 将 AI 的隐性偏误显性化                      │
  └──────────────────────────────────────────────────────────┘
```
