# 01 注意力稀缺性

## 什么是注意力稀释

```
  ┌──────────────────────────────────────────────────────────┐
  │  定义：                                                  │
  │  注意力稀释 = 上下文增长时，AI 对任意单条指令的         │
  │  服从概率下降的现象。                                    │
  │                                                          │
  │  直觉：                                                  │
  │  10 条指令时，AI 大约能遵循 9 条                        │
  │  100 条指令时，AI 大约能遵循 60 条                      │
  │  1000 条指令时，AI 大约能遵循 200 条                    │
  │  → 不是线性关系，而是对数关系                            │
  │  → 指令越多，每条指令被"看见"的概率越低                 │
  │                                                          │
  │  类比：                                                  │
  │  注意力 = 手电筒的光束                                   │
  │  照射区域固定 → 覆盖的目标越多 → 每个目标越暗          │
  │  或者：                                                  │
  │  注意力 = CPU 时间片                                     │
  │  进程越多 → 每个进程获得的 CPU 时间越少                 │
  └──────────────────────────────────────────────────────────┘
```

## 为什么 AI 的注意力是稀缺的

```
  ┌──────────────────────────────────────────────────────────┐
  │  原因 1: 位置效应 (Recency Bias)                         │
  │  ──────────────────────────────────────────              │
  │  LLM 的注意力机制天然偏向最近的 token                    │
  │  → 距离当前位置越远的指令，注意力权重越低                │
  │  → 这是 Transformer 架构的结构性特征                    │
  │  → 不是 bug——是模型处理长序列的基本方式                │
  │                                                          │
  │  在 Superpowers 中的表现：                               │
  │  → using-superpowers 在上下文的最远处（第一个消息）      │
  │  → 10 轮对话后，它的权重可能是初始的 1/5                │
  │  → 20 轮对话后，可能降到 1/10                           │
  │  → "Might any skill apply?" 的检查可能被跳过            │
  │                                                          │
  │  原因 2: 竞争效应 (Competition)                          │
  │  ──────────────────────────────────────────              │
  │  上下文中的指令不是独立的——它们争夺同一个注意力预算     │
  │  → brainstorming 的 HARD-GATE 和 subagent-driven 的      │
  │    "派发实现者" 在上下文中竞争                          │
  │  → AI 必须判断当前哪个规则有效                          │
  │  → 当两条规则竞争时，AI 的判断可能出错                  │
  │  → 竞争越激烈（同时存在的规则越多），出错概率越高       │
  │                                                          │
  │  原因 3: 稀释效应 (Dilution)                             │
  │  ──────────────────────────────────────────              │
  │  对话历史中的非指令内容（代码、解释、问答）              │
  │  稀释了指令的信号强度                                    │
  │  → 100 条指令 + 900 行代码 = 指令被稀释 10 倍           │
  │  → 代码和解释"吸收"了注意力                            │
  │  → AI 可能更关注代码细节而忽略行为规则                  │
  │                                                          │
  │  三种效应的叠加：                                        │
  │  位置效应：远处的指令权重降低                            │
  │  竞争效应：同时存在的指令互相争抢                        │
  │  稀释效应：非指令内容降低信号强度                        │
  │  → 三者叠加 = 注意力的指数级衰减                        │
  └──────────────────────────────────────────────────────────┘
```

## Superpowers 面临的注意力挑战

```
  ┌──────────────────────────────────────────────────────────┐
  │  挑战 1: 多技能上下文                                    │
  │  ────────────────────────                                │
  │  主链技能共享对话上下文                                  │
  │  → brainstorming 完成后，规则仍在上下文中                │
  │  → writing-plans 加载后，两套规则共存                    │
  │  → subagent-driven 加载后，三套规则共存                  │
  │  → 规则数量线性增长，但注意力预算不变                    │
  │                                                          │
  │  挑战 2: 长对话会话                                      │
  │  ────────────────────────                                │
  │  完整的功能开发可能需要 20+ 轮对话                      │
  │  → brainstorming 的问答：5-10 轮                        │
  │  → writing-plans 的计划：3-5 轮                         │
  │  → subagent-driven 的执行：10+ 轮（每个任务 + 审阅）     │
  │  → finishing 的收尾：2-3 轮                              │
  │  → 总计 20-30 轮对话，早期指令极度稀释                  │
  │                                                          │
  │  挑战 3: 代码与规则的混合                                │
  │  ────────────────────────                                │
  │  实现阶段产生大量代码                                    │
  │  → 每个子智能体的输出包含代码和解释                     │
  │  → 审阅者的反馈包含详细的问题列表                       │
  │  → 这些内容稀释了行为规则的注意力权重                   │
  │  → AI 可能更关注"代码是否正确"而忽略"流程是否正确"    │
  │                                                          │
  │  挑战 4: 子智能体的注意力隔离                            │
  │  ────────────────────────                                │
  │  子智能体的上下文是隔离的——这是优势                     │
  │  → 子智能体没有稀释问题（上下文短、规则少）             │
  │  → 但 Controller 本身仍有稀释问题                       │
  │  → Controller 累积了所有子智能体的报告和审阅             │
  │  → Controller 的注意力随着任务推进而持续稀释            │
  │                                                          │
  │  核心矛盾：                                              │
  │  Superpowers 需要在注意力稀缺的条件下，                  │
  │  保证关键指令始终被服从。                                │
  │  这就是注意力工程的出发点。                              │
  └──────────────────────────────────────────────────────────┘
```
