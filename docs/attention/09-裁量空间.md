## 裁量空间在大模型中的数学原理

在大语言模型中，“裁量空间”指的是：给定上下文 \( C \)，模型预测下一个 token 时，在概率分布中允许选择的范围。这个空间的大小直接影响模型输出的确定性或多样性，以及“遵守规则”的程度。

---

### 1. 概率分布与裁量空间

模型前向传播后，通过 softmax 层得到词表 \( V \) 上的概率分布：

\[
P(x_t \mid x_{<t}) = \text{softmax}\left( \frac{\mathbf{z}_t}{T} \right)
\]

其中 \( \mathbf{z}_t \) 是 logits 向量，\( T \) 是温度参数。  
**裁量空间**可以理解为：温度 \( T>0 \) 时，所有概率非零的 token 构成的多项式采样空间。当 \( T \to 0^+ \)，分布趋近于 one-hot（总是选最高概率 token），裁量空间收缩为单点。当 \( T \to \infty \)，分布趋于均匀，裁量空间覆盖整个词表。

**数学定义**：  
令阈值 \( p_{\min} \) 为一个最小可接受概率（例如 \( 10^{-5} \)），则裁量空间为：

\[
\mathcal{D}(C) = \left\{ v \in V \mid P(v \mid C) > p_{\min} \right\}
\]

其大小为 \( |\mathcal{D}(C)| \)，对数尺度可视为熵的指数形式。

---

### 2. 规则约束如何压缩裁量空间

通过系统提示词、指令微调或强化学习，模型将规则内化为**对特定 token 分布的修改**。好的铁律相当于对某些 token 的概率置零（或极低），从而将裁量空间削减到仅允许合规 token。

**例子：TDD 铁律“NO CODE WITHOUT TEST”**

假设模型正在生成一个 Python 函数的实现。在普通指令下，模型可能会直接输出函数体。在注入 TDD 铁律后，模型必须首先生成测试代码。这个过程可以看作：

- **无约束时**：下一个 token 概率分布中，`def`（开始写函数）的概率很高，`import pytest` 等测试相关 token 概率较低。
- **铁律约束时**：通过语言建模，上下文被强制带上“必须看到 `test_` 前缀”的约束。在注意力机制中，铁律文本会占据高注意力权重，使得“写测试”路径的 logits 被抬高，而“直接写代码”路径的 logits 被压低。最终采样时，采样空间从“所有合法 token”收缩到“仅`test`开头或测试框架导入”的子集。

**数学形式**：设规则 \( R \) 指示一个必然的前缀序列 \( S_R \)。模型在给定上下文 \( C \) 时，必须使 \( P(S_R \mid C) \approx 1 \)。这要求规则上下文对 logits 的调节满足：

\[
\mathbf{z}_t' = \mathbf{z}_t + \mathbf{b}_R
\]

其中 \( \mathbf{b}_R \) 是一个偏置向量，对不合规 token 施加大的负值（可视为无限大），使得 softmax 输出趋近于 0。

---

### 3. 温度与采样空间的交互

即使铁律将合规 token 集压缩为单一路径（例如必须先生成测试函数名），温度参数仍引入裁量空间：在合规集合内部，不同的具体 token（如 `test_add` vs `test_calculate`）可以有相对概率分布。

**数学**：合规词表 \( V_R \subset V \)，裁量空间 \( \mathcal{D}_R = V_R \)（假设所有合规 token 概率均高于阈值）。其大小 = \( |V_R| \)。温度 \( T \) 调节合规 token 之间的相对概率：

\[
P(v \mid C, R) = \frac{\exp(z_v / T)}{\sum_{u \in V_R} \exp(z_u / T)}
\]

当 \( T \to 0 \) 时，即使有多个合规 token，模型也会选择其中概率最高的一个，裁量空间退化为单点；当 \( T=1 \) 时，保留原始相对概率；当 \( T \) 增大时，统一化程度更高，裁量空间被视为“平滑选择”。

---

### 4. 具体举例：TDD 中的“最小实现”裁量空间

假设测试要求 `add(1,2)` 返回 `3`。合规模块必须让测试通过。在铁律下，**不能返回错误结果**，但内部实现有多个可能性（硬编码 `3` vs 加法运算）。模型在生成代码时，裁量空间由所有“能让测试通过”的程序组成。

- **无铁律**：裁量空间极大（任何返回 `3` 的代码，甚至包括网络调用、复杂算法）。
- **铁律 + 最小实现原则**：模型被训练或提示“写最少代码”。此时合规集缩小到最小程序集合。数学上，这是一个**集合的最简化优化**。模型利用其归纳偏置（如倾向于使用加法运算符 `+`）来选择一个解。裁量空间从无限缩小到少数几个“简单”实现，最终模型往往选择 `return a+b`。

**数学建模**：设所有通过测试的程序集合为 \( \mathcal{P} \)。引入成本函数 \( \text{cost}(p) \)（如 token 长度、复杂度）。铁律要求解必须属于 \( \mathcal{P} \) 且最小化 \( \text{cost}(p) \)。模型的输出是搜索空间中该优化问题的近似解。裁量空间即为所有最优解（可能不唯一）。如果模型具有确定性解码（贪婪），则选唯一最优，裁量空间缩为单点；如果采样，则可能在几个最优解间浮动。

---

### 5. 铁律如何做到“裁量空间 = 单点”

大模型本质上是一个概率模型，永远存在一个非零概率生成不合规 token（除非通过 logit 掩码强制置零）。真正将裁量空间变为单点需要**外部确定性约束**，例如：

- 在解码循环中直接屏蔽不合规 token（`logit_mask` 使非法 token 的 logits = -∞）。这时可行域确实只有单个 token（如果只有一个合规 token）。
- 或者使用有限状态机约束解码，强制必须按特定序列生成。

在 Superpowers 提示词中，铁律通过自然语言要求模型“不要生成 X，只能生成 Y”，但没有硬件级别的屏蔽。因此，裁量空间并非严格单点，而是具有极低概率的“逃逸”可能。数学上，它等效于：

\[
\forall v \notin V_R: P(v \mid C) \approx 0
\]

但非严格零。优秀的指令微调可以使约等于 0 足够小，实践中模型从不违规。

---

### 总结

| 概念 | 数学描述 |
|------|----------|
| 裁量空间 | \( \mathcal{D}(C) = \{v \in V \mid P(v \mid C) > \epsilon\} \) |
| 规则约束 | 通过 logit 偏置或掩码，使 \( P(V_R \mid C) \approx 1 \) |
| 温度调节 | 控制合规 token 间的相对概率分布，影响采样子空间的大小 |
| 铁律实现 | 强制 \( V_R \) 为单点集合，\(\epsilon\) 极小，实际为零 |

**实例**：TDD 铁律要求“先写测试” → \( V_R = \{\text{`test\_' 作为下一个 token}\} \)（假设上下文已确定测试是唯一合法行为）。此时 \(\mathcal{D}(C)\) 大小 = 1，裁量空间消失。
