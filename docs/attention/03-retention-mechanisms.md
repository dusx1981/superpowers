# 03 注意力保持：冗余、重复、首尾锚定

捕获注意力是第一步，保持注意力是第二步。在长对话中，被捕获的指令可能因上下文膨胀而再次被稀释。Superpowers 用三种策略保持注意力。

## 策略 1: 冗余——同一约束在多处出现

```
  ┌──────────────────────────────────────────────────────────┐
  │  以 brainstorming 的 HARD-GATE 为例：                    │
  │                                                          │
  │  出现位置 1: HARD-GATE 标签（第 12-14 行）               │
  │  → "Do NOT invoke any implementation skill..."            │
  │  → 结构性信号增强（XML 标签）                            │
  │                                                          │
  │  出现位置 2: Anti-Pattern 标题（第 16-18 行）             │
  │  → "This Is Too Simple To Need A Design"                  │
  │  → 预设了最常见的绕过路径                                │
  │                                                          │
  │  出现位置 3: 流程图终态（第 66 行）                       │
  │  → "The terminal state is invoking writing-plans."         │
  │  → "Do NOT invoke frontend-design, mcp-builder..."        │
  │  → 视觉路径锚定（流程图的唯一出口）                      │
  │                                                          │
  │  出现位置 4: Implementation 节（第 136 行）               │
  │  → "Do NOT invoke any other skill.                        │
  │     writing-plans is the next step."                       │
  │  → 离开技能前的最后一道防线                              │
  │                                                          │
  │  四重冗余的注意力分布：                                  │
  │  ┌──────────────────────────────────────────────────┐    │
  │  │  上下文位置    冗余层           注意力权重          │    │
  │  │  ──────────    ──────           ──────────          │    │
  │  │  开头         HARD-GATE        高（首次看到）       │    │
  │  │  开头+1       Anti-Pattern     高（紧随其后）       │    │
  │  │  中间         流程图终态       中（视觉锚定）       │    │
  │  │  末尾         Implementation   中（离开前最后）     │    │
  │  │                                                     │    │
  │  │  → 任一位置被注意到即可生效                         │    │
  │  │  → 覆盖了 AI 可能在上下文中"扫视"的不同位置        │    │
  │  └──────────────────────────────────────────────────┘    │
  │                                                          │
  │  冗余 vs. DRY 的权衡：                                   │
  │  DRY（Don't Repeat Yourself）= 代码原则                  │
  │  → 代码中重复 = 维护负担                                │
  │  → 代码中重复 = 不一致风险                              │
  │                                                          │
  │  冗余 = 提示词原则                                       │
  │  → 提示词中重复 = 注意力保险                            │
  │  → 提示词中重复 = 覆盖不同注意力位置                   │
  │  → 代码中 DRY，提示词中冗余                             │
  │  → 这是 Superpowers 的核心设计决策之一                  │
  └──────────────────────────────────────────────────────────┘
```

## 策略 2: 首尾锚定——在最强位置放置关键约束

```
  ┌──────────────────────────────────────────────────────────┐
  │  注意力权重的位置分布（经验模型）：                      │
  │                                                          │
  │  权重                                                    │
  │  ▲                                                       │
  │  │  █                                            █       │
  │  │  █                                            █       │
  │  │  █         █                         █        █       │
  │  │  █    █    █    █    █    █    █    █   █    █       │
  │  │  █    █    █    █    █    █    █    █   █    █       │
  │  └──┼────┼────┼────┼────┼────┼────┼────┼───┼────┼──►  │
  │     开头  1/4  1/3  1/2      2/3  3/4     末尾  位置    │
  │                                                          │
  │  → 开头和末尾的注意力权重最高                           │
  │  → 中间偏低，特别是 1/3 到 2/3 之间                    │
  │  → 这就是"首因效应"和"近因效应"                       │
  │                                                          │
  │  Superpowers 的首尾锚定策略：                            │
  │                                                          │
  │  ┌─ 首部锚定 ──────────────────────────────────────┐   │
  │  │  技能文件的第一条实质性内容 = 最强约束              │   │
  │  │                                                     │   │
  │  │  brainstorming: HARD-GATE（第 12 行）               │   │
  │  │  TDD: "Violating the letter..." + Iron Law           │   │
  │  │  debugging: "Violating the letter..." + Iron Law     │   │
  │  │  verification: "Violating the letter..." + Iron Law  │   │
  │  │  using-superpowers: EXTREMELY-IMPORTANT              │   │
  │  │                                                     │   │
  │  │  → AI 打开技能文件，第一眼看到的就是最强约束       │   │
  │  │  → 首因效应确保约束被记住                          │   │
  │  └──────────────────────────────────────────────────┘   │
  │                                                          │
  │  ┌─ 尾部锚定 ──────────────────────────────────────┐   │
  │  │  技能文件的最后一条实质性内容 = 重复最强约束        │   │
  │  │                                                     │   │
  │  │  TDD: Final Rule（第 364 行）                       │   │
  │  │    "Production code → test exists and failed first"  │   │
  │  │    → 重述 Iron Law                                  │   │
  │  │                                                     │   │
  │  │  writing-skills: The Bottom Line（第 651 行）        │   │
  │  │    → 重述 Iron Law                                  │   │
  │  │                                                     │   │
  │  │  brainstorming: Implementation 节                    │   │
  │  │    → 重述唯一出口                                   │   │
  │  │                                                     │   │
  │  │  → AI 离开技能文件前，最后看到的是重述的约束       │   │
  │  │  → 近因效应确保约束在后续行为中仍有效              │   │
  │  └──────────────────────────────────────────────────┘   │
  │                                                          │
  │  首尾锚定 = 注意力的"三明治"结构：                      │
  │  开头：强约束（不要做什么）                              │
  │  中间：详细指导（怎么做）                                │
  │  末尾：重述约束（再次强调不要做什么）                    │
  │  → 约束包裹了指导，指导服务于约束                      │
  └──────────────────────────────────────────────────────────┘
```

## 策略 3: 理性化反驳表——保持注意力不被"借口"劫持

```
  ┌──────────────────────────────────────────────────────────┐
  │  AI 的注意力不只是被"稀释"——还可能被"劫持"。           │
  │  劫持者 = AI 自身的理性化（rationalization）。           │
  │                                                          │
  │  理性化的例子：                                          │
  │  "This is just a simple question" → 跳过技能检查        │
  │  "I'll test after" → 跳过 TDD                           │
  │  "Should work now" → 跳过验证                           │
  │  "Just one quick fix" → 跳过调试流程                    │
  │                                                          │
  │  理性化劫持注意力的机制：                                │
  │  → 理性化产生了一个"替代行动"                          │
  │  → 替代行动看起来比遵循规则更"高效"                    │
  │  → AI 的注意力被替代行动吸引                            │
  │  → 规则的注意力权重被替代行动"偷走"                    │
  │  → 结果：AI 做了不该做的事                              │
  │                                                          │
  │  理性化反驳表如何防止劫持：                              │
  │                                                          │
  │  using-superpowers 的红旗表（12 条）：                    │
  │  | Thought | Reality |                                   │
  │  | "This is just a simple question"                      │
  │  | Questions are tasks. Check for skills. |              │
  │  | "I need more context first"                           │
  │  | Skill check comes BEFORE clarifying questions. |      │
  │  | ... |                                                 │
  │                                                          │
  │  TDD 的理性化表（11 条）：                               │
  │  | Excuse | Reality |                                    │
  │  | "Too simple to test"                                  │
  │  | Simple code breaks. Test takes 30 seconds. |          │
  │  | "I'll test after"                                     │
  │  | Tests passing immediately prove nothing. |            │
  │  | ... |                                                 │
  │                                                          │
  │  反驳表的工作原理：                                      │
  │  1. 列出 AI 可能产生的"想法"（理性化）                  │
  │  2. 每个想法旁边放一个"现实"（反驳）                    │
  │  3. AI 读到自己的想法 → 立即看到反驳                    │
  │  4. 注意力从"替代行动"被拉回"规则"                    │
  │  → 反驳表 = 注意力的"反劫持"机制                       │
  │                                                          │
  │  反驳表 vs. 简单重复：                                   │
  │  简单重复："Don't skip TDD. Don't skip TDD."             │
  │  → AI 可能仍然认为"这次不一样"                         │
  │  → 重复没有解决"为什么这次可以跳过"的推理              │
  │                                                          │
  │  反驳表：                                                │
  │  AI 想："I'll test after"                                │
  │  表中说："Tests passing immediately prove nothing"         │
  │  → AI 的推理被直接反驳                                  │
  │  → 注意力不是被"拉回来"——而是被"说服回来"            │
  │  → 反驳比重复更有效——因为它消灭了偏航的"理由"         │
  └──────────────────────────────────────────────────────────┘
```
