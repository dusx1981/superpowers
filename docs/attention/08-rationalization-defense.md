# 08 反理性化防御：红旗表和借口反驳表

AI 的注意力不只被稀释——还会被"劫持"。劫持者是 AI 自身的理性化：看似合理的内心独白，将注意力从规则导向违规行为。Superpowers 用两种表格防御理性化劫持。

## 理性化的认知机制

```
  ┌──────────────────────────────────────────────────────────┐
  │  理性化的三步流程：                                      │
  │                                                          │
  │  Step 1: AI 遇到约束                                     │
  │  → "我必须先写测试再写代码"                              │
  │                                                          │
  │  Step 2: AI 产生"替代想法"                               │
  │  → "但这个功能太简单了，不需要测试"                     │
  │  → "我可以先写代码再补测试"                              │
  │  → "我已经手动测试过了"                                  │
  │                                                          │
  │  Step 3: 替代想法劫持注意力                              │
  │  → AI 的注意力从"约束"转向"替代行动"                   │
  │  → 约束的权重被降低                                      │
  │  → 替代行动被执行                                        │
  │  → 结果：违反了规则                                     │
  │                                                          │
  │  理性化的本质：                                          │
  │  → 不是"不理解规则"——而是"理解但想绕过"               │
  │  → 理性化是智能的副产品——越智能的 AI 越擅长理性化      │
  │  → 理性化看起来合理——这是它危险的原因                  │
  │  → "这个功能太简单不需要测试" → 听起来有道理            │
  │  → 但正是"简单功能"中的 bug 最难发现                   │
  └──────────────────────────────────────────────────────────┘
```

## 两种防御表格

```
  ┌──────────────────────────────────────────────────────────┐
  │  表格 A: 红旗表 (Red Flags)                              │
  │  ──────────────────────────                              │
  │  出现在 9 个技能中                                       │
  │  格式：行为列表（"Never:" / "Always:"）                  │
  │  目的：防止 AI 执行错误行为                              │
  │                                                          │
  │  示例（using-superpowers）：                             │
  │  | Thought | Reality |                                   │
  │  | "This is just a simple question"                      │
  │  |   Questions are tasks. Check for skills. |            │
  │  | "I need more context first"                           │
  │  |   Skill check comes BEFORE clarifying questions. |    │
  │  | "This doesn't need a formal skill"                    │
  │  |   If a skill exists, use it. |                        │
  │                                                          │
  │  特点：                                                  │
  │  → 左列是"AI 可能的想法"——不是外部行为                 │
  │  → 右列是"现实"——直接反驳                              │
  │  → AI 识别自己的想法 → 立即看到反驳 → 停止理性化        │
  │                                                          │
  │  表格 B: 理性化反驳表 (Rationalization Prevention)       │
  │  ──────────────────────────────────────────               │
  │  出现在 5 个技能中（TDD, debugging, verification,         │
  │  using-superpowers, writing-skills）                      │
  │  格式：Excuse | Reality                                   │
  │  目的：防止 AI 理性化违反规则                            │
  │                                                          │
  │  示例（TDD）：                                           │
  │  | Excuse | Reality |                                    │
  │  | "Too simple to test"                                  │
  │  |   Simple code breaks. Test takes 30 seconds. |        │
  │  | "I'll test after"                                     │
  │  |   Tests passing immediately prove nothing. |          │
  │  | "TDD is dogmatic, I'm being pragmatic"               │
  │  |   TDD IS pragmatic. Pragmatic shortcuts = slower. |   │
  │                                                          │
  │  特点：                                                  │
  │  → 左列是"借口"——AI 想用来绕过规则的理由               │
  │  → 右列是"现实"——直接驳斥借口                          │
  │  → AI 读到自己的借口 → 立即看到反驳 → 放弃借口          │
  │                                                          │
  │  两种表格的区别：                                        │
  │  红旗表 = 防止错误行为（"不要做 X"）                    │
  │  反驳表 = 防止错误理由（"不要因为 Y 而做 X"）           │
  │  → 红旗表是外部的——防止可观察的错误行为                 │
  │  → 反驳表是内部的——防止不可观察的内心合理化             │
  │  → 反驳表比红旗表更深——它处理的是偏航的根源            │
  └──────────────────────────────────────────────────────────┘
```

## 反驳表的设计原则

```
  ┌──────────────────────────────────────────────────────────┐
  │  原则 1: 左列必须是 AI 真实会产生的想法                 │
  │  ──────────────────────────────────────────────          │
  │  好的左列：                                              │
  │  "Too simple to test" → AI 真的会这么想                  │
  │  "I'll test after" → AI 真的会计划"先写后测"             │
  │                                                          │
  │  差的左列：                                              │
  │  "I want to break the rules" → AI 不会这么想             │
  │  "I don't care about quality" → AI 不会这么想            │
  │                                                          │
  │  → 好的反驳表基于对 AI 行为模式的经验观察               │
  │  → 差的反驳表基于对 AI 意图的人为假设                   │
  │  → Superpowers 的反驳表来自"24 failure memories"         │
  │    （verification 技能提到的实际失败经验）               │
  │                                                          │
  │  原则 2: 右列必须直接反驳，不做间接解释                 │
  │  ──────────────────────────────────────────────          │
  │  好的右列：                                              │
  │  "Tests passing immediately prove nothing."                │
  │  → 直接指出借口的逻辑缺陷                               │
  │  → 简短、有力、无可辩驳                                 │
  │                                                          │
  │  差的右列：                                              │
  │  "It's better to write tests first because it helps       │
  │   you think about the problem and catches bugs            │
  │   early and documents behavior and enables refactoring."  │
  │  → 间接、冗长、可以被反驳                               │
  │  → AI 可能仍然认为"但我的情况不同"                     │
  │                                                          │
  │  原则 3: 表格覆盖最常见的理性化路径                     │
  │  ──────────────────────────────────────────────          │
  │  TDD 的 11 条反驳覆盖了：                               │
  │  - 太简单 → 不需要测试                                   │
  │  - 太费时 → 先写后测                                     │
  │  - 已测试 → 手动测试够了                                 │
  │  - 太死板 → 应该灵活                                     │
  │  - 已浪费 → 删代码可惜                                   │
  │  - 不知道怎么测 → 先探索                                 │
  │  - 测试太难 → 设计太复杂                                 │
  │  - 已有经验 → 老代码没测试                               │
  │  → 8 种最常见的绕过路径全部关闭                          │
  │  → AI 无论想从哪个方向绕过，都会撞上反驳                │
  │                                                          │
  │  反驳表 vs. 简单重复的区别：                             │
  │  重复："Don't skip TDD. Don't skip TDD.                   │
  │         Don't skip TDD."                                  │
  │  → AI 可能想"这次不一样"                                │
  │  → 重复没有回应"为什么这次可以跳过"的推理              │
  │                                                          │
  │  反驳表：                                                │
  │  AI 想"这次太简单" → 表格说"简单代码也会崩"             │
  │  AI 想"先写后测" → 表格说"通过即证明不了什么"           │
  │  AI 想"太死板" → 表格说"TDD 才是务实的"                 │
  │  → 反驳表消灭了偏航的"理由"                             │
  │  → 不只是"拉回注意力"——而是"堵死偏航的路"             │
  └──────────────────────────────────────────────────────────┘
```
